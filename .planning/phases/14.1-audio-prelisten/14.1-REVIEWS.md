# Cross-AI Plan Review — Phase 14.1

## Codex Review

### Plan 14.1-01: Audio Prelisten Backend

#### Summary
This plan is close to the right architectural shape: it reuses the existing `NJClient`, introduces explicit UI-thread command/event plumbing, and keeps prelisten as a mode rather than a second client instance. That said, in its current form it is not safe to implement because it misses multiple research-confirmed correctness issues that would break the core receive-only behavior, leak local channels into preview sessions, and create incorrect identity/state handling. It also under-specifies how prelisten audio should be mixed so the current volume approach risks scaling the entire plugin output instead of only the preview stream.

#### Strengths
- Reuses the main `NJClient` instead of introducing a second connection stack.
- Adds explicit command/event types, which keeps browser UI and network thread responsibilities separate.
- Includes stop/switch behavior as first-class commands, which aligns with the one-prelisten-at-a-time requirement.
- Recognizes the need to suppress normal local-channel setup while prelistening.
- Covers license auto-accept and connection failure signaling, which are necessary for "one click to hear audio."

#### Concerns
- **HIGH**: The plan does not incorporate the zero-channel fix. `Disconnect()` does not clear `m_locchannels`, so prelisten after a normal session can still announce local channels unless `DeleteLocalChannel()` is called before prelisten `Connect()`.
- **HIGH**: The username construction is wrong. Using `client->GetUser()` after `Disconnect()` will yield an empty user, producing only `"[preview]"` and losing the actual identity suffix.
- **HIGH**: The plan's processor-volume step is likely incorrect for the requirement. "Scale all output channels by `prelisten_volume` when `prelisten_mode` is true" would attenuate the entire plugin output, not just preview audio. In plugin mode this can wrongly change host-routed audio or any future local monitoring path.
- **HIGH**: The plan does not explicitly state that `justmonitor` must remain `false`. Given the misleading comment already identified in research, this is an implementation hazard likely to regress into silent preview.
- **MEDIUM**: No explicit transition contract is defined for switching from prelisten to normal connect. Clearing `prelisten_mode` on `ConnectCommand` is not enough unless disconnect/order/state reset is deterministic.
- **MEDIUM**: `PrelistenStateEvent { bool connected; std::string server_name; }` is too weak for robust UI reconciliation. The UI needs at least host+port to survive server-list refreshes and row reordering.
- **MEDIUM**: Auto-subscribing to all remote channels is mentioned in the objective but not concretely planned as a task. If that logic depends on existing post-connect flows, it should be called out explicitly.
- **LOW**: No explicit handling is described for disconnect/error events that occur asynchronously after a successful prelisten connection. The UI may remain in a listening state if the server drops.

#### Suggestions
- Replace username derivation with a stable source such as `processor.lastUsername`, and define fallback behavior if it is empty.
- Add an explicit prelisten preparation step:
  - disconnect current session if needed
  - iterate existing local channels and call `DeleteLocalChannel(...)`
  - set prelisten mode
  - connect with `justmonitor=false`
- Expand `PrelistenStateEvent` to include `host`, `port`, and optionally an enum/status (`connecting`, `connected`, `stopped`, `failed`) rather than a single boolean.
- Separate preview gain from master output gain. The plan should state where preview samples are mixed and apply `prelisten_volume` only to that path.
- Add explicit handling for async disconnect/failure callbacks so prelisten state is always cleared and the UI is notified.
- Make "auto-subscribe to all remote channels" a concrete implementation task with verification notes.
- Define connect/prelisten ordering clearly so a normal join always tears down preview cleanly before local channel setup resumes.

#### Risk Assessment
**HIGH**. The missing research fixes are not cosmetic; they directly affect correctness of the preview connection and can cause wrong channel advertisement, silent preview, incorrect user identity, and incorrect audio gain behavior. The architecture is salvageable, but the current plan is not implementation-safe.

### Plan 14.1-02: Audio Prelisten UI and Editor Wiring

#### Summary
This plan covers the visible UX well and maps reasonably onto the existing browser/editor structure, but it is weaker on state durability and side-effect containment. The biggest gap is that it still treats active preview primarily as a row index/UI decoration problem rather than a connection state problem keyed by server identity. It also does not yet absorb the research finding that prelisten can leak into other UI systems like OSC/channel strips/connection indicators, which makes the current editor wiring incomplete for the stated milestone quality bar.

#### Strengths
- Matches the user decisions closely: dedicated per-row button, one active preview, title-bar volume slider, and no listen action for empty rooms.
- Preserves existing single-click and double-click semantics instead of overloading row interaction.
- Includes explicit enable/disable logic when already in a non-prelisten session.
- Provides active-row visual feedback, which supports quick switching and stopping.
- Includes a manual integration checklist, which is appropriate for this cross-thread UI/network feature.

#### Concerns
- **HIGH**: The active preview state is still index-based (`prelistenRow`) and therefore vulnerable to server list refresh/reordering. The known issue is acknowledged but not fixed in the plan.
- **HIGH**: `setPrelistenState` matching by `host` alone is insufficient. Multiple rooms can share a host with different ports, so host-only matching can highlight or stop the wrong row.
- **HIGH**: The plan does not include the research-confirmed OSC/UI suppression work. Without that, prelisten may populate channel strips, broadcast room users through OSC, and falsely show the app as "Connected."
- **MEDIUM**: "Clear prelisten on browser dismiss" may conflict with the requirement if users expect preview audio to continue while they inspect other UI regions. This needs to be an explicit product decision, not an implicit cleanup.
- **MEDIUM**: Polling prelisten volume at 20 Hz is acceptable for a slider, but the plan does not explain ownership/source of truth. If both processor atomics and browser state exist, stale or flickering UI is possible.
- **MEDIUM**: The button-hit-testing approach in `listBoxItemClicked` can be fragile for accessibility, hover, and row-layout changes. It is workable, but easy to regress.
- **LOW**: No explicit loading/connecting state is defined, so a slow connection may leave the button/state ambiguous between click and confirmed audio start.
- **LOW**: The test checklist is mentioned but not shaped around the actual critical regressions found in research.

#### Suggestions
- Replace `prelistenRow` as the source of truth with `activePrelistenServer { host, port }`; derive row index from current server data during every `updateList()`.
- Match prelisten state by `host + port`, not host alone.
- Add explicit editor/browser handling for prelisten-only mode so:
  - connection bar does not show a normal session
  - channel strips are not repopulated from preview users
  - OSC output does not broadcast preview-room user/channel state
- Decide whether browser dismissal should stop preview. If yes, document it as an intentional UX rule; if no, remove that behavior from the plan.
- Add a transient `connecting` UI state for the Listen button so rapid clicking/switching is deterministic.
- Clarify volume ownership: ideally browser/editor owns the visible slider state, processor owns the atomic runtime value, and updates are one-way except on initialization.
- Expand the manual test plan to cover:
  - refresh while listening
  - same-host different-port rooms
  - switching rooms rapidly
  - joining a room while previewing
  - OSC/channel strip/connection-bar behavior during preview
  - browser dismiss behavior

#### Risk Assessment
**MEDIUM-HIGH**. The visible UX is well scoped, but state management is not robust enough yet, and the missing OSC/editor-side suppression work means the feature may appear functionally correct while still creating major UX regressions elsewhere in the app.

## Consensus Summary

*Single reviewer (Codex) — consensus analysis based on cross-plan patterns.*

### Agreed Strengths
- Architecture is sound: reusing main NJClient, explicit command/event plumbing, clean backend/UI split
- Good alignment with user decisions and success criteria
- Reasonable scope — no over-engineering
- Correct choice to avoid second NJClient instance

### Agreed Concerns
1. **Critical research findings are not integrated into task steps** — plans are acknowledged-but-not-fixed, making them not execution-ready
2. **State identity is inconsistent** — backend events use weak identifiers (`server_name` string), UI uses row indices, refresh handling is not robust
3. **Preview audio gain scaling is under-specified** — current approach scales entire plugin output, not just preview stream
4. **Cross-feature side effects are unaddressed** — OSC, channel strips, connection bar all leak prelisten state into normal UI
