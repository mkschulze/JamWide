## Flag Compatibility

| Option | Description | Selected |
|--------|-------------|----------|
| Visible instamode channel | Use flag 0x02 only. Other users see an extra channel called '_sync' but it sends silence. | ✓ |
| Zero-volume instamode | Use flag 0x02 with volume=0. Server relays, other clients play at zero volume. | |
| No probe channel | Skip instamode, measure via WebRTC timestamps. Less precise but zero protocol changes. | |

**User's choice:** Visible instamode channel (recommended)
**Notes:** Server code at usercon.cpp:581 confirms `0x80` filler flag sets `active=false` — server won't relay data for filler channels. Combined `0x82` approach from original design is not viable.

**Follow-up: Channel naming**
User clarified they want the channel to double as a **voice talkback** channel for real-time communication between musicians. Named "Instatalk". This reframes the feature from a hidden probe to a visible, useful voice chat channel where latency measurement is a free side effect.

**Follow-up: Talk mode**

| Option | Description | Selected |
|--------|-------------|----------|
| Push-to-talk | User holds a button/key to talk. Saves bandwidth, avoids background noise. | ✓ |
| Always-on (voice-activated) | Channel always sending with noise gate. | |
| Toggle on/off | Click to enable/mute. | |

**User's choice:** Push-to-talk (recommended)

## Timestamp Encoding

| Option | Description | Selected |
|--------|-------------|----------|
| Timing-only | Don't encode anything. Just measure WHEN instamode data arrives vs WHEN interval plays. | ✓ |
| GUID-based marker | Encode sender's interval counter in the 16-byte GUID of upload messages. | |
| Sideband WebSocket | Send timing data via VideoCompanion WebSocket instead of NINJAM. | |

**User's choice:** Timing-only (recommended)
**Notes:** Since the channel carries real voice audio (PTT), we can't corrupt the audio stream with encoded data. Timing-only is both simplest and compatible with voice content.
