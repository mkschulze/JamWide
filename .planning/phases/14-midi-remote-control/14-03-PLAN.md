<objective>
Wire OSC and UI remote-user change paths through APVTS so MIDI feedback and DAW automation stay synchronized. Currently, OSC and UI change remote user parameters via cmd_queue only, bypassing the new APVTS parameters added in Plan 01. This means MIDI controllers and DAW automation lanes show stale values when remote users are controlled via OSC or the mouse.

Purpose: Without this plan, MIDI-01 success criterion 3 ("Parameter changes in JamWide send MIDI CC feedback to the controller") fails for remote users when changes originate from OSC or the UI. This plan makes APVTS the single source of truth that all control paths update, with MidiMapper's timerCallback as the sole path to NJClient for remote group controls.

Output: Updated OscServer.cpp and ChannelStripArea.cpp that write to APVTS remote params, MidiMapper with public echo suppression API for cross-system coordination.
</objective>

**Step 3: Handle MidiMapper timerCallback double-dispatch prevention**

MidiMapper::timerCallback syncs APVTS -> NJClient via cmd_queue. When OscServer updates APVTS AND also pushes cmd_queue directly (Step 2), the same change could be dispatched twice. To prevent this, update MidiMapper::timerCallback to also track whether an external source already pushed the change:

The simplest approach: MidiMapper::timerCallback reads APVTS and compares to lastSynced arrays. If OscServer already pushed the cmd_queue AND updated APVTS, the APVTS value changed, so timerCallback will detect a diff and push ANOTHER cmd_queue entry. This is harmless -- SetUserStateCommand is idempotent (sets the value, doesn't increment). The NJClient just gets the same value twice.

However, to be clean, update MidiMapper::timerCallback to track the APVTS values it has observed. When it sees a change and dispatches, it updates its lastSynced. When OscServer updates APVTS, MidiMapper will see the change on next tick, but since it also updates lastSynced, subsequent ticks won't re-dispatch. This is the existing behavior and requires no additional code changes.

So: no changes needed to timerCallback. The double-push from OscServer (cmd_queue direct + APVTS -> timerCallback -> cmd_queue) results in one extra idempotent SetUserStateCommand which is harmless. The timerCallback's lastSynced arrays will converge on next tick.
