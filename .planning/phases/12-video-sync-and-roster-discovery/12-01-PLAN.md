Message types sent from plugin to companion over WebSocket (localhost:7170):

1. config (existing Phase 11):
   {"type":"config","room":"jw-abc123","push":"Daveguitar","noaudio":true,"wsPort":7170}

2. roster (existing Phase 11):
   {"type":"roster","users":[{"idx":0,"name":"Dave@guitar","streamId":"Daveguitar"},{"idx":1,"name":"Alice","streamId":"Alice"}]}
   NOTE: Each roster message is a FULL STATE REPLACEMENT, not a diff. The users array
   contains ALL currently connected users. Receivers must clear any previous state and
   rebuild from the new array. This handles add, remove, rename, and reorder atomically.

3. bufferDelay (NEW in Phase 12, D-04):
   {"type":"bufferDelay","delayMs":8000}
   - delayMs: integer, milliseconds, formula: floor((60.0 / bpm) * bpi * 1000)
   - Sent: on initial WS client connection (after config), and on every BPM/BPI change
   - Not sent: if bpm <= 0 or bpi <= 0 or bpm is NaN or video not active

Delivery order on initial client connection:
  1. config message
  2. bufferDelay message (if cached delay > 0)

Reconnect semantics: Companion page reconnects -> receives fresh config + bufferDelay.
Roster is sent on next roster change event (not cached for reconnect).

Terminology (addresses review concern: inconsistency):
- "derived room password": The SHA-256-derived string passed as VDO.Ninja password
  (NOT called "hash" in code or comments to avoid confusion with URL hash fragments)
- Computed as: hex(SHA-256(ninjam_password + ":" + room_id)).substring(0, 16)
- Passed via: companion URL fragment #password=DERIVED_VALUE
- Truncation justification: 16 hex chars = 64 bits. VDO.Ninja internally truncates
  passwords to 4 hex chars (16 bits) via SHA-256(encodeURIComponent(pw)+salt).substring(0,4).
  64 bits is 2^48 times stronger than VDO.Ninja's own truncation. The NINJAM session
  password is the primary access control; the derived password prevents casual unauthorized
  viewers from joining the VDO.Ninja room. Brute-forcing 64 bits is computationally infeasible.

juce::String VideoCompanion::deriveRoomPassword(const juce::String& password,
                                                const juce::String& roomId)
{
    if (password.isEmpty()) return {};  // D-07: no password for public rooms

    // D-06: SHA-256(ninjam_password + ":" + room_id)
    // Passed to VDO.Ninja as a password (not hash) per RESEARCH.md Open Question 1:
    // VDO.Ninja sees a password and handles its own internal hashing.
    // All JamWide users with same NINJAM password derive the same VDO.Ninja password.
    //
    // Truncation: 16 hex chars = 64 bits >> VDO.Ninja's internal 4 hex chars (16 bits).
    // See header comment for full rationale.
    juce::String input = password + ":" + roomId;
    juce::SHA256 sha(input.toUTF8());
    return sha.toHexString().substring(0, 16);
}
```

void VideoCompanion::broadcastBufferDelay(float bpm, int bpi)
{
    // D-02: Called on BPM/BPI change only, not on every beat
    // Addresses review concern: invalid BPM/BPI guards (zero, NaN, transitional state)
    if (bpm <= 0.0f || bpi <= 0) return;
    if (std::isnan(bpm)) return;  // Guard against NaN during session transitions
    if (!isActive()) return;      // Pitfall 6: guard if video not active

    // D-03: delay_ms = (60.0 / bpm) * bpi * 1000, truncated to integer
    // Integer truncation (not rounding) is deliberate -- sub-millisecond precision
    // is irrelevant for video buffering at multi-second intervals.
    cachedDelayMs_ = static_cast<int>((60.0 / static_cast<double>(bpm)) * bpi * 1000.0);

    // D-04: {"type":"bufferDelay","delayMs":N}
    juce::String json = "{\"type\":\"bufferDelay\",\"delayMs\":"
                        + juce::String(cachedDelayMs_) + "}";

    std::lock_guard<std::mutex> lock(wsMutex_);
    if (!wsServer_) return;
    auto clients = wsServer_->getClients();
    for (auto& client : clients)
        client->send(json.toStdString());
}
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Plugin -> Companion (WebSocket) | Plugin sends JSON to companion page over localhost WebSocket. Messages contain room config and buffer delay (no secrets). |
| Plugin -> Browser (URL) | Companion URL contains derived room password in fragment portion. Fragment never sent to server per RFC 3986 section 3.5. |
| NINJAM password -> SHA-256 | Password is hashed before any network transmission. Only the derived password appears in URLs. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-01 | Information Disclosure | VideoCompanion::buildCompanionUrl | mitigate | Derived room password appended as URL fragment (#password=), never query parameter (?). Fragment is browser-only per RFC 3986 section 3.5. Verified in code: url += "#password=" not "?password=". |
| T-12-02 | Information Disclosure | VideoCompanion::sendConfigToClient | accept | Config WebSocket message intentionally excludes password (carried forward from Phase 11 D-14). Buffer delay message contains only integer milliseconds -- no sensitive data. |
| T-12-03 | Spoofing | deriveRoomPassword | accept | SHA-256 with password+roomId salt makes password unpredictable without knowing NINJAM password. 16 hex chars = 64 bits entropy, which is 2^48 times stronger than VDO.Ninja's own internal 4 hex char (16 bit) truncation. An attacker who guesses the NINJAM password already has session access, so derived password guessing adds no additional attack surface. |
| T-12-04 | Tampering | WebSocket bufferDelay message | accept | Buffer delay is a non-critical visual parameter. Tampering would only affect video buffering timing, not audio or security. WebSocket is localhost-only (127.0.0.1 binding from Phase 11 T-11-01). |
</threat_model>
