# Phase 15.1 Plan 02: Atomic Promotion Summary

**One-liner:** Promote `m_beatinfo_updated`, `m_bpm`, `m_bpi`, and `m_interval_pos` to `std::atomic<int>` with release/acquire pairing on the synchronizing flag and relaxed reads/writes on the payload, eliminating `m_misc_cs` from the audio thread (closes CR-03) and closing the `m_interval_pos` UI/audio race (AUDIT line 421); document edge-triggered best-effort semantics per Codex L-10 and verify with a 4-test stress scaffold.

## What Was Built

### Header changes — `src/core/njclient.h`

Four new atomic fields placed in the existing "Lock-free status access for audio thread" region (lines 153-187), alongside `cached_status`:

```cpp
std::atomic<int> m_beatinfo_updated{0};
std::atomic<int> m_bpm{0};
std::atomic<int> m_bpi{0};
std::atomic<int> m_interval_pos{0};
```

Above the field declarations is a 21-line comment block documenting the **edge-triggered best-effort publication protocol** — including the explicit worked example that the reader sees the LATEST payload only and may miss intermediate updates, which is correct behavior because BPM/BPI are config values where only the most recent matters. This addresses Codex L-10 review feedback that previously the protocol could be misread as a last-value latch.

`SetIntervalPosition` was updated to use `m_interval_pos.store(pos, relaxed)`. The original bare-int declarations of all four fields were removed from their original blocks (lines 322-338) with breadcrumb comments pointing to the new home.

### Source changes — `src/core/njclient.cpp`

**`_reinit()` (lines 652-665):** Initial values set via `.store(..., relaxed)`. Same-thread-as-construction so relaxed is correct.

**`updateBPMinfo()` (lines 752-762):** `m_misc_cs.Enter/Leave` REMOVED. Payload writes are `relaxed`; the synchronizing flag `m_beatinfo_updated.store(1, release)` publishes the prior payload writes to any subsequent acquiring reader.

**`GetPosition()` (lines 762-779):** Now reads `m_interval_pos.load(relaxed)` — closes the previously-undefined-behavior UI-thread/audio-thread race noted at AUDIT line 421.

**`AudioProc()` interval-boundary block (lines 832-879):** `m_misc_cs.Enter/Leave` bracketing the `if (m_beatinfo_updated)` block REMOVED. The flag check is now `m_beatinfo_updated.load(acquire)`; on success, `m_bpm.load(relaxed)` and `m_bpi.load(relaxed)` see the values published by the matching release-store. The flag is cleared with `relaxed` (edge-clear; another publish may race past — that's fine, we'll see it next interval). `m_interval_pos` reads/writes inside this loop use a local variable cached from a single relaxed load, with `store(relaxed)` on the back-write.

**`process_samples()` metronome loop (line 2279):** Updated reference to use `m_interval_pos.load(relaxed)`.

## Atomic Ordering Rationale

| Operation | Ordering | Rationale |
|---|---|---|
| Writer payload (`m_bpm`, `m_bpi`) store | `relaxed` | The synchronizing flag write below provides happens-before; payload visibility piggybacks on the release. |
| Writer flag (`m_beatinfo_updated`) store of `1` | `release` | Synchronizes with the reader's `acquire` load; publishes payload writes that happened-before. |
| Reader flag load | `acquire` | Synchronizes with the writer's `release` store; payload reads that happen-after see the values the writer published. |
| Reader payload (`m_bpm`, `m_bpi`) load | `relaxed` | Already synchronized via the acquire on the flag; relaxed is sufficient and cheaper. |
| Reader flag clear (store 0) | `relaxed` | Edge-clear; we don't need to publish anything. Next writer publish will re-set with release. |
| `m_interval_pos` (audio-thread same-thread reader/writer) | `relaxed` | Same thread; ordering is not the concern — atomicity for the cross-thread `GetPosition` reader IS. |
| `m_interval_pos` (UI-thread reader in `GetPosition`) | `relaxed` | Reader doesn't need ordering with respect to anything else — only freedom from torn reads, which `std::atomic<int>` guarantees on natural alignment (lock-free for `int` per C++ standard). |

This pattern matches the project convention established by `cached_status` (writer at `juce/NinjamRunThread.cpp:661` uses release; reader at `juce/JamWideJuceProcessor.cpp:474` uses acquire).

## Edge-Triggered Best-Effort Semantics (Codex L-10 response)

The header comment block in `njclient.h` (lines 156-178) documents the protocol unambiguously:

> If the writer publishes 5 times between two reader runs, the reader's single observation will see the 5th (most recent) bpm/bpi and miss the 1st-4th. This is correct: BPM/BPI are config values, only the most recent matters; on_new_interval recomputes interval state from whatever the reader observed.
>
> This is NOT a last-value latch protocol (which would buffer every intermediate value); it is an edge-triggered "something has changed, here is the current state" signal.

The `test_missed_update_semantics` test verifies this empirically — and would fail if a future change accidentally introduces last-value-latch behavior (e.g., by trying to enqueue every BPM change on an SPSC queue).
