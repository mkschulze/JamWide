# v1.1 Design: OSC Remote Control + VDO.Ninja Video

**Date:** 2026-04-05
**Milestone:** v1.1
**Status:** Draft ã open research questions flagged

## Summary

JamWide v1.1 adds two independent features:

1. **OSC Server** ã full bidirectional OSC control of all plugin parameters, enabling remote control via TouchOSC (iPad/phone) or any OSC client
2. **VDO.Ninja Video** ã browser-based video companion synced to the NINJAM session's interval timing, with grid and per-user popout display modes

Both features share the JUCE plugin core but are architecturally independent ã either can ship without the other.

## Feature 1: OSC Server

### Architecture

- Built on JUCE's `juce_osc` module (no external dependencies)
- Follows the IEM Plugin Suite pattern: `OSCParameterInterface` with bidirectional UDP
- Receiver: realtime callback thread for incoming messages (low latency)
- Sender: timer-based (configurable interval, default 100ms) for outgoing state changes ã only sends changed values

### OSC Namespace

### User Index Stability

- Remote users are assigned stable indices (1-based) in join order
- When a user disconnects, their index is freed and reused by the next joiner
- On any roster change, the plugin broadcasts `/JamWide/remote/count` and `/JamWide/remote/{idx}/name` for all active indices
- TouchOSC layouts bind faders to fixed indices (1-8) and update labels from `/name` messages
- Username-based addressing (`/remote/Dave/volume`) is deferred to a future version

### Configuration

- OSC receiver port: configurable in plugin UI (default: 9000, range: 1000-15000)
- OSC sender: configurable target IP + port (for TouchOSC feedback)
- Send interval: configurable (1-1000ms, default: 100ms)
- Config persisted in plugin state (saved/restored with DAW session)
- UI: footer status indicator (IEM-style) with config dialog on click

IEM Plugin Suite (`/Users/cell/dev/IEMPluginSuite`):

- `resources/OSC/OSCUtilities.h` ã OSCReceiverPlus/OSCSenderPlus wrappers

## Feature 2: VDO.Ninja Video

### Architecture

Two communication paths between plugin and VDO.Ninja:

1. **Companion page** (browser): Plugin serves a local HTML page that embeds VDO.Ninja iframes. Controls video display (grid/popout) and relays `setBufferDelay` via `postMessage`.
2. **VDO.Ninja External API** (direct from plugin): Plugin connects to `wss://api.vdo.ninja:443` using an `&api=APIKEY` token. Can send commands (mic, camera, hangup, getGuestList) and receive events without going through the iframe. This is the primary control path.

The companion page handles video rendering; the external API handles roster discovery and command dispatch.

### VDO.Ninja URL Parameters

For music sync (from the music-sync-buffer-demo + docs research):
```
&chunked              ã custom chunked transport over WebRTC data channels (Chromium-only)
&noaudio              ã suppress audio playback (NINJAM handles audio)
&chunkbufferadaptive=0 ã fixed buffer, no drift (alias: &fixedchunkbuffer)
&chunkbufferceil=180000 ã allow up to 3 minutes of buffering
&buffer2=0            ã RTT compensated
&lowlatency           ã disable echo cancellation/AGC/noise suppression for music
&cleanoutput          ã remove all VDO.Ninja UI elements
&api=APIKEY           ã enable external API control from plugin
&room=ROOMNAME        ã join a specific room
&push=STREAMID        ã publish with a specific stream ID (use NINJAM username)
&password=PASS        ã room password (or &hash=HASH for URL-safe alternative)
```

### External API (wss://api.vdo.ninja)

The plugin connects directly to VDO.Ninja's API WebSocket for roster discovery and control:

```
Connect: wss://api.vdo.ninja:443 ã send {"join":"APIKEY"}
```

**Key commands:**
- `getGuestList` ã discover who's in the room (returns stream IDs + labels)
- `getStreamIDs` ã list all active streams
- `getDetailedState` ã full room state
- `mic`, `camera`, `speaker` ã per-stream or global control
- `volume` ã per-stream volume (0-200)
- `forceKeyframe` ã resolve sync issues
- `hangup` ã disconnect a stream

**Events received:**
- `guest-connected` (with `streamID` and `label`)
- `view-connection` / `push-connection` (connect/disconnect)

**Important:** WebSocket connections timeout every ~minute ã reconnection logic is required.

### Room Security

- VDO.Ninja supports `&password=PASS` (alphanumeric, case-sensitive, 1-49 chars)
- `&hash=HASH` alternative avoids plaintext in URLs (derived from password + room name salt)
- Placing credentials after `#` instead of `?` prevents them from being sent to the signaling server
- Plugin can derive the VDO.Ninja room password from the NINJAM server password

### Sync Mechanism

1. Plugin tracks NINJAM interval boundaries (BPM, BPI, interval count)
2. On each interval boundary, calculates target buffer delay:
   ```
   delay_ms = (60000 / BPM) * BPI * bars_to_delay + extra_offset_ms
   ```
3. Pushes `{ setBufferDelay: delay_ms }` to companion page via local WebSocket
4. Companion page forwards to VDO.Ninja iframe via `postMessage`
5. VDO.Ninja adjusts its chunked buffer to match

Per-stream delay is supported: `{ setBufferDelay: ms, streamID: "id" }`.

### Room Management

- **Auto-generated room ID**: Derived from NINJAM server address (e.g., `jamwide-ninbot-com-2049`)
- **User override**: Settings field to specify a custom room name
- Everyone on the same NINJAM server joins the same VDO.Ninja room by default
- Room capacity: VDO.Ninja default 80 guests (far above typical NINJAM sessions)

### User ã Stream Mapping

- Plugin uses NINJAM username as VDO.Ninja `&push=` stream ID for 1:1 correlation
- Roster discovery via external API: `getGuestList` returns `{streamID, label}` pairs
- On roster change, plugin broadcasts updated mapping to companion page + OSC clients

### Display Modes

**Grid Mode (default):**
- Single companion browser window
- All remote users displayed in a responsive grid
- VDO.Ninja `&view` parameter shows all room participants

**Popout Mode:**
- Each remote user gets their own browser window
- Windows can be dragged to separate monitors
- Triggered per-user via UI button or OSC (`/JamWide/video/popout/{idx}`)
- Each popout window hosts one VDO.Ninja iframe (`&view=streamID`)
- Individual `setBufferDelay` per stream

### Browser Requirements

- **Chunked mode requires Chromium-based browsers** (Chrome, Edge, Brave, Vivaldi)
- Firefox and Safari are not supported for chunked video
- Plugin should detect/warn if user's default browser is not Chromium-based

### Performance Considerations

- **Per-viewer encoding**: Each viewer costs CPU on the sender. In a 10-person jam, each user encodes 9 video streams. Document this for users.
- **TURN relay**: ~10% of users need TURN (free servers in CA, DE, US, FR/UK). Self-hosting is possible via `&turn=` parameter.
- **Chunked FEC**: `&chunkfec=4` adds forward error correction (1 parity per 4 data chunks) ã reduces visual glitches at bandwidth cost.

## Open Research Questions

These grey areas need investigation during the planning phase before implementation.

### Security

- [x] **Local server exposure**: Bind to `127.0.0.1` only. Add random auth token in URL query param to prevent other tabs from connecting.
- [ ] **Firewall prompts**: Will Windows Defender / macOS firewall prompt when binding to localhost? Needs testing on Windows.
- [x] **VDO.Ninja privacy**: STUN/TURN can expose IPs. Document in user-facing help. Offer `&relay` mode for privacy-sensitive users (forces TURN-only, higher latency).
- [x] **OSC security**: UDP OSC has no auth ã acceptable for local network. Document that OSC port should not be exposed to WAN.
- [x] **Room password security**: Use `&hash=` instead of plaintext `&password=` in URLs. Place credentials after `#` fragment to prevent server logging.

### Performance

- [ ] **Audio thread impact**: OSC + WebSocket servers must run on dedicated threads, never on the audio thread. Research JUCE threading patterns for this.
- [ ] **Memory overhead**: What's the footprint of juce_osc + a lightweight HTTP/WS server? Needs benchmarking.
- [ ] **Scalability**: Test with 10+ remote users (10 OSC parameter sets, 10 video streams).
- [ ] **Browser resource usage**: Multiple popout windows each running a VDO.Ninja iframe ã needs benchmarking. Consider documenting max recommended popout count.

### Latency & Sync

- [ ] **Sync chain jitter**: Plugin ã local WebSocket ã postMessage ã VDO.Ninja setBufferDelay. Expected to be <10ms on localhost ã needs verification.
- [ ] **Buffer delay accuracy**: Does VDO.Ninja's chunked mode maintain the exact requested delay, or does it drift? How often does `setBufferDelay` need to be refreshed?
- [ ] **OSC round-trip**: Typical UDP OSC round-trip on WiFi is 1-5ms. 100ms send interval should be sufficient for smooth fader feedback ã verify with TouchOSC.

### VDO.Ninja Specifics

- [x] **Stream ID discovery**: Solved ã use `getGuestList`, `getStreamIDs`, `getDetailedState` via the external API. Use NINJAM username as `&push=` stream ID for direct correlation.
- [ ] **Reconnection handling**: External API WebSocket drops every ~minute. Need robust reconnection with exponential backoff. Does the stream ID persist across reconnects?
- [x] **NAT traversal**: ~10% need TURN. Free TURN servers provided. Can self-host or use `&relay` to force TURN.
- [ ] **Video codec**: Chunked mode codec support is "a moving target". Default is VP8. Can use `&codec=h264` or `&codec=vp9`. Need to test which is most efficient for music jam video.

### Implementation

- [ ] **HTTP/WebSocket library choice**: Use JUCE's built-in networking, or embed a lightweight library (e.g., cpp-httplib, uWebSockets)? What's the smallest reliable option?
- [ ] **Companion page hosting**: Embed HTML/JS as binary resources in the plugin (no external files to manage). JUCE's `BinaryData` is ideal.
- [ ] **Multi-instance**: If user loads two JamWide instances, use dynamic port allocation (try default, increment if taken). Each instance gets its own API key.
- [ ] **Standalone vs plugin**: Local server should work identically in both modes ã verify no DAW sandboxing interferes.

## Decisions Made

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Milestone scope | Both OSC + video in v1.1 | Independent features, medium-sized, no conflicts |
| Video display | Hybrid: plugin button ã browser | Keeps plugin lightweight, browser handles rendering |
| Video modes | Grid (default) + per-user popout | Multi-monitor support for musicians |
| Room management | Auto-generate from NINJAM server + user override | Zero-config default, flexible for edge cases |
| OSC library | JUCE juce_osc | No external deps, proven in IEM suite |
| OSC namespace | Hierarchical with index-based remote users | Maps naturally to TouchOSC pages, stable indices |
| User identification | Index is stable address, name broadcast separately | TouchOSC layout stays consistent on roster changes |
| Video control via OSC | Yes | Single TouchOSC layout controls audio + video |
| OSC reference impl | IEM Plugin Suite pattern | Battle-tested across 20+ plugins, ~1000 LOC |
| Video sync reference | VDO.Ninja music-sync-buffer-demo | Purpose-built for NINJAM-style interval sync |
| Plugin ã VDO.Ninja control | External WebSocket API (`wss://api.vdo.ninja`) | Direct command/event channel without iframe middleman |
| Stream ID strategy | NINJAM username as `&push=` stream ID | 1:1 correlation between NINJAM and VDO.Ninja users |
| Room password | Use `&hash=` with `#` fragment, not plaintext `&password=` | Prevents server logging of credentials |
| Browser requirement | Chromium-only for chunked mode | Firefox/Safari don't support chunked transport |
