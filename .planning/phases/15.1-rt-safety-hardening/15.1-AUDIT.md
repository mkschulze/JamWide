## Findings

### CRITICAL (audio dropouts / xruns guaranteed if hit)

#### CR-01 — `m_users_cs.Enter()` (WDL_Mutex / pthread mutex) on audio thread
- **Sites**: `src/core/njclient.cpp:786` (top of `AudioProc`), `:2124` (`process_samples`), `:2723` (`on_new_interval`)
- *Why*: `WDL_Mutex` is a `pthread_mutex_t` (Linux/macOS) or `CRITICAL_SECTION` (Win); see `wdl/mutex.h:60-101`. Held by the run thread for arbitrary durations. Audio thread blocking on it is a classic priority-inversion source and the leading documented cause of the CPU-spike pattern this phase exists to fix.
- *Fix*: Per D-01, replace audio-thread reads with snapshot pulled from an SPSC queue produced on the run thread. The audio thread keeps a local `RemoteUser` array (or a `RemoteUserSnapshot` POD) and applies queued mutations at the top of `AudioProc` before the read loop. Run thread continues to use `m_users_cs` internally — it's only the audio thread that must not lock.

#### CR-02 — `m_locchan_cs.Enter()` (WDL_Mutex) on audio thread
- **Sites**: `src/core/njclient.cpp:1961` (top of `process_samples`), `:2698` (top of `on_new_interval`)
- *Why*: Same pthread-mutex priority-inversion class as CR-01. Held while UI thread mutates local channels via `SetLocalChannelInfo`, `DeleteLocalChannel`, etc.
- *Fix*: Same SPSC pattern as CR-01. The audio thread's view of local channels becomes a locally-owned array; UI/run thread queues mutations.

#### CR-03 — `m_misc_cs.Enter()` (WDL_Mutex) on audio thread
- **Sites**: `src/core/njclient.cpp:825` (interval boundary in `AudioProc`)
- *Why*: Mutex acquisition every interval boundary to read `m_beatinfo_updated` / `m_bpm` / `m_bpi`. Same priority-inversion class.
- *Fix*: These are scalar config values — promote `m_beatinfo_updated`, `m_bpm`, `m_bpi` to `std::atomic<int>` (matches the existing pattern used for `config_metronome*` and `m_encoder_fmt_requested` in `njclient.h:125-141`). No SPSC queue needed; D-01's gray-area note in CONTEXT explicitly allows atomic primitives for primitive-typed snapshot fields.
- **NOT in starting hypothesis** — picked up by exhaustive trace.

#### CR-04 — `writeLog(...)` on audio thread (lock + `vfprintf` to FILE*)
- **Sites**:
  - `src/core/njclient.cpp:2693` (`on_new_interval` — every interval, ~3-4s in typical session)
  - `src/core/njclient.cpp:2687` (inside `writeUserChanLog`, called from `:2467` in `mixInChannel` per-block in llmode advance, and `:2667` in tail recursion, and `:2753` in `on_new_interval`)
- *Why*: `writeLog` (`njclient.cpp:683`) acquires `m_log_cs` then calls `vfprintf` on a stdio FILE*. `vfprintf` takes libc's internal FILE lock and ultimately makes a `write()` syscall — both blocking I/O.
- *Fix*: Per D-03, eliminate these calls in production. The `interval N` log is diagnostic noise. Keep the `writeLog` API for the destructor/error-path use only (clearly labeled non-RT). Audio-path callers either remove the call or push a small POD onto a logging SPSC drained by the run thread that does the actual `vfprintf`.

#### CR-05 — `delete chan->ds` (heap dealloc) in `on_new_interval`
- **Sites**: `src/core/njclient.cpp:2743`
- *Why*: `~DecodeState()` (`njclient.cpp:168-177`) calls `delete decode_codec` (which destroys libvorbis/libflac state — significant `free()`s), `fclose(decode_fp)` (blocking syscall), and `decode_buf->Release()` (which may `delete this` on `DecodeMediaBuffer`, freeing internal queues). Worst-case heap activity per peer per interval.
- *Fix*: Per D-02, deferred-delete. Push the `DecodeState*` onto an SPSC queue at this site; run thread drains and runs the destructor. The audio thread still owns the *advance* of `chan->ds = chan->next_ds[0]` — only the dealloc is deferred.

#### CR-06 — `delete chan->next_ds[0]` (heap dealloc, conditional) in `on_new_interval`
- **Sites**: `src/core/njclient.cpp:2746`
- *Why*: Same `~DecodeState` cost as CR-05. Triggered when `(submask & chanpresentmask) & (1u<<ch)` is false — a discarded next-interval decoder.
- *Fix*: Same deferred-delete SPSC as CR-05. Same queue can hold both kinds of pointer.

#### CR-07 — `delete userchan->ds` (heap dealloc) in `mixInChannel` — multiple sites
- **Sites**: `src/core/njclient.cpp:2396, :2406, :2439, :2459, :2659`
  - `:2396` — sessionmode + transport stopped
  - `:2406` — sessionmode + seek/exhaustion
  - `:2439` — start_decode failure rollback
  - `:2459` — llmode advance to `next_ds[0]`
  - `:2659` — tail-recursion advance
- *Why*: Per-block heap activity. Same destructor cost as CR-05. The llmode site (`:2459`) is the one most-likely-hot — it fires every time a low-latency (Instatalk / voicechat) chunk completes, which is multiple times per second per active speaker.
- *Fix*: Same deferred-delete SPSC. All five sites collapse to a single push. Note: the planner must verify the destructor's `decode_buf->Release()` is safe to defer (`DecodeMediaBuffer` is refcounted; deferral just delays the final `delete this` if refcnt hits 0 — that's fine, it just happens later).
- **`:2396, :2406, :2439, :2459, :2659` NOT in starting hypothesis** — only `:2743` and `:2746` were listed. The mixInChannel sites are the higher-frequency ones.

#### CR-08 — `start_decode(...)` heap alloc + blocking file I/O on audio thread
- **Site**: called from `mixInChannel:2419` in sessionmode-rearm path
- *Why*: `start_decode` (`njclient.cpp:1870-1926`) does:
  - `new DecodeState` (`:1872`)
  - `WDL_String s` mutations including `Append(".XXXXXXXXX")`, `SetLen`, `Append(tmp)` — each may realloc the underlying `WDL_HeapBuf` (`:1887-:1898`)
  - `fopenUTF8(s.Get(), "rb")` — blocking syscall, possibly hits filesystem (`:1899`)
  - `new VorbisDecoder` or `new FlacDecoder` (`:1909, :1911`) — both heavy (libvorbis allocates internal state via `vorbis_synthesis_init` etc; libflac allocates a `FLAC__StreamDecoder` and inits stream)
  - `while (Available() <= 0) runDecode()` (`:1916`) — spins reading from the file
- *Fix*: This must move off the audio thread. Sessionmode pre-arm already happens through `next_ds[0]/[1]` — the run thread should be responsible for opening the next session-mode decoder and handing the constructed `DecodeState*` to the audio thread via SPSC, exactly the same way regular-interval `next_ds` works today. The audio thread's role becomes *advance only*, never construct.
- **NOT in starting hypothesis** — `start_decode` was not on the list. This is a sessionmode-only path so the user may not have hit it; it remains latent.

#### CR-09 — `lc->m_bq.AddBlock(...)` on audio thread (per-block in `process_samples`, per-interval in `on_new_interval`)
- **Sites in `process_samples`**: `:2010, :2025, :2031, :2044`
- **Sites in `on_new_interval`**: `:2708, :2717`
- *Why*: `BufferQueue::AddBlock` (`njclient.cpp:3562-3617`) takes `m_cs` twice, may `new WDL_HeapBuf`, may `Resize()` (realloc), and `m_samplequeue.Add()` grows a `WDL_PtrList` (heap-grow). Hits **every audio block** when `lc->bcast_active` (broadcasting) — which is the normal connected case.
- *Fix*: This is the biggest structural rework. Two options the planner should weigh:
  1. Replace `BufferQueue` with a JamWide-owned SPSC of fixed-size float-block records, sized for ≥ N blocks at the worst-case rate. Audio thread `try_push`es; encoder thread `try_pop`s. No locks, bounded heap (preallocated at `prepareToPlay`).
  2. Keep `BufferQueue`'s shape but replace internals: pre-allocate a pool of `WDL_HeapBuf` records at construction (sized for max block size), and let `AddBlock` only swap pre-allocated buffers via SPSC — failing/dropping when pool is empty rather than allocating.
- Option 1 is cleaner; option 2 is less invasive. Either way, the lock and the alloc both go.
- The wavebq site (`:2179`) shares this primitive — same fix applies.

#### CR-10 — `m_wavebq->AddBlock(...)` on audio thread
- **Site**: `src/core/njclient.cpp:2179`
- *Why*: Same `BufferQueue` as CR-09 — same lock + heap-alloc footprint. Gated on `waveWrite || (m_oggWrite&&m_oggComp)` (optional local recording / mixed-ogg output).
- *Fix*: Falls out of CR-09's BufferQueue replacement.

#### CR-11 — Decoder allocates inside `runDecode` chain (audio thread)
- **Sites**:
  - libvorbis path: `decode_codec->DecodeWrote(...)` → `vorbis_synthesis_pcmout` → `m_buf.Add(NULL, samples*vi.channels)` (`wdl/vorbisencdec.h:170, 207`) — `WDL_TypedQueue::Add` → `WDL_HeapBuf::Resize` → `realloc`.
  - libflac path: `FlacDecoder::DecodeGetSrcBuffer` → `m_inbuf.Add(nullptr, srclen)` (`wdl/flacencdec.h:182`) — same realloc path. `FlacDecoder::write_cb` → `m_outbuf.Add(nullptr, samples*nch)` (`wdl/flacencdec.h:289`) — same.
  - libvorbis/libogg internals also allocate via their own heap routines (`ogg_sync_buffer`, `vorbis_synthesis_blockin`).
- *Why*: Per-block heap growth on the audio thread. The `WDL_HeapBuf` allocator amortizes — once the queue has been grown to peak demand it tends to stay there, so the *typical* case is reuse. But the **first few blocks after a peer joins** and any **block-size jump** will re-allocate. Not always-hot, but always-latent.
- *Fix*: Two-step. Cheap step: force pre-allocation by calling `Prealloc(...)` on the codec's internal queues to a known maximum (e.g., `samples_per_channel * max_channels * sizeof(float)`) at decoder construction. Expensive step: the libvorbis/libflac internals are out of our reach — only mitigation is to keep `runDecode` off the audio thread entirely (run-thread pre-decode), but D-04 says don't restructure `mixInChannel`. Recommend the planner accept this as an out-of-scope risk and surface it in `15.1-RESEARCH.md` for a follow-up phase if profiling shows it matters.
- **NOT in starting hypothesis**.

#### CR-12 — `DecodeMediaBuffer::Read` lock on audio thread (network-stream decoder path)
- **Site**: `runDecode` (`njclient.cpp:261`) calls `decode_buf->Read(srcbuf, sz)`; `DecodeMediaBuffer::Read` (`njclient.cpp:129-141`) takes `mutex.Enter()`.
- *Why*: Another WDL_Mutex acquisition on audio thread. The producer (run-thread / network thread writing decoded packets) may hold this mutex briefly, but it's still an unbounded pthread mutex from the audio thread's perspective.
- *Fix*: Replace `DecodeMediaBuffer` with an SPSC byte-queue (similar to CR-09's BufferQueue replacement). Or, since the audio thread is the *only consumer* and the run/network thread is the *only producer*, the existing `spsc_ring.h` byte variant can host this directly.
- **NOT in starting hypothesis**.

### HIGH

#### H-01 — `JAMWIDE_DEV_BUILD` `fopen("/tmp/jamwide.log", "a")` block on audio thread
- **Site**: `src/core/njclient.cpp:2133-2139`
- *Why*: Even though gated by `m_debug_logged_remote` (one-shot per session) and only compiled in dev builds, this is `fopen + fprintf + fclose` directly on the audio thread. Any stat-cache miss at `/tmp` will pause audio. The "one-shot" guard means it fires once per session-connect — which is exactly when CPU is otherwise busy starting decoders.
- *Fix*: Per D-03, delete the block unconditionally. It's been there since dev bringup and provides no ongoing diagnostic value.

#### H-02 — `writeUserChanLog` synchronous formatting + writeLog
- **Sites called from**: `:2467` (mixInChannel llmode advance), `:2667` (mixInChannel tail-recursion advance), `:2753` (on_new_interval per-channel)
- *Why*: Even ignoring the `writeLog` lock+vfprintf (already CR-04), the formatting alone — `guidtostr` (16× snprintf), two `lstrcpyn_safe` of `WDL_String::Get()` results, character sanitization loops — is hundreds of cycles per call. At llmode rates that's nontrivial overhead even before the syscall.
- *Fix*: Per D-03, eliminate these in production. If the planner decides any are useful (e.g., "user X swapped to ds N" for one-shot debug), route through a single SPSC of fixed-size POD log records drained off-thread. Otherwise just delete.

#### H-03 — `lc->cbf(src, len, lc->cbf_inst)` plugin-channel callback on audio thread
- **Site**: `src/core/njclient.cpp:1996` (inside `process_samples`'s locchannel loop)
- *Why*: This is a function-pointer callback registered via `SetLocalChannelProcessor` (`njclient.h:214`). The audit cannot prove the registered processor is RT-safe — it's a third-party extension point. JamWide doesn't appear to register one today, but the API surface allows it.
- *Fix*: Document the contract on `SetLocalChannelProcessor`: "callback runs on the audio thread; must be RT-safe." If JamWide never uses it, consider removing the API entirely. Out of scope for the queue-migration sub-plans, but worth the planner's awareness.
- **NOT in starting hypothesis**.

#### H-04 — `DecodeState::runDecode` performs `fread` on audio thread (file path)
- **Site**: `src/core/njclient.cpp:256` (`fread(srcbuf, 1, sz, decode_fp)`)
- *Why*: Audio thread reads from a `FILE*` opened by `start_decode`. Each call is `sz=256` bytes (default in `mixInChannel:2503`), buffered by libc, but every libc-buffer-fill triggers a `read()` syscall and possibly a disk seek. This is the pre-existing NINJAM session-mode path; rare in live JamWide use but still on the hot path when sessionmode files are present.
- *Fix*: Strongly tied to CR-08's recommendation — move all `start_decode`-initiated decoders off the audio thread, using the run-thread-fills-`next_ds` pattern that already exists for live intervals.
- **NOT in starting hypothesis**.

### MEDIUM

#### M-01 — `tmpblock.Resize(bytelen)` may realloc on audio thread
- **Site**: `src/core/njclient.cpp:1982` (inside `process_samples` locchannel loop)
- *Why*: `tmpblock` is a member `WDL_HeapBuf` (`njclient.h:357`). The check `if (tmpblock.GetSize() < bytelen)` only resizes on growth. Once the audio block size stabilizes, this is a no-op. But: the first few blocks per session, or any host-driven block-size change, may realloc. JUCE hosts can change block size mid-stream (e.g. ProTools, FL).
- *Fix*: Pre-grow `tmpblock` in `prepareToPlay`-equivalent. There's no NJClient-level prepareToPlay, but you can size it in `_reinit` or `Connect` to max expected block size × `sizeof(float)`. JUCE caller already knows the max — pass it through.

#### M-02 — `WDL_TypedQueue::Compact` may realloc on audio thread (decoder Skip path)
- **Site**: indirectly via `decode_codec->Skip(...)` — `m_buf.Advance(amt); m_buf.Compact();` (`vorbisencdec.h:191-195`).
- *Why*: `WDL_Queue::Compact` (`wdl/queue.h:110-128`) calls `m_hb.Resize(olen, allocdown)` after a memmove. With the default `allocdown=false`, Resize generally won't shrink — but the *condition* it triggers (`m_pos > olen/2`) is hit every block once the queue is steady, so the Resize() runs every call. Verifying `Resize` with newsize equal to current m_size and no allocdown is a fast path (returns immediately), so this is borderline.
- *Fix*: Audit assertion: trace through one block to confirm steady-state Resize is the no-op fast path. If yes, downgrade to LOW. If no, eliminate `Compact()` calls from audio-thread Skip path (it's purely a memory-tidy operation; the queue continues to function without it, just with looser memory bounds).
- **NOT in starting hypothesis**.

#### M-03 — `juce::AudioBuffer::setSize` with `avoidReallocating=true` on audio thread
- **Sites**: `juce/JamWideJuceProcessor.cpp:226` (`inputScratch.setSize`), `:484` (`outputScratch.setSize`)
- *Why*: With `avoidReallocating=true`, JUCE skips realloc when the new size fits within current allocation. Both buffers are pre-allocated in `prepareToPlay`. This is RT-safe **only if** the host never asks for a bigger block than `prepareToPlay` was told about. JUCE sometimes lies about that; the safety net then *would* allocate. The current code already uses `avoidReallocating=true` so this is best-effort RT.
- *Fix*: No change needed if the prepareToPlay-allocated size is sized to the JUCE host's maximum expected block (which is `getMaximumExpectedSamplesPerBlock`, allocated once). Worth a one-line assertion in debug builds that `numSamples <= prevPreparedSize`.

#### L-01 — Recursive call in `mixInChannel`

- *Why*: Tail-recursive in some paths; depth bounded by `next_ds[2]` chain (max 2 advances per call). Not RT-unsafe in itself, but prevents the compiler from inlining the function and adds a few hundred cycles of stack manipulation. Just noting.

## Proposed sub-plan breakdown (15.1-02 → 15.1-07)

The decisions in CONTEXT (D-05) sketched a sequence, but the auditor's findings let us refine groupings. Recommended ordering, **landing one sub-plan per commit per D-06**:

### 15.1-02 — Atomic config promotion (warm-up; smallest blast radius)
- **Findings**: CR-03 only.
- **Why first**: Trivial mechanical change (add `std::atomic`, swap reads/writes). Builds confidence in the pattern. No SPSC infrastructure needed.
- **Verification**: TSan run + listening test; auditor re-run should clear CR-03.

### 15.1-03 — Eliminate audio-path logging
- **Findings**: CR-04, H-01, H-02, L-02.
- **Why second**: Independent of state-sync queues. Per D-03 these are simply deleted (with `writeLog` API kept for non-RT callsites only). No new infrastructure.
- **Risk**: Loss of post-mortem visibility into interval transitions. Acceptable per D-03 — interval N spam is not actionable.
- **Verification**: TSan + auditor re-run + grep confirms no `writeLog` / `writeUserChanLog` / `fopen` callers reachable from audio path.

### 15.1-04 — SPSC infrastructure (spike + helpers; foundation for the rest)
- **Findings**: none directly closed; required by 15.1-05 / 15.1-06 / 15.1-07 / 15.1-08.
- **Scope**: Add the typed-payload helpers / patterns over `spsc_ring.h` that subsequent sub-plans need: a generic deferred-delete `SpscRing<void*, N>` (or per-type variants), a state-update record format, unit tests for the new wrappers. No production code path is changed; just the primitives lands.
- **Why before 15.1-05**: Lets the per-violation sub-plans focus on integration, not primitive design.
- **Verification**: New unit tests pass under TSan.

### 15.1-05 — Deferred-delete for DecodeState (closes the highest-frequency dealloc)
- **Findings**: CR-05, CR-06, CR-07.
- **Scope**: At every `delete chan->ds` / `delete chan->next_ds[0]` / `delete userchan->ds` site, push the pointer onto a deferred-delete SPSC. Run thread drains at its 20ms tick (per CONTEXT) and runs the actual destructor.
- **Why before the big mutex sub-plans**: The dealloc removal is mechanical and isolated; it doesn't depend on the snapshot infra that 15.1-06 / 15.1-07 introduce. Lands fastest user-visible win.
- **Verification**: TSan (proves no race on the queue or on the deferred destruction order) + listening test + auditor re-run.

### 15.1-06 — Replace `m_locchan_cs` audio-thread reads with snapshot
- **Findings**: CR-02 (both sites).
- **Scope**: Audio thread holds a local `LocalChannelSnapshot[]`. Run thread (or UI thread, via run thread's serialization) pushes mutations through SPSC. `process_samples` and `on_new_interval` consume the snapshot.
- **Why before 15.1-07 (m_users_cs)**: Local channels are simpler — fewer fields, owned by one local user. Validates the pattern before applying to the (larger) RemoteUser table.
- **Verification**: TSan + listening test (especially the case "DeleteLocalChannel during connect"; the JamWide UI does this) + auditor re-run.

### 15.1-07 — Replace `m_users_cs` audio-thread reads with snapshot + BufferQueue replacement
- **Findings**: CR-01, CR-09, CR-10, CR-12.
- **Scope**:
  1. Audio thread holds a local `RemoteUserSnapshot[]`. Run thread queues mutations.
  2. Replace `BufferQueue` with a fixed-pool SPSC variant (per the CR-09 fix). Both `lc->m_bq` and `m_wavebq` use the new primitive.
  3. Replace `DecodeMediaBuffer` with an SPSC byte queue (CR-12).
- **Why grouped**: All three are co-affected by the m_users_cs path. Splitting risks intermediate states where one half locks and the other half snapshots. BufferQueue replacement is the largest single change in the phase; bundling it with the m_users_cs swap means one big rewrite, one verification pass.
- **Optional split**: If the planner wants smaller commits, this can be 15.1-07a (m_users_cs snapshot) and 15.1-07b (BufferQueue + DecodeMediaBuffer), in that order.
- **Verification**: TSan + populated-server UAT (3+ peers, peer churn, broadcast on/off toggling) + auditor re-run. This is the highest-risk sub-plan.

### 15.1-08 — Pre-allocation hardening + tmpblock
- **Findings**: M-01, M-02 (if confirmed), M-03 (assertion only), CR-11 (mitigation only).
- **Scope**: `tmpblock.Prealloc(...)` in `_reinit`; `Prealloc` on decoder internal queues at construction; debug-build assertion that `numSamples <= prevPreparedSize` in `processBlock`.
- **Why last**: Smallest, lowest-risk; cleans up the medium findings after the architectural rework lands. Can also be deferred if priority is the critical findings.
- **Verification**: Auditor re-run; allocator counters via Instruments (optional).

### 15.1-09 — Codec call-site integration (IN scope, post-Codex realignment)

**Originally written as "deferred / out of scope." Promoted into the phase after the Codex adversarial review found the deferral conflicted with the success criterion "auditor reports zero CRITICAL findings on the audio path."** These are **call-site integration** fixes, not codec-algorithm changes — they remove allocating/blocking codec calls *from the audio thread* without touching the codec's internals.

- **CR-08** (`start_decode` on audio thread, sessionmode): move sessionmode pre-arm to the run thread. Audio thread sets a `kInstaArmRequested` atomic flag (or an SPSC enqueue); run thread sees it next tick, calls `start_decode`, and atomically swaps in the new state. Sessionmode-only path; isolated change, no codec edits.
- **H-04** (`fread` in `DecodeState::runDecode`): subsumed by CR-08's fix — once `start_decode` and the sessionmode read loop run on the run thread, the `fread` is no longer on the audio thread by construction.
- **CR-11 partial** (decoder internal allocations): mitigation via `Prealloc` lands in 15.1-08. The remaining algorithmic allocations inside libvorbis/libflac (where the codec implementation itself decides to grow buffers) are **out of scope** — that would be a codec-algorithm change. The audit's runtime measurement during 15.1-10 verification will confirm whether `Prealloc` is sufficient or whether the residual allocations are frequent enough to matter; if not sufficient, a follow-up phase covers the algorithmic change.

### 15.1-10 — Verification (IN scope)

- Auditor re-run; CRITICAL count must be zero.
- Manual UAT under TSan standalone with 3+ peers.

### Truly deferred / out-of-scope follow-ups (NOT in Phase 15.1)

Out of scope per CONTEXT.md "codec algorithm/format changes." Would be a separate phase if 15.1-10 verification finds it material.
