# Phase 14.2: Instamode Video Sync - Research

**Researched:** 2026-04-15
**Domain:** NINJAM protocol (instamode channels), JUCE audio pipeline, VDO.Ninja video buffering
**Confidence:** HIGH

## Summary

This phase adds an instamode channel named "Instatalk" (flag `0x02`) to measure the actual delay between realtime audio arrival and interval-buffered audio playback, then sends this measured delay to the VDO.Ninja companion page to calibrate video buffering. The channel doubles as a push-to-talk voice talkback feature.

The NINJAM codebase already has full instamode support: sender-side encoding with 64-byte blocks (`LIVE_ENC_BLOCKSIZE2`), receiver-side decoding with 128-byte prebuffer (`LIVE_PREBUFFER`), and per-channel flag-based routing that skips interval boundaries. The server relays all channel data to subscribed users regardless of the `0x02` flag -- only the `0x80` flag suppresses relay by setting `active=false`. This confirms the CONTEXT.md decision (D-01) that `0x02` alone is correct.

The companion page already handles `setBufferDelay` via VDO.Ninja's chunked mode (`&chunked` + `&buffer=N` URL params + postMessage). VDO.Ninja manages an internal per-stream buffer. The new companion work is limited to: per-tile black overlay with fade-in transition (D-09), footer sync mode indicator (D-12), and integrating the three-tier delay priority model (D-11).

**Primary recommendation:** Implement in two plans: (1) Plugin-side instamode channel + measurement + push-to-talk audio routing, (2) Companion-side per-tile fade-in + sync mode footer indicator.

## User Constraints (from CONTEXT.md)

### Locked Decisions
- **D-01:** The probe channel is a visible instamode channel named "Instatalk" with flag `0x02` (instamode only, NOT `0x80` filler -- the server marks filler channels inactive and won't relay data).
- **D-02:** The channel serves dual purpose: voice talkback (push-to-talk) and latency probe. PTT sends voice audio; when PTT is released, the channel sends silence. Probe timing works regardless of voice activity because instamode data flows continuously.
- **D-03:** Push-to-talk mode -- user holds a key/button to talk. Saves bandwidth, avoids background noise. The channel is always transmitting (silence or voice) so the probe measurement is always available.
- **D-04:** Other NINJAM users will see the "Instatalk" channel in their mixer. It plays near-realtime audio (voice or silence). This is expected and acceptable -- the channel provides real functionality (voice chat).
- **D-05:** Timing-only measurement -- no data encoded in the audio stream. The receiver simply compares wall-clock timestamps: `t_insta` = when instamode data arrives from a remote user, `t_interval` = when that user's interval audio starts playing. `measured_delay = t_interval - t_insta`. This requires zero protocol changes and works with any audio content on the instamode channel.
- **D-06:** Single measurement -- use the first clean measurement immediately. Fastest calibration (~1 interval). No warmup discard, no averaging.
- **D-07:** Trust the measurement unconditionally -- if the measured delay differs wildly from the BPM/BPI calculation, the measurement is MORE correct because it reflects actual network conditions. No sanity-check fallback.
- **D-08:** Measurement happens automatically when video is active and instamode data is received from any remote user. No explicit "calibrate" step needed.
- **D-09:** Per-tile black + fade-in. Each remote user's video tile starts black with a subtle syncing indicator. Once that user's buffer fills to the measured delay, their tile fades in independently. Users appear one by one as their buffers fill.
- **D-10:** Ring buffer per remote user in the companion page. Buffer size = `measured_delay_ms / frame_interval_ms` frames.
- **D-11:** Three-tier delay priority: manual slider (highest) > auto-measured (instamode probe) > auto-calc (BPM/BPI formula, lowest). Default to auto-measured when probe data is available. Manual slider always wins if user touches it.
- **D-12:** Companion footer shows active mode: "Sync: measured (Xms)" or "Sync: calculated (Xms)" or "Sync: manual (Xms)".
- **D-13:** If no instamode data is received (no remote user has a JamWide client with Instatalk), fall back to BPM/BPI calculation (current Phase 12 1 behavior). The companion page doesn't need to know which method produced the delay -- it receives the same `{"type":"bufferDelay","delayMs":N}` message either way.

### Pattern 1: Adding a Local Instamode Channel
**What:** Create a new local NINJAM channel with the instamode flag at connect time
**When to use:** When the Instatalk feature should be active (always on connect)
**Example:**
```cpp
// Source: [VERIFIED: NJClient::SetLocalChannelInfo at njclient.cpp:3083]
// In NinjamRunThread::handleStatusChange(), after setting up channels 0-3:

// Channel index 4 = Instatalk (instamode voice talkback + latency probe)
// flags=0x02 selects instamode encoding (64-byte blocks, 128-byte prebuffer)
// src_channel: mono input from first input pair (bit 10 NOT set = mono)
// broadcasting=true: always transmitting (silence when PTT off)
client->SetLocalChannelInfo(4, "Instatalk",
    true, 0,              // src_channel=0 (first input, mono)
    true, 64,             // bitrate=64 kbps (low bandwidth for voice)
    true, true,           // broadcasting=true (always active)
    false, 0,             // outch not needed
    true, 0x02);          // flags=0x02 (instamode)

client->NotifyServerOfChannelChange();
```

**What:** Priority system for delay source: manual > measured > calculated
**When to use:** In VideoCompanion when deciding what delay to broadcast

void VideoCompanion::broadcastBufferDelay(float bpm, int bpi, int measuredDelayMs)
{
    // D-11: Three-tier priority
    // Manual slider (handled companion-side, always wins if user touched it)
    // Measured (instamode probe) -- use if available
    // Calculated (BPM/BPI formula) -- fallback

    int delayMs;
    SyncMode mode;
    if (measuredDelayMs > 0) {
        delayMs = measuredDelayMs;
        mode = SyncMode::Measured;

        mode = SyncMode::Calculated;

    // {"type":"bufferDelay","delayMs":N,"syncMode":"measured"|"calculated"}

### Pattern 4: PTT Audio Routing (Audio Thread)
**What:** Route input audio or silence to the Instatalk channel based on PTT state
**When to use:** In processBlock, before NJClient::AudioProc
**Example:**
```cpp
// Source: [VERIFIED: njclient.cpp:1986-1989, SetLocalChannelProcessor API]
// NJClient supports per-channel audio processors via SetLocalChannelProcessor.
// The callback is called from process_samples() for each local channel.
// PTT can be implemented by having the callback either pass through audio
// or zero the buffer based on an atomic bool.

// In NinjamRunThread::handleStatusChange (on connect):
client->SetLocalChannelProcessor(4,
    [](float* buf, int ns, void* inst) {
        auto& proc = *static_cast<JamWideJuceProcessor*>(inst);
        if (!proc.pttActive.load(std::memory_order_relaxed)) {
            // PTT not pressed: send silence
            memset(buf, 0, ns * sizeof(float));
        }
        // else: pass through input audio unchanged
    },
    &processor);
```

### Anti-Patterns to Avoid
- **Encoding data in audio frames:** D-05 explicitly rejects this. Wall-clock timing comparison is simpler and works regardless of audio content.
- **Using `0x80` filler flag:** The NINJAM server sets `active=false` for filler channels (verified at `usercon.cpp:581`). Other clients won't auto-subscribe, breaking data relay. Use `0x02` only.
- **Averaging multiple measurements:** D-06 specifies single measurement. First clean measurement is used immediately. Averaging adds latency to calibration.
- **Locking in the audio thread:** Use atomics for cross-thread measurement data. Never acquire mutexes in processBlock or process_samples.
- **Custom ring buffer in companion:** VDO.Ninja's chunked mode (`&chunked` + `setBufferDelay`) handles per-stream buffering internally. Do not re-implement.

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Audio encoding for instamode | Custom encoder | NJClient's built-in Vorbis encoder path | Instamode flag `0x02` automatically selects 64-byte block size and 128-byte prebuffer. The encoding path at njclient.cpp:1718-1719 handles this transparently. |
| Video ring buffer | Canvas capture / frame queue | VDO.Ninja's `&chunked` mode + `setBufferDelay` | VDO.Ninja manages internal per-stream buffers with frame-level accuracy. Supports delays of minutes. Already working via Phase 12.1. |

**Key insight:** NJClient's instamode implementation is complete and battle-tested. The sender and receiver paths handle encoding block sizes, prebuffering, and continuous streaming automatically based on the `0x02` flag. No protocol changes or custom encoding needed.

## Common Pitfalls

### Pitfall 1: Channel Index Collision with Existing Local Channels
**What goes wrong:** Using channel index 0-3 for Instatalk overwrites a user's audio channel.
**Why it happens:** JamWide uses channels 0-3 for local audio (set up in NinjamRunThread.cpp:316-331). The server's `m_max_localch` limits the total.
**How to avoid:** Use channel index 4 for Instatalk. Verify `GetMaxLocalChannels()` returns >= 5 (the server default is 32 per `mpb_server_auth_reply.maxchan`). [VERIFIED: njclient.h:208, mpb.h:64]
**Warning signs:** "Channel 4 not transmitting" in logs, or existing audio channels silenced.

### Pitfall 2: Measurement Must Use the Same Remote User
**What goes wrong:** Measuring t_insta from user A's instamode channel and t_interval from user B's regular channel gives meaningless data.
**Why it happens:** Different users have different network latencies.
**How to avoid:** Track both timestamps per-user. Only compute delay when both t_insta and t_interval exist for the SAME remote user. Use user index (not name) as the key.
**Warning signs:** Wildly varying measurements, or delay values that change when users join/leave.

### Pitfall 3: on_new_interval() Fires Before Audio Actually Plays
**What goes wrong:** `on_new_interval()` fires at the interval boundary, but the DecodeState for regular channels may not have arrived yet (still downloading).
**Why it happens:** The interval boundary marks when playback SHOULD start, but the audio data may still be in transit. The DecodeState is only assigned when `startPlaying()` is called after `config_play_prebuffer` bytes arrive.
**How to avoid:** Record t_interval when the regular channel's `ds` is first consumed in `mixInChannel` (when `chan->ds` transitions from null to non-null at an interval boundary), not in `on_new_interval()` itself. Alternatively, record t_interval at `on_new_interval()` and accept that it's an approximation (the interval boundary IS when NJClient intends to start playing, even if actual audio lags by a few ms).
**Warning signs:** Measured delay is consistently shorter than expected (because t_interval is recorded before audio actually plays).

### Pitfall 4: Audio Thread Allocation
**What goes wrong:** Creating std::string, allocating memory, or acquiring mutexes on the audio thread causes glitches or priority inversion.
**Why it happens:** The measurement code runs in process_samples (audio thread).
**How to avoid:** Use `std::atomic` for all measurement state. Store per-user measurement data in pre-allocated arrays indexed by user index (0-31, since MAX_USER_CHANNELS=32). No dynamic allocation.
**Warning signs:** Audio dropouts when measurement code runs, DAW showing high audio thread usage.

### Pitfall 5: Instamode Channel Not Removed on Disconnect
**What goes wrong:** Instatalk channel lingers in NJClient after disconnect, causing state corruption on reconnect.
**Why it happens:** SetLocalChannelInfo creates channels but they're not automatically cleaned up.
**How to avoid:** Call `DeleteLocalChannel(4)` in the disconnect path, or let NJClient's destructor handle it. Verify that NinjamRunThread's existing disconnect logic covers this (channels 0-3 are not explicitly deleted either -- NJClient handles teardown internally on disconnect).
**Warning signs:** Stale Instatalk channel visible on reconnect, or "channel already exists" errors.

### Pitfall 6: RemoteChannelInfo Missing flags Field
**What goes wrong:** Cannot identify which remote channels are instamode from the snapshot API.
**Why it happens:** `NJClient::RemoteChannelInfo` struct does not include a `flags` field (verified at njclient.h:175-188). `GetRemoteUsersSnapshot()` copies everything except flags.
**How to avoid:** Add `int flags = 0;` to `RemoteChannelInfo` and populate it in `GetRemoteUsersSnapshot()` (njclient.cpp:2800, where other fields are copied from `RemoteUser_Channel`).
**Warning signs:** Unable to detect instamode channels from remote users in the run thread.

### Pitfall 7: PTT Callback Must Not Capture by Reference
**What goes wrong:** SetLocalChannelProcessor's callback outlives the captured reference, causing use-after-free.
**Why it happens:** The callback is a C function pointer with void* user data. If the processor is destroyed while the callback is still registered, the void* becomes dangling.
**How to avoid:** Use the processor pointer as user_data (same pattern as ChatMessage_Callback). Ensure channel is deleted before processor is destroyed (already handled by NJClient destruction order).
**Warning signs:** Crash in audio thread after plugin unload.

### Pitfall 8: Companion-Side Manual Mode Must Override Measured Delay
**What goes wrong:** The three-tier priority (D-11) is implemented plugin-side but the companion's manual slider doesn't properly override.
**Why it happens:** The companion's Phase 12.1 dual-state model tracks `manualMode` separately. If the plugin sends a measured delay, the companion must respect its own manual override.
**How to avoid:** The companion already has this: `manualMode` prevents `setLastAutoDelay()` from updating `activeDelayMs`. The measured delay just becomes a better source for the "auto" value. The companion doesn't need to distinguish measured vs calculated -- both are "auto" from the companion's perspective. D-13 confirms the same `bufferDelay` message type is used.
**Warning signs:** Slider position changes when measurement arrives despite user having manually set it.

## Open Questions (RESOLVED)

1. **Where exactly to hook t_insta measurement** — RESOLVED: Use mixInChannel ds assignment path (audio thread). When `llmode && userchan->next_ds[0]` transitions, capture wall-clock via atomic. This is the actual audio output moment.
   - What we know: Remote instamode data enters via `mixInChannel` with `llmode=true`. The `next_ds[0]` queue advances inline (not at interval boundary).
   - What's unclear: Best hook point -- when `startPlaying()` first queues the DecodeState, or when `mixInChannel` first consumes it? The `startPlaying` path (njclient.cpp:3411) happens in Run() thread; the mixing happens in audio thread.
   - Recommendation: Use the audio thread path. In `mixInChannel`, when `llmode && !chan && userchan->next_ds[0]` transitions to having data for the first time after an interval, record the wall-clock time via a `std::atomic<int64_t>`. This ensures the measurement is at the actual audio output moment.

2. **Whether to add measurement hooks inside NJClient or externally** — RESOLVED: Add minimal measurement state (atomics) directly into NJClient. Less invasive than callbacks, consistent with existing JamWide fork pattern.
   - What we know: NJClient is shared upstream code (from NINJAM). Modifying it requires careful consideration.
   - What's unclear: Whether to add a callback/hook mechanism to NJClient for timestamp recording, or to detect instamode activity externally from NinjamRunThread.
   - Recommendation: Add minimal measurement state to NJClient (atomic timestamps per remote user channel). This is less invasive than a callback and stays within the existing pattern of NJClient managing decode state. The JamWide fork already modifies njclient.h (atomic config fields, RemoteUserInfo, etc.).

3. **Per-tile fade-in implementation in companion page** — RESOLVED: Global overlay (single full-frame black overlay, fade-out after buffer fills). Per-tile impossible due to cross-origin iframe. D-09 updated to accept global overlay.
   - What we know: VDO.Ninja renders video in an iframe. The companion page cannot directly access individual video elements inside the iframe (cross-origin restriction).
   - What's unclear: How to overlay black tiles on specific VDO.Ninja streams and fade them independently.
   - Recommendation: Use CSS overlay divs positioned over the VDO.Ninja iframe, one per expected roster entry. Since VDO.Ninja's exact layout depends on number of participants, the overlay positioning needs to match VDO.Ninja's grid algorithm (which is a standard equal-size grid for rooms).
