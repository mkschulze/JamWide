# Phase 14.2: Instamode Video Sync - Context

**Gathered:** 2026-04-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Accurate audio-video synchronization using NINJAM's instamode channel as a latency probe. The instamode channel doubles as a voice talkback channel ("Instatalk") for low-latency communication between musicians. Video buffering in the companion page is calibrated by measuring the actual time difference between instamode audio arrival and interval-buffered audio playback, replacing the theoretical BPM/BPI calculation with a real network measurement. The companion page buffers video per-tile with a black-to-fade-in transition.

Does NOT include: changes to the NINJAM protocol, VDO.Ninja API integration, or new codec support.

</domain>

<decisions>
## Implementation Decisions

### Channel Design
- **D-01:** The probe channel is a visible instamode channel named "Instatalk" with flag `0x02` (instamode only, NOT `0x80` filler — the server marks filler channels inactive and won't relay data).
- **D-02:** The channel serves dual purpose: voice talkback (push-to-talk) and latency probe. PTT sends voice audio; when PTT is released, the channel sends silence. Probe timing works regardless of voice activity because instamode data flows continuously.
- **D-03:** Push-to-talk mode — user holds a key/button to talk. Saves bandwidth, avoids background noise. The channel is always transmitting (silence or voice) so the probe measurement is always available.
- **D-04:** Other NINJAM users will see the "Instatalk" channel in their mixer. It plays near-realtime audio (voice or silence). This is expected and acceptable — the channel provides real functionality (voice chat).

### Measurement Protocol
- **D-05:** Timing-only measurement — no data encoded in the audio stream. The receiver simply compares wall-clock timestamps: `t_insta` = when instamode data arrives from a remote user, `t_interval` = when that user's interval audio starts playing. `measured_delay = t_interval - t_insta`. This requires zero protocol changes and works with any audio content on the instamode channel.
- **D-06:** Single measurement — use the first clean measurement immediately. Fastest calibration (~1 interval). No warmup discard, no averaging.
- **D-07:** Trust the measurement unconditionally — if the measured delay differs wildly from the BPM/BPI calculation, the measurement is MORE correct because it reflects actual network conditions. No sanity-check fallback.
- **D-08:** Measurement happens automatically when video is active and instamode data is received from any remote user. No explicit "calibrate" step needed.

### Companion Buffering UX
- **D-09:** Global syncing overlay. A single black overlay covers the video area with a "Syncing video..." indicator, fading out after the buffer fills. Per-tile fade-in is not feasible because VDO.Ninja renders inside a cross-origin iframe (cannot position overlays over individual tiles). Global overlay is functionally equivalent since all tiles share the same measured delay. *(Updated 2026-04-16: relaxed from per-tile to global overlay based on cross-origin constraint.)*
- **D-10:** Ring buffer per remote user in the companion page. Buffer size = `measured_delay_ms / frame_interval_ms` frames.
- **D-11:** Three-tier delay priority: manual slider (highest) > auto-measured (instamode probe) > auto-calc (BPM/BPI formula, lowest). Default to auto-measured when probe data is available. Manual slider always wins if user touches it.
- **D-12:** Companion footer shows active mode in the format "X.Xs (mode)", e.g. "4.2s (measured)", "4.0s (calculated)", "5.0s (manual)". Follows the existing Phase 12.1 footer pattern. *(Updated 2026-04-16: aligned format with established "Xs (mode)" pattern from Phase 12.1.)*

### Fallback Behavior
- **D-13:** If no instamode data is received (no remote user has a JamWide client with Instatalk), fall back to BPM/BPI calculation (current Phase 12 behavior). The companion page doesn't need to know which method produced the delay — it receives the same `{"type":"bufferDelay","delayMs":N}` message either way.

### Claude's Discretion
- PTT key binding (could be a configurable hotkey or a UI button)
- Exact fade-in CSS transition duration for video tiles
- Ring buffer implementation details (MediaStream vs canvas capture vs frame queue)
- Whether to show a small latency measurement indicator in the plugin UI

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `VideoCompanion::broadcastBufferDelay(bpm, bpi)` — can be extended to accept a measured delay override
- `NJClient::on_new_interval()` — fires at each interval boundary, ideal for recording `t_interval`
- Existing `bufferDelay` WebSocket message — companion already handles this, no new message type needed
- Phase 12.1 manual delay slider — already has auto/manual toggle infrastructure

### Established Patterns
- Channel flags set via `CLIENT_SET_CHANNEL_INFO` message (`sci.build_add_rec(name, vol, pan, flags)`)
- Instamode uses 64-byte encoding blocks (`LIVE_ENC_BLOCKSIZE2`) and 128-byte prebuffer
- Audio encoding uses Vorbis codec for instamode (same as regular channels, just smaller blocks)
- `config_play_prebuffer` controls regular channel buffering (default 8192 bytes)

### Integration Points
- `NinjamRunThread` — where local channels are configured and sent to server
- `JamWideJuceProcessor::processBlock()` — where instamode data from remote users would be timestamped
- `VideoCompanion` — where measured delay replaces/supplements BPM/BPI calculation
- Companion page `setBufferDelay` handler — where ring buffer logic would be added

</code_context>

<specifics>
## Specific Ideas

- Channel named "Instatalk" — communicates purpose to other users
- Voice talkback is the primary feature; latency probe is a "free" side effect
- User suggested using this to eliminate the Phase 12.1 delay slider's reliance on theoretical calculations
- Measurement captures all real-world factors: network latency, server relay, codec processing, jitter buffers

</specifics>

<deferred>
## Deferred Ideas

- **Voice activity detection (VAD):** Could replace PTT with automatic voice detection. More complex, belongs in a future polish phase.
- **Per-user delay calibration:** Currently using single room-level measurement. Per-user calibration would handle asymmetric network paths but adds significant complexity.
- **Instatalk as voice chat UI:** A dedicated chat panel for voice (like Discord) rather than just a mixer channel. Would need its own phase.

</deferred>
