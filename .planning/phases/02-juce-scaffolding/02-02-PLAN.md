  - Class `NinjamRunThread : public juce::Thread`
  - Constructor takes `JamWideJuceProcessor& processor` reference
  - Destructor calls `stopThread(5000)` (CRITICAL: must stop before destruction per JUCE requirements)
  - `void run() override` -- the main thread loop
  - Private: reference to processor

  **NinjamRunThread.cpp:**
  - Constructor: Initialize juce::Thread with name "NinjamRun", store processor reference
  - Destructor: Call `stopThread(5000)` with 5-second timeout
  - `run()` implementation:
    ```
    while (!threadShouldExit())
    {
        // Phase 3 will add: command queue processing and NJClient::Run() call

    ```
    Add clear comments indicating where Phase 3 will add NJClient::Run() and command queue processing. The thread currently does nothing but proves the lifecycle works.

  **JamWideJuceProcessor.h modifications:**
  - Add `#include "NinjamRunThread.h"` (or forward declare and include in .cpp)
  - Add private member: `std::unique_ptr<NinjamRunThread> runThread;`

  **JamWideJuceProcessor.cpp modifications:**
  - In `prepareToPlay()`: Create and start the run thread:
    ```cpp
    runThread = std::make_unique<NinjamRunThread>(*this);
    runThread->startThread(juce::Thread::Priority::normal);
    ```
  - In `releaseResources()`: Stop and destroy the run thread:
    ```cpp
    if (runThread)
    {
        runThread->signalThreadShouldExit();
        runThread->stopThread(5000);
        runThread.reset();
    }
    ```
  - In destructor: Ensure runThread is stopped (safety net in case releaseResources wasn't called):
    ```cpp
    runThread.reset(); // unique_ptr destruction calls NinjamRunThread destructor which calls stopThread
    ```

  **CMakeLists.txt:**
  - Add `juce/NinjamRunThread.cpp` to the JamWideJuce target_sources list.

  IMPORTANT: Do NOT include NJClient headers or call any NJClient functions. The thread is a skeleton that proves lifecycle management works. Phase 3 adds the actual NJClient::Run() integration.

  IMPORTANT: Use `wait(50)` in the run loop, NOT `Thread::sleep(50)`. wait() is interruptible by signalThreadShouldExit(), ensuring the thread shuts down promptly rather than blocking for up to 50ms.

<task type="auto">
  <name>Task 2: Create GitHub Actions CI workflow with pluginval validation</name>
  <files>.github/workflows/juce-build.yml</files>
  <action>
  Create `.github/workflows/juce-build.yml` -- a NEW workflow file separate from the existing `build.yml` (which handles CLAP releases).

  **Trigger:** Push to any branch, pull requests to main. NOT tag-based (that's the release workflow).

  ```yaml
  name: JUCE Build + Validate

  on:
    push:
      branches: ['**']
    pull_request:
      branches: [main]

  env:
    BUILD_TYPE: Release
  ```

  **macOS job (macos-14 Apple Silicon runner):**
  1. Checkout with `submodules: recursive`
  2. Setup CMake via `lukka/get-cmake@latest`
  3. Configure: `cmake -B build -DCMAKE_BUILD_TYPE=Release -DCMAKE_OSX_ARCHITECTURES="arm64;x86_64" -DJAMWIDE_BUILD_JUCE=ON -DJAMWIDE_BUILD_CLAP=OFF -DJAMWIDE_DEV_BUILD=OFF`
  4. Build: `cmake --build build --config Release -j $(sysctl -n hw.ncpu)`
  5. Verify artifacts exist (ls the artefacts directory)
  6. Download pluginval: `curl -L https://github.com/Tracktion/pluginval/releases/latest/download/pluginval_macOS.zip -o pluginval.zip && unzip pluginval.zip`
  7. Force AU re-scan: `killall -9 AudioComponentRegistrar 2>/dev/null || true`
  8. Validate VST3: `./pluginval.app/Contents/MacOS/pluginval --validate-in-process --strictness-level 5 --timeout-ms 120000 --validate "build/JamWideJuce_artefacts/Release/VST3/JamWide JUCE.vst3"`
  9. Validate AU: `./pluginval.app/Contents/MacOS/pluginval --validate-in-process --strictness-level 5 --timeout-ms 120000 --validate "build/JamWideJuce_artefacts/Release/AU/JamWide JUCE.component"` (continue-on-error: true -- AU validation in CI can be flaky per research)
  10. Validate Standalone: `./pluginval.app/Contents/MacOS/pluginval --validate-in-process --strictness-level 5 --timeout-ms 120000 --validate "build/JamWideJuce_artefacts/Release/Standalone/JamWide JUCE.app"` (continue-on-error: true -- standalone may need audio hardware)

  **Windows job (windows-latest):**
  1. Checkout with `submodules: recursive`
  2. Setup CMake via `lukka/get-cmake@latest`
  3. Configure: `cmake -B build -G "Visual Studio 17 2022" -A x64 -DJAMWIDE_BUILD_JUCE=ON -DJAMWIDE_BUILD_CLAP=OFF -DJAMWIDE_DEV_BUILD=OFF`
  4. Build: `cmake --build build --config Release`
  5. Verify artifacts exist
  6. Download pluginval: Download from `https://github.com/Tracktion/pluginval/releases/latest/download/pluginval_Windows.zip`
  7. Validate VST3: `pluginval.exe --validate-in-process --strictness-level 5 --timeout-ms 120000 --validate "build\JamWideJuce_artefacts\Release\VST3\JamWide JUCE.vst3"`

  NOTE: Do NOT add `-DCLAP_WRAPPER_DOWNLOAD_DEPENDENCIES=TRUE` -- that flag is for the old CLAP build. The JUCE build does not use clap-wrapper. The clap-juce-extensions submodule has its own CLAP SDK included.

  IMPORTANT: If clap-juce-extensions needs the CLAP SDK headers, they come from the clap-juce-extensions submodule's own deps, not from the project's libs/clap. Check if clap-juce-extensions auto-fetches its CLAP SDK -- if it has a git submodule for it, the `submodules: recursive` checkout handles it.
  </action>
