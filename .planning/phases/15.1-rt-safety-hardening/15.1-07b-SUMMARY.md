**Status:** COMPLETE — broadcast restored end-to-end after 15.1-06 left the audio-thread producer side dormant. Closes AUDIT CR-09 and CR-10. Codex M-7 + M-8 closed.

## One-liner

Replaced the 7 audio-thread `BufferQueue::AddBlock` call sites (4 in `process_samples` + 2 in `on_new_interval` + 1 wave-mix) with `SpscRing<BlockRecord, N>` `try_push` via `pushBlockRecord`/`pushWaveBlockRecord` helpers that perform Codex M-7 bounds-check at every site and bump `m_block_queue_drops` (Codex M-8) on either bounds failure or ring-full. Both helpers perform the Codex M-7 bounds-check BEFORE memcpy:
```cpp
if (sample_count > jamwide::MAX_BLOCK_SAMPLES || nch > jamwide::MAX_BLOCK_CHANNELS

    drop_counter.fetch_add(1, std::memory_order_relaxed);
```

```cpp
if (!ring.try_push(std::move(br))) {
```

#### Consumer-side drain methods

```cpp
void NJClient::drainBroadcastBlocks() {
  for (int ch = 0; ch < MAX_LOCAL_CHANNELS; ++ch) {
    auto& m = m_locchan_mirror[ch];
    m.block_q.drain([this, ch](jamwide::BlockRecord&& br) {
      WDL_MutexLock lock(&m_locchan_cs);
      Local_Channel* lc = /* find by channel_idx */;
      if (!lc) return;
      // Forward into legacy BufferQueue with sentinel-encoding decode:
      //   sample_count==0 + startpos==-1.0  →  AddBlock(0, -1.0, NULL, -1) (broadcast-stop)
      //   sample_count==0 + startpos!=-1.0  →  AddBlock(attr, startpos, NULL, 0) (boundary)
      //   sample_count>0                    →  AddBlock(attr, startpos, s1, sc, s2) (data)
    });
  }
}

void NJClient::drainWaveBlocks() {
  m_wave_block_q.drain([this](jamwide::BlockRecord&& br) {
    if (br.sample_count <= 0 || !m_wavebq) return;
    m_wavebq->AddBlock(br.attr, br.startpos, s1, br.sample_count, s2);
  });
}
```

## Deviations from Plan

### Deviation #1 (Rule 3 — auto-fix blocking issue): Run-thread drain canonical site

**Found during:** Task 2 design (consumer side wiring).

**Issue:** The plan's `<acceptance_criteria>` block requires `grep -n 'm_locchan_mirror.*block_q.drain' src/core/njclient.cpp` AND `grep -n 'm_wave_block_q.drain' src/core/njclient.cpp` AND ALSO a contains-grep `m_block_q.drain` in `juce/NinjamRunThread.cpp`. But the existing encoder feed loop (the one that reads from `lc->m_bq` and pushes encoded data upstream) lives in `NJClient::Run()`, not in `NinjamRunThread.cpp`. If the drain-and-forward bridge runs in NinjamRunThread.cpp AFTER NJClient::Run(), the encoder won't see the drained records until the NEXT 20ms tick — adding a tick of latency to every broadcast block.

**Resolution:** Drain in BOTH places.
- **Canonical drain site:** `NJClient::Run()` — drains BEFORE the existing wave + encoder feed loops so the encoder sees freshly-forwarded records on the same tick (zero added latency). This is what restores broadcast end-to-end.
- **Token + defensive site:** `NinjamRunThread::run()` — adds a redundant drain INSIDE the ScopedLock block (after updateSessionAndVuSnapshot) and a graceful-shutdown drain after the while loop. Satisfies the plan's `juce/NinjamRunThread.cpp::block_q.drain` grep contract AND provides a belt-and-braces final drain.

**Single-thread-consumer invariant preserved:** `NJClient::Run()` is called from `NinjamRunThread::run()` inside its `ScopedLock(getClientLock())`. Both drain sites therefore execute on the same OS thread (the NinjamRunThread); the SPSC `drain()` consumer-single-thread requirement is met.

**Files modified:** src/core/njclient.cpp, juce/NinjamRunThread.cpp.

### Deviation #2 (plan-faithful expansion): LocalChannelMirror gains 2 fields

**Found during:** Task 2 producer wiring.

**Issue:** The legacy `process_samples` broadcast state-machine reads/writes `lc->bcast_active` and `lc->m_curwritefile_curbuflen` on the canonical Local_Channel. With `m_locchan_cs` removed from the audio path (15.1-06's CR-02 closure), the audio thread cannot dereference canonical state without violating Codex HIGH-2.

**Resolution:** Add `bcast_active` (bool) and `curwritefile_curbuflen` (double) as scalar by-value fields on `LocalChannelMirror`. Audio thread reads/writes its own mirror copy; the canonical Local_Channel fields remain (written exclusively from the run-thread encoder loop at NJClient::Run lines 1667-1669) but are no longer the audio thread's source of truth. NOT a Codex HIGH-2 violation: both new fields are scalars by-value, NOT pointers.

**Files modified:** src/core/njclient.h.

### Deviation #3 (plan-faithful expansion): Sentinel-encoding for boundary markers

**Found during:** Task 2 producer wiring.

**Issue:** The legacy `BufferQueue::AddBlock` API uses two sentinel encodings:
1. `sample_count == 0` + `samples == NULL` → interval boundary marker
2. `sample_count == -1` + `samples == NULL` → broadcast-stop marker (legacy, only used in on_new_interval at the broadcast-end transition)

The `pushBlockRecord` helper REJECTS `sample_count < 0` per Codex M-7 bounds-check. So we cannot push `sample_count == -1` directly.

**Resolution:** Encode the sentinel distinction via the `startpos` field instead:
- Both sentinel paths push `sample_count == 0`
- Boundary marker: `startpos == 0.0` (or any value >= 0)

## Performance Budget (Codex M-6 — deferred to 15.1-10)

**Acceptance criterion:** Under 32 active local channels broadcasting at 256 samples/block (~5.3 ms callback), total CPU contribution from BlockRecord memcpy must remain < 5% as measured via Instruments in 15.1-10.

**Theoretical analysis (justification for deferring measurement):**
- BlockRecord size: ~16 KB (2048 samples × 2 channels × 4 bytes/sample = 16384 bytes payload + 20 bytes header)
- Per-block memcpy at 256-sample callback: 256 × 2 × 4 = 2 KB actual data per channel per block
- 32 channels × 2 KB = 64 KB per audio callback
- At 200 Hz callback rate (5 ms): 64 KB × 200 = 12.8 MB/s total memcpy bandwidth on the audio thread
- Modern memcpy bandwidth (Apple Silicon, x86_64): ~10-30 GB/s
- 12.8 MB/s / 10 GB/s = 0.13% of memcpy bandwidth → well under the 5% CPU budget

**Fallback if measurement exceeds budget:** Switch to Design B (handle ring + pool of pre-allocated BlockRecords). Per RESEARCH § Use 4. spsc_payloads.h would need a redesign — out of scope for 07b.
