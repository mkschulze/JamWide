## SPSC Primitive Inventory

### What `src/threading/spsc_ring.h` provides today

`SpscRing<T, N>` (header read end-to-end, lines 1-152) is a templated single-producer/single-consumer ring buffer with these properties:

| Property | Value | Implication for 15.1 |
|---|---|---|
| Capacity `N` | Compile-time, must be power of two (line 34 `static_assert`) | Per-queue size locked at construction; choose conservatively |
| Element type `T` | Must be trivially copyable or movable (line 29 doc) | POD records OK, raw pointers OK, `std::variant<...>` OK iff its alternatives are trivially copy/movable |
| Construction | Default ctor sets head/tail to 0 (line 38); buffer is `std::array<T, N>` (line 143) — **stack-or-class-member, not heap** | Embedding inside `RemoteUser` / `Local_Channel` / `NJClient` is fine; the ring is its own storage |
| Cache-line padding | `alignas(64)` on head/tail (lines 146-147) | False-sharing-free; suitable for high-frequency producer/consumer |
| Producer API | `try_push(const T&)`, `try_push(T&&)` — returns `false` if full (lines 51-81) | Producer must handle full-queue case: drop, retry, or grow record (run thread can retry next 20ms tick; audio thread must never block) |
| Consumer API | `try_pop()` returning `std::optional<T>`, plus `drain(Func)` helper (lines 87-116) | `drain()` calls a callback for each element until empty — exactly what we want at the top of `AudioProc` and at the run-thread tick |
| Memory ordering | Producer: relaxed load + acquire load + release store. Consumer: relaxed load + acquire load + release store. | Standard SPSC release/acquire pairing — no extra ordering work needed for record visibility |
| Non-copyable/non-movable | Lines 41-44 explicit `delete` | Don't try to put one inside a `std::vector<SpscRing>` — must be heap-allocated or a fixed `std::array` of rings if multiple are needed |

### What 15.1 specifically needs from the primitive

The audit groups violations into five SPSC use cases. For each, here's what the planner needs to land:

#### Use 1 — Deferred-delete queue (CR-05, CR-06, CR-07)

Holds `DecodeState*` pointers. Audio thread pushes; run thread pops and `delete`s.

- **Type:** `SpscRing<DecodeState*, N>` works directly. `T = void*` is not needed — typed pointer is safer.
- **Capacity:** worst case is per-block deletes in `mixInChannel` (CR-07 sites) plus per-interval deletes in `on_new_interval` (CR-05/06). At llmode rates (Instatalk-style chunks fire several times per second per active speaker) and the run thread draining at 20ms (50 Hz), one drain interval can accumulate roughly: `peers × channels × llmode_advance_rate × 20ms`. With 8 peers × 4 channels × 5 advances/sec × 0.02s = ~3.2 deletes per drain. Plus interval-boundary adds worst-case 8 × 32 = 256 deletes once every interval (~3-4s at 100 BPM/16 BPI). **Recommended capacity: 256** (covers a full interval boundary worst-case in a single drain miss).
- **Granularity decision (Claude's discretion per CONTEXT):** **One shared `SpscRing<DecodeState*, 256>` per `NJClient`** is the right choice. Per-RemoteUser queues are wasted memory (8KB × 8 peers); per-callsite is ridiculous. The destructor is the same (`delete` calls `~DecodeState()`) regardless of which site enqueued the pointer.
- **No helper wrapper needed.** Push raw pointer, drain calls `delete p` lambda.

#### Use 2 — RemoteUserSnapshot state-update queue (CR-01)

The run thread mutates `m_remoteusers` (add peer, remove peer, change channel mask, mute/solo, codec swap, `next_ds[0]/[1]` advance). Audio thread needs a local mirror.

- **Naive approach (REJECTED):** `SpscRing<RemoteUserSnapshot, N>` where each push is a full snapshot of all peers. Snapshot can be ~kilobytes (32 channels × per-channel struct × MAX_USER_CHANNELS); copying that on every state change is wasteful and the queue capacity issue dominates.
- **Recommended approach:** **typed-record / variant** mutation queue. The 15.1-04 sub-plan must define a record type:

  ```cpp
  // src/threading/njclient_state_updates.h (NEW in 15.1-04)
  namespace jamwide {

  struct PeerAddedUpdate    { int slot; /* user identity payload */ };
  struct PeerRemovedUpdate  { int slot; };
  struct PeerChannelMaskUpdate { int slot; int submask; int chanpresentmask; int mutedmask; int solomask; };
  struct PeerVolPanUpdate   { int slot; bool muted; float volume; float pan; };
  struct PeerNextDsUpdate   { int slot; int channel; int slot_idx; DecodeState* ds; };  // 0 or 1
  struct PeerCodecSwapUpdate{ int slot; int channel; unsigned int new_fourcc; };

  using RemoteUserUpdate = std::variant<
      PeerAddedUpdate,
      PeerRemovedUpdate,
      PeerChannelMaskUpdate,
      PeerVolPanUpdate,
      PeerNextDsUpdate,
      PeerCodecSwapUpdate
  >;

  } // namespace jamwide
  ```

  The audio thread's local mirror — a `RemoteUserMirror[MAX_PEERS]` or similar — is mutated by `drain([&](RemoteUserUpdate&& u) { std::visit(applyVisitor{mirror}, u); })`. This is the standard SPSC mutation-stream pattern.

- **Capacity:** state changes are run-thread-paced (peer joins are rare; mute/solo from UI is rare; mask changes are at most every block). Conservative: `N = 64` per drain interval. **Recommended: `SpscRing<RemoteUserUpdate, 64>` per NJClient.**
- **Helper wrapper needed in 15.1-04:** the `RemoteUserUpdate` variant + mirror struct + visitor. This is the bulk of 15.1-04's responsibility.
- **Variant trivial-copy concern:** `std::variant<plain PODs>` is trivially copyable iff all alternatives are. All payloads above are pure-POD — trivially copyable. Verified static_assert candidates: `static_assert(std::is_trivially_copyable_v<RemoteUserUpdate>);`. Add this assert to the header to lock the contract.

#### Use 3 — LocalChannelSnapshot state-update queue (CR-02)

Same pattern as Use 2 but smaller surface. Local channels are bounded (`MAX_LOCAL_CHANNELS = 32`) and only this user mutates them.

- **Recommended:** `SpscRing<LocalChannelUpdate, 32>` per NJClient. Variant payloads: `LocalChannelAdded`, `LocalChannelRemoved`, `LocalChannelInfoUpdate` (name/srcch/bitrate/bcast/outch/flags), `LocalChannelMonitoringUpdate` (vol/pan/mute/solo).
- 15.1-04 defines this alongside the RemoteUser variant.

#### Use 4 — BufferQueue replacement (CR-09, CR-10)

The audio thread currently calls `lc->m_bq.AddBlock(...)` and `m_wavebq->AddBlock(...)` to hand sample blocks to the encoder thread / wavewriter. Both go through `BufferQueue` which is mutex+heap-grow.

- The audit recommends two options (lines 224-227 of AUDIT.md). **Option 1 (cleaner) is the recommended path:** replace `BufferQueue` entirely with a typed SPSC-of-fixed-size-blocks.
- **Type:** the producer needs to hand off (a) a block of float samples (variable size, but bounded by max block × channels × 4 bytes) and (b) attribute metadata (`int attr`, `double startpos`, `int sample_count`, `int nch`). Two designs:

  | Design | Pros | Cons |
  |---|---|---|
  | **A: SPSC of fixed-size records with inline sample buffer** — `struct BlockRecord { int attr; double startpos; int sample_count; int nch; float samples[MAX_BLOCK_SAMPLES * 2]; };` and `SpscRing<BlockRecord, N>` | Self-contained; no aux pool; record is trivially copyable | `MAX_BLOCK_SAMPLES * 2 * 4 = ~16KB` per record × N — significant memory; record copy on push/pop is a memcpy of the buffer |
  | **B: SPSC of pointers into a preallocated pool** — `struct BlockHandle { int pool_idx; int attr; double startpos; int sample_count; int nch; }; SpscRing<BlockHandle, N>` plus `std::array<std::array<float, MAX_BLOCK_SAMPLES * 2>, N>` pool | Tiny ring records; zero-copy of sample data into the pool slot | Two-step: producer claims a pool slot index, fills it, then pushes the handle. Pool slot lifetime = pushed but not yet consumed; needs a free-list (which is itself an SPSC, but bounded by N so trivial) |

  **Recommended: Design A** for 15.1-07. The simpler path. Memory cost is acceptable: with `MAX_BLOCK_SAMPLES = 2048` and `N = 16`, that's `16 × 2048 × 2 × 4 = 256KB` per local channel. With `MAX_LOCAL_CHANNELS = 32`, total is 8MB for local channels — large but not catastrophic. If this proves too much, the planner may switch to Design B in 15.1-07b.

- **Helper wrapper needed in 15.1-04:** the `BlockRecord` struct and its constants (`MAX_BLOCK_SAMPLES`, etc.). Plus a tiny shim that translates from the old `BufferQueue::AddBlock(...)` signature into a `try_push` of the new record (so `mixInChannel`, `process_samples`, and `on_new_interval` callsites change minimally).
- **Capacity:** `N = 16` blocks of buffer headroom should let the encoder thread fall ~320ms behind without underrun (at 50ms blocks). Conservative.
- **Trivially copyable:** `BlockRecord` with float arrays is trivially copyable. Lock the contract with `static_assert`.

#### Use 5 — DecodeMediaBuffer byte-queue replacement (CR-12)

`DecodeMediaBuffer::Read()` currently calls `mutex.Enter()` (njclient.cpp:131). Audit recommends replacing with byte SPSC.

- **Type:** `SpscRing<uint8_t, N>` does NOT work efficiently — single-byte records are wasteful in a ring with cache-line atomics.
- **Recommended:** **chunked byte SPSC** with fixed-size records: `struct DecodeChunk { int len; uint8_t data[CHUNK_BYTES]; };` plus `SpscRing<DecodeChunk, N>`. Producer (network/run thread) writes complete chunks; consumer (audio thread) drains chunks into a small consumer-side `std::array<uint8_t, ...>` linear buffer that `runDecode` reads from with normal pointer math.
- `CHUNK_BYTES`: NINJAM packets are typically ≤ 8KB. Use `CHUNK_BYTES = 4096`, `N = 32`. Total memory per peer: `32 × (4 + 4096) ≈ 128KB`. Across 8 peers × 32 channels = 32MB — too much.
- **Alternative:** one shared `DecodeMediaBuffer` SPSC primitive per active stream (channels with subscribed=true and codec_fourcc set), not per channel slot. Audit notes `DecodeMediaBuffer` is refcounted (line 174 `decode_buf->Release()`), so this is structurally a per-stream object already. Keep it that way — replace the **internal** mutex+queue with the chunked SPSC; don't multiply the count.
- **Helper wrapper needed in 15.1-04:** the `DecodeChunk` struct and the consumer-side adapter. The class signature `DecodeMediaBuffer::Read(void* buf, int len)` should remain unchanged — only the implementation changes.

### Existing pattern to follow

The codebase already uses `std::atomic<T>` for primitive config fields (lines 124-141):
```cpp
std::atomic<float> config_metronome{0.5f};
std::atomic<float> config_metronome_pan{0.0f};
std::atomic<bool>  config_metronome_mute{false};
std::atomic<int>   config_metronome_channel{-1};

// ... etc
std::atomic<unsigned int> m_encoder_fmt_requested{0};
std::atomic<int> cached_status{NJC_STATUS_DISCONNECTED};  // line 154
```

These use `std::memory_order_relaxed` for both reads (audio thread) and writes (run/UI thread). That's the correct ordering for these scalar values — no other state's visibility depends on them, so seq_cst is unnecessary cost.

### Fields to migrate for CR-03

The audit identifies these fields read inside `m_misc_cs` at `njclient.cpp:825-846`:

| Field | Current type | Current location | Read site | Write site | Migration |
|---|---|---|---|---|---|
| `m_beatinfo_updated` | `int` | njclient.h:325 | AudioProc:826 (under m_misc_cs) | `updateBPMinfo`:757 (under m_misc_cs) | `std::atomic<int>` with relaxed reads/writes; the "did the value change since I last consumed it?" semantic is preserved by audio thread storing 0 after consuming |
| `m_bpm` | `int` | njclient.h:324 | AudioProc:828, 842 | updateBPMinfo:755 | `std::atomic<int>` with relaxed reads/writes |
| `m_bpi` | `int` | njclient.h:324 | AudioProc:832, 843 | updateBPMinfo:756 | `std::atomic<int>` with relaxed reads/writes |

### Memory ordering analysis

`m_beatinfo_updated` is the synchronization flag; `m_bpm` and `m_bpi` are the payload. Standard publish/subscribe pattern:

- **Run thread (writer):** write `m_bpm` (relaxed), write `m_bpi` (relaxed), then write `m_beatinfo_updated = 1` with **release**.
- **Audio thread (reader):** read `m_beatinfo_updated` with **acquire**; if `1`, then read `m_bpm`, `m_bpi` (relaxed); then store `m_beatinfo_updated = 0` (relaxed — we're the only reader/clearer).

This is stronger than the current mutex-protected code in two ways: (a) the audio thread never blocks, (b) the read-then-clear of `m_beatinfo_updated` is now a race-free pattern under release/acquire ordering.

```cpp
// src/core/njclient.h — PROPOSED for 15.1-02
std::atomic<int> m_beatinfo_updated{0};
std::atomic<int> m_bpm{0};
std::atomic<int> m_bpi{0};

// src/core/njclient.cpp — PROPOSED updateBPMinfo (no longer needs m_misc_cs)
void NJClient::updateBPMinfo(int bpm, int bpi)
{
    m_bpm.store(bpm, std::memory_order_relaxed);
    m_bpi.store(bpi, std::memory_order_relaxed);
    m_beatinfo_updated.store(1, std::memory_order_release);  // publishes both above
}

// src/core/njclient.cpp — PROPOSED AudioProc (no longer needs m_misc_cs)
if (m_beatinfo_updated.load(std::memory_order_acquire))   // sees both writes above
{
    int bpm = m_bpm.load(std::memory_order_relaxed);
    int bpi = m_bpi.load(std::memory_order_relaxed);
    // ... compute interval_length etc using bpm, bpi (locals)
    m_beatinfo_updated.store(0, std::memory_order_relaxed);
    m_active_bpm = bpm;
    m_active_bpi = bpi;
    // ...
}
```

### `m_misc_cs` after migration

After `m_beatinfo_updated`/`m_bpm`/`m_bpi` migrate, **the only remaining audio-thread acquisition of `m_misc_cs` should be gone** — verify with grep before closing 15.1-02. The mutex itself stays in the class because the run-thread `updateBPMinfo` no longer takes it (the writes are atomic), but **the audit shows no other audio-thread acquisition of `m_misc_cs`**. If verification finds one, surface it; otherwise remove the audio-thread-side acquisition and keep `m_misc_cs` for any non-audio-thread callers. (Bonus cleanup: if no callers remain, delete the mutex entirely.)

### Bonus from audit (line 421 of AUDIT.md): `m_interval_pos` race

The audit noted `m_interval_pos` is written from `processBlock` (audio thread) and read from `GetPosition` (UI thread) — that's a real race, though not RT-safety. The auditor recommends lifting to `std::atomic<int>` while doing the CR-03 work. **Recommendation: include this in 15.1-02** — it's the same mechanical change, same blast radius, and closes a documented data race for free.

## Pre-Allocation Strategy (15.1-08)

[VERIFIED: read of `wdl/heapbuf.h` lines 1-269, `wdl/queue.h` lines 1-271, `wdl/vorbisencdec.h` lines 124-230, `wdl/flacencdec.h` lines 160-326]

### `WDL_HeapBuf::Prealloc` (heapbuf.h:56)

```cpp
void Prealloc(int sz) {
    if (m_alloc < sz && WDL_NORMALLY(m_size < sz)) {
        const int oldsz = m_size;
        Resize(sz, false);
        m_size = oldsz;
    }
}
```

**What it does:** Calls `Resize(sz, false)` (which `realloc()`s if `newsize > m_alloc`), then restores `m_size` to its prior value. So the *allocation* grows to `sz` but the *logical size* is preserved. Subsequent `Resize(...)` calls up to `sz` no longer trigger `realloc`.

**Thread safety:** `Prealloc` itself is NOT thread-safe — it mutates `m_alloc`, `m_size`, `m_buf`. **Must be called from the owning thread, off the audio thread, before the audio thread can race.** Safe call sites: constructor, `_reinit()`, `prepareToPlay`-equivalent hooks.

[VERIFIED: lines 56, 132-205 of heapbuf.h]

[VERIFIED: line 132 of queue.h]

### Decoder internal queues

#### VorbisDecoder (vorbisencdec.h:230)
Single internal queue: `WDL_TypedQueue<float> m_buf;` — accumulates decoded float samples. Currently no `Prealloc` call anywhere on this queue.

To pre-allocate: at decoder construction (off the audio thread), call:
```cpp
m_buf.Prealloc(max_samples_per_block * MAX_CHANNELS);
```
where `max_samples_per_block` is the max decoded frame size (Vorbis blocksize is up to 8192 samples per frame; typical NINJAM ~2048). With `MAX_CHANNELS = 2` for stereo, `Prealloc(16384)` is conservative.

[VERIFIED: line 230 of vorbisencdec.h; the `m_buf.Add(...)` call on line 170 and `Skip` call on lines 191-194]

#### FlacDecoder (flacencdec.h:322-323)
Two internal queues:
- `WDL_Queue m_inbuf;` — undecoded compressed bytes from network/file.
- `WDL_TypedQueue<float> m_outbuf;` — decoded float samples.

Prealloc both at decoder construction (in `initDecoder()` or constructor):
```cpp
m_inbuf.Prealloc(max_compressed_bytes);   // bytes; ~16KB conservative for NINJAM frames
m_outbuf.Prealloc(max_samples_per_block * MAX_CHANNELS);
```

[VERIFIED: lines 182, 289, 322-323 of flacencdec.h]

### Where these are constructed

`start_decode` (njclient.cpp:1909-1911) creates them on the audio thread today (CR-08). Once 15.1-09 moves `start_decode` to the run thread, the decoder's constructor (which runs on the run thread) is the right hook for the `Prealloc` calls. **15.1-08 and 15.1-09 are coupled** — Prealloc must happen, but it must happen on the run thread, and that requires 15.1-09 first or both at once.

**Recommended sequencing:** land 15.1-09 (move `start_decode` off audio thread) BEFORE or COUPLED WITH 15.1-08 (Prealloc decoder internals). The audit's proposed ordering suggested 15.1-08 last, but the coupling means it should run after 15.1-09 if they're separated, or be merged.

### `tmpblock` Prealloc (M-01)

[VERIFIED: njclient.h:357 declares `WDL_HeapBuf tmpblock;`. njclient.cpp:1982 `if (tmpblock.GetSize() < bytelen) tmpblock.Resize(bytelen);`]

`bytelen = len * sizeof(float)` (line 1981). The maximum `len` is the max audio block size from JUCE — `prepareToPlay`'s `getMaximumExpectedSamplesPerBlock()`.

**Problem:** NJClient has no `prepareToPlay` equivalent. The `_reinit()` method exists (njclient.h:297) but doesn't take a max-block-size parameter, and it's called from `Connect`/`Disconnect` paths (not from JUCE's lifecycle).

**Recommendation:** Add a public method `void NJClient::SetMaxAudioBlockSize(int max_samples_per_block);` that calls `tmpblock.Prealloc(max_samples_per_block * sizeof(float));`. Wire it from `JamWideJuceProcessor::prepareToPlay(double sampleRate, int samplesPerBlock)`. The processor already has visibility to `samplesPerBlock` and to `client`.

This change is mechanical: one new public NJClient method, one call site in `JamWideJuceProcessor::prepareToPlay`. No call-graph changes elsewhere.

### Proposed `--tsan` flag

Add to `scripts/build.sh`:


### macOS-specific TSan caveats

1. **Codesigning vs sanitizer.** CMakeLists.txt:252-258 attaches a `POST_BUILD` `codesign` command to every JUCE target, including Standalone. With ad-hoc signing (`JAMWIDE_CODESIGN_IDENTITY="-"`, line 237), the sign succeeds but the resulting binary may fail to launch under TSan because TSan injects a runtime library that isn't covered by the signature. **Fix:** when `--tsan` is set, the build either (a) skips the codesign step, or (b) re-signs after TSan injection. Simplest path: pass `-DJAMWIDE_TSAN=ON` and gate the codesign block on `if(NOT JAMWIDE_TSAN)`. Add this gate in 15.1-04. Hardened runtime should be OFF for TSan builds (it blocks unsigned dylibs from loading).

2. **JIT/dyld ordering.** TSan is a `DYLD_INSERT_LIBRARIES`-style runtime; some JUCE plugin hosts on macOS sandboxed environments will reject TSan injection. **Standalone (which is its own .app bundle, not loaded as a plugin) does not have this issue.** This is exactly why CONTEXT D-07 mandates the standalone target.

3. **AddressSanitizer interaction.** Don't combine `-fsanitize=thread` with `-fsanitize=address` — they're mutually exclusive. The current build does not use ASan, so no conflict.

4. **OpenSSL & libvorbis under TSan.** Both have known *false positives* in TSan. Suppressions can be added via a `tsan.supp` file at the repo root and `TSAN_OPTIONS=suppressions=tsan.supp` env var at run time. Not needed for the initial implementation; add if signal becomes noisy.

5. **Apple Silicon vs x86_64.** TSan works on both. `scripts/build.sh` defaults to x86_64-only on macOS (line 73-76). TSan should respect that — no change needed.

## Codec Internal Allocation Residual (CR-11)

### Vorbis decoder steady-state allocation analysis

[VERIFIED: read of `wdl/vorbisencdec.h` lines 124-230]

The hot path through the Vorbis decoder under `mixInChannel → runDecode → DecodeWrote` is:

1. `DecodeGetSrcBuffer(srclen)` (line 124) → calls `ogg_sync_buffer(&oy, srclen)`. This is libogg's API; it grows the libogg internal buffer as needed via `_ogg_realloc`. **Audio-thread libogg realloc**.
2. `DecodeWrote(srclen)` (line 129) → `ogg_sync_wrote`, `ogg_sync_pageout`, `ogg_stream_packetout`, `vorbis_synthesis_blockin`, `vorbis_synthesis_pcmout`, then **line 170**: `m_buf.Add(NULL, samples * vi.channels)` — `WDL_TypedQueue::Add` → `WDL_HeapBuf::ResizeOK` → `realloc`.
3. `Skip(amt)` (line 191) → `m_buf.Advance(amt)` then `m_buf.Compact()` (line 194). `Compact` calls `m_hb.Resize(olen, allocdown=false)` which can `realloc`.

#### Step 2 (`m_buf.Add`) — covered by Prealloc

`WDL_HeapBuf::Resize` (lines 132-205) only re-allocates if `newsize > m_alloc` OR `(resizedown && newsize < m_alloc/2)`. After `Prealloc(max_samples * channels)` lifts `m_alloc` to a known peak, `m_buf.Add` enters the `m_size += new_data` path; since `m_alloc >= peak`, the `newsize > m_alloc` branch never fires. **Steady-state heap-free under Prealloc — VERIFIED by code path.**

#### Step 3 (`m_buf.Compact`) — borderline

`WDL_TypedQueue::Compact(allocdown=false, force=false)` (queue.h:241-259) calls `m_hb.Resize(olen, allocdown)` after a `memmove`. Since `allocdown=false`, the resize-down path inside `WDL_HeapBuf::Resize` (lines 134-141) is skipped. But:

- `Resize(newsize, false)` with `newsize < m_alloc` and `resizedown=false` → enters the outer condition `if (newsize != m_size || ...)` — `newsize != m_size` is true (we just shrunk). It then enters the inner conditional `if (newsize > m_alloc || newsize < resizedown_under)`. With `resizedown=false`, `resizedown_under = 0`, so `newsize < 0` is false. And `newsize > m_alloc` is false (we're shrinking). Therefore **no realloc**, just `m_size = newsize`. **Steady-state heap-free — VERIFIED.**

So M-02's "borderline" worry is resolvable to LOW: `Compact(allocdown=false)` does NOT realloc on the steady-state shrink path.

#### Step 1 (`ogg_sync_buffer`) — out of our reach

libogg's `ogg_sync_buffer` is `_ogg_realloc(oy->data, oy->storage)` when `oy->storage < oy->fill + size`. Once `oy->storage` has reached the peak NINJAM packet size, **subsequent calls are heap-free**. NINJAM packet sizes are bounded (~8KB max per frame); after the first few packets, `oy->storage` is steady. **Algorithmic residual: first ~3-5 frames after a peer joins or codec reset.** Latent, not always-hot. Audit's recommendation (15.1-AUDIT.md:240) to "accept this as out-of-scope risk" is correct.

### FLAC decoder steady-state allocation analysis

[VERIFIED: read of `wdl/flacencdec.h` lines 180-298]

1. `DecodeGetSrcBuffer(srclen)` (line 180) → `m_inbuf.Add(nullptr, srclen)` → realloc until `m_alloc` peaks. **Covered by `m_inbuf.Prealloc(...)` at construction.**
2. `read_cb` (line 254) → `m_inbuf.Advance(toread); m_inbuf.Compact();` (lines 269-270). `Compact` with default `allocdown=false` — same analysis as Vorbis: heap-free.
3. `write_cb` (line 275) → `m_outbuf.Add(nullptr, samples * nch)` (line 289). **Covered by `m_outbuf.Prealloc(...)` at construction.**
4. libFLAC internals: `FLAC__stream_decoder_process_single` may grow internal libFLAC buffers (`subframe_bps_`, `cpu_info_`, etc.) on the first few frames. After warm-up, libFLAC reuses its own heap blocks. **Algorithmic residual: similar to libogg, bounded to first few frames per stream.**

### Verdict on CR-11

| Layer | Audio-thread heap activity at steady state | Mitigation |
|---|---|---|
| WDL queues (`m_buf`, `m_inbuf`, `m_outbuf`) | NONE after Prealloc (verified by Resize logic) | 15.1-08 Prealloc — sufficient |
| `Compact` paths | NONE with `allocdown=false` (the default in vorbisencdec.h:194 and flacencdec.h:228) — verified | None needed; downgrade M-02 to LOW |
| libogg / libFLAC internals | Latent realloc on first 3-5 frames per stream / codec reset; heap-free at steady state thereafter | **Out of scope** — would require codec-algorithm work; surface as deferred follow-up |

**Phase 15.1-08's Prealloc strategy is sufficient for the steady state.** The audit's CR-11 finding correctly identifies this as latent, not always-hot. The "deferred to follow-up phase if 15.1-10 verification finds it material" framing is correct.

**Recommendation:** during 15.1-10 UAT under TSan, take an Instruments allocations snapshot of a 5-minute populated session AFTER the first 60 seconds of warm-up. If heap activity from libogg/libFLAC contexts is < 1 alloc/sec, declare CR-11 closed for v1.2. If higher, file a follow-up phase ticket.

## Run-Thread Drain Integration

[VERIFIED: read of `juce/NinjamRunThread.cpp` lines 600-700]

### Existing run-thread cadence

`NinjamRunThread::run()` (line 626) is the single run-thread driver. The main loop is lines 641-674:

```
while (!threadShouldExit())
{
    client = processor.getClient();
    if (!client) { wait(50); continue; }

    processCommands(client);            // line 650 — UI command dispatch
    pollServerList();                   // line 651 — async HTTP server list

    {
        const juce::ScopedLock sl(processor.getClientLock());
        while (!client->Run())
        {
            if (threadShouldExit()) return;
        }

        int currentStatus = client->GetStatus();
        client->cached_status.store(currentStatus, std::memory_order_release);

        handleStatusChange(client, currentStatus);
        handleUserInfoChange(client);
        updateRemoteVuLevels(client);
        detectBpmBpiChanges(client);
        syncInstatalkBroadcast(client);
        pollInstamodeDelay(client);
        updateSessionAndVuSnapshot(client);
    }

    wait(lastStatus_ == NJClient::NJC_STATUS_OK ? 20 : 50);   // 20ms when connected — line 673
}
```

The 20ms cadence is at line 673 — **adaptive: 20ms when connected, 50ms when disconnected**. This matches CONTEXT.md's documentation of the existing run-thread cadence. The drain calls land here.

### Where to add SPSC drains

The drain calls go INSIDE the `juce::ScopedLock sl(processor.getClientLock())` block, after `client->Run()` and the existing snapshot calls. **NOT before** — the drain operates on NJClient internal state, and putting it before `client->Run()` could miss queued state from the audio thread that ran since the last tick. **Recommended insertion point: between line 669 (`updateSessionAndVuSnapshot`) and the closing `}` at line 670**, or anywhere within the locked block.

### Drain order: does it matter?

For 15.1, three SPSC queues are drained on the run thread:

1. **State-update queues** (RemoteUser mutations, LocalChannel mutations) — but wait: these are PRODUCED by the run thread, CONSUMED by the audio thread. The run thread doesn't drain them; the audio thread does (at the top of `AudioProc`). **Run thread doesn't drain these.**

2. **Deferred-delete queue** (CR-05/06/07): produced by audio thread, consumed by run thread. Drain on run thread; runs `delete p` for each popped pointer.

3. **(Optional) audio-path log queue** (D-03 fallback if any logging is retained): produced by audio thread, consumed by run thread. Drains and writes to the actual log file.

So only (2) and (3) drain on the run thread. Their order:

| Order | Reasoning |
|---|---|
| **Drain log first, deferred-delete second** | Log records may reference DecodeState pointers that the deferred-delete will free. If log drain runs second, the log record's pointer is dangling. |
| **Drain deferred-delete first** | Same risk inverted: deferred-delete drains and frees a pointer; later log drain finds the same pointer in a log record. Still dangling. |

**Resolution:** log records must NOT reference DecodeState pointers (or any heap-owned pointer). They should embed copies of the relevant primitives only. With that contract, **drain order does not matter**, and the planner can choose either order. Recommended: **drain deferred-delete LAST** so any log entries written during the same audio block reach the log file before their referenced state goes away. The risk is purely defensive — the contract above eliminates the actual hazard.

### Run-thread drain code template

```cpp
// juce/NinjamRunThread.cpp — PROPOSED, inside the locked block
// [...existing snapshot calls...]
updateSessionAndVuSnapshot(client);

// 15.1-05+: drain deferred-delete queue
client->drainDeferredDelete();   // calls SpscRing<DecodeState*>::drain([](DecodeState* p){ delete p; })

// 15.1-09: drain sessionmode-arming requests (if implemented as a queue rather than atomic flag)
client->drainArmRequests();      // calls start_decode for each request, swaps result via state-update queue

// 15.1-03 (optional): drain audio-path log queue
client->drainAudioLogQueue();    // pops fixed-size log records, fprintfs them
```

These methods are added to `NJClient` as public methods. Names are illustrative; planner picks final names.

### Why the current `processor.getClientLock()` doesn't conflict

The client lock (line 654) is a JUCE `juce::ScopedLock` over `processor.getClientLock()` — a JUCE `CriticalSection` distinct from the WDL `m_users_cs` / `m_locchan_cs` mutexes. The client lock prevents the JUCE editor (UI thread) from racing with the run thread on the JUCE-side wrapper state — it does NOT touch the audio thread's `AudioProc` path. The drain operations do not interfere with this lock; they only operate on NJClient internals.

After 15.1-07 lands, the run thread will produce RemoteUser/LocalChannel state-update records onto the state-update queues. The audio thread drains those records at the top of `AudioProc`. The run thread STILL holds `m_users_cs` / `m_locchan_cs` for its OWN consistency (NJClient internal consistency vis-à-vis the network thread, the UI thread, etc.) — it just no longer expects the audio thread to take those locks.

## Open Questions for Planner

The audit and this research nail down most decisions. The following remain for the planner:

1. **Sequencing of 15.1-08 vs 15.1-09.**
   - Option A: 15.1-09 (move `start_decode` off audio thread) lands before 15.1-08 (Prealloc decoder internals at construction). Decoder construction is now on the run thread, so Prealloc sites are naturally safe.
   - Option B: 15.1-08 first — Prealloc happens on the audio thread initially. Risky during the gap between 15.1-08 and 15.1-09; the Prealloc itself could realloc on the audio thread.
   - **Research recommendation: Option A** (or merge them). Audit's proposed ordering (15.1-08 last) was written before the Codex realignment; revisit.

2. **15.1-07 split or single sub-plan.**
   - Audit's option to split into 15.1-07a (m_users_cs snapshot) and 15.1-07b (BufferQueue + DecodeMediaBuffer) reduces commit size. The planner should evaluate based on confidence: if the BufferQueue replacement design (Design A vs B above) is uncertain, splitting lets BufferQueue land standalone with isolated UAT.

3. **Audio-path logging — keep any?**
   - CONTEXT D-03 says "eliminate diagnostic logs entirely." This research doesn't override that. But: the auditor's recommendation in 15.1-AUDIT.md:185 says "Keep the `writeLog` API for the destructor/error-path use only (clearly labeled non-RT)." The planner should confirm whether ANY audio-thread error path warrants an SPSC-routed log (e.g., "first decoder error per peer" — once-per-peer-per-session, low-volume). Default: no logging at all from audio thread; planner may exempt one path if it surfaces a real need.

4. **`SetLocalChannelProcessor` plugin-callback contract (H-03).**
   - JamWide doesn't appear to register a callback today. Planner should confirm with grep, then either: (a) document the RT-safety contract on the API, (b) remove the API entirely. Out of scope for the queue-migration sub-plans, but worth a tracker note.

5. **`m_remoteusers` vs `RemoteUserMirror[N]` exact data model.**
   - The audio-thread mirror needs to support iteration over peers with mute/solo/submask logic intact. The current code does `m_remoteusers.GetSize()` and iterates. The mirror could be:
     - `std::array<RemoteUserMirror, MAX_PEERS>` with a `bool active` flag — fixed size, no allocation, but wastes memory for empty slots.
     - `std::vector<RemoteUserMirror>` resized only via the SPSC update — but `vector.push_back` is a heap op; the audio thread can't own a `vector` that grows.
   - **Recommendation: fixed array.** Pick `MAX_PEERS = 64` (NINJAM servers cap at 32-64; 64 is conservative). Memory cost: ~64 × per-peer-mirror struct ≈ a few KB.

6. **`InstaMeasState` atomics under TSan.**
   - The Phase 14.2 instamode state machine (njclient.h:264-277) uses `std::atomic<int>` with mixed `relaxed`/`release`/`acquire` orderings. TSan may flag the `relaxed` reads alongside the `release`/`acquire` ones if it can't infer the synchronizes-with relation. Planner should run the standalone under TSan with a populated session WITH instamode active, before calling 15.1-07 done — this is the highest-risk pre-existing pattern that interacts with the new state-update queues.

7. **Test infrastructure for the JUCE-enabled TSan build.**
   - The existing `tests/` are CLI binaries built without JUCE (`JAMWIDE_BUILD_JUCE=OFF`). They cannot exercise `processBlock`. The TSan standalone is run **manually** under UAT — there's no CI hook for it. Planner should decide whether 15.1-10 should establish a CI step that launches the standalone under TSan with a scripted simulated session, or whether manual UAT is the documented bar (matching CONTEXT D-07 #2). **Default: manual UAT** — CONTEXT explicitly scopes TSan automation to `--tests`.
