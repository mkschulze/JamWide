# Phase 14.1: Audio Prelisten - Research (Gray Area Analysis)

**Researched:** 2026-04-14
**Domain:** NJClient receive-only mode, JUCE ListBox hit-testing, prelisten side-effects
**Confidence:** HIGH -- findings verified against source code

## Summary

This research investigates seven specific gray areas in the prelisten plan. Six of seven contain confirmed risks, three of which are CRITICAL (will cause bugs if not addressed). The plan correctly identifies the overall architecture but misses several NJClient internal behaviors that undermine the "zero local channels" and "justmonitor" assumptions.

**Primary recommendation:** Fix the three CRITICAL issues before implementation begins: (1) pass `prelisten_mode` as `justmonitor` to AudioProc, (2) build username from `processor.lastUsername` instead of `client->GetUser()`, and (3) store the active prelisten host+port to survive list refreshes.

## Gray Area 1: NJClient Zero-Channel Edge Cases

**Risk Level:** CRITICAL

### Finding: `NotifyServerOfChannelChange()` Fires Automatically in AUTH_REPLY

The NJClient core, NOT the NinjamRunThread, calls `NotifyServerOfChannelChange()` inside the `MESSAGE_SERVER_AUTH_REPLY` handler at `njclient.cpp:1232`. This happens unconditionally -- before `handleStatusChange()` in NinjamRunThread ever runs. [VERIFIED: njclient.cpp lines 1189-1248]

The plan's guard in Task 3.2 (skip `SetLocalChannelInfo` when `prelisten_mode`) fires too late. By the time the NinjamRunThread sees `NJC_STATUS_OK`, the NJClient has already sent whatever was in `m_locchannels` to the server.

### Finding: `Disconnect()` Does NOT Clear `m_locchannels`

`Disconnect()` at `njclient.cpp:865-912` clears encoders and file handles for each Local_Channel, but does NOT remove the entries from `m_locchannels`. The `_reinit()` call at line 908 also does not clear them -- it only resets channel indices. [VERIFIED: njclient.cpp lines 888-903, 634-672]

**Impact:** After a normal connection (which sets up 4 local channels via `handleStatusChange`), a subsequent prelisten connection will announce those 4 channels to the server. The prelisten user will NOT appear with zero channels -- they will appear as a full participant with 4 channels (all sending silence because `m_audio_enable` is initially 0, but still visible on the server).

### Finding: `on_new_interval()` Iterates `m_locchannels` Safely

Even with stale `m_locchannels` entries, `on_new_interval()` at line 2679 iterates them and manages broadcast state. When `justmonitor=1`, the encoding path is skipped. When `justmonitor=0` (the plan's current behavior -- see Gray Area 4), encoders run on these stale channels. [VERIFIED: njclient.cpp lines 2670-2739]

### Recommended Mitigation

1. **Option A (Minimal):** Accept that the prelisten user appears on the server with channels. The `[preview]` prefix already signals intent. The plan's guard in `handleStatusChange` prevents adding NEW channels, but old ones persist. This is cosmetically imperfect but functional.

2. **Option B (Cleaner):** Before calling `client->Connect()` in the PrelistenCommand handler, clear `m_locchannels` by calling `client->DeleteLocalChannel(ch)` for each existing channel. This must be done under `clientLock` (which the plan already holds). This ensures `NotifyServerOfChannelChange()` in the AUTH_REPLY sends an empty channel list.

Recommendation: **Use Option B**. The `DeleteLocalChannel` API exists for this purpose. Call it for channels 0-3 before Connect.

## Gray Area 2: Server List Refresh While Prelistening

**Risk Level:** HIGH

### Finding: `updateList()` Replaces `servers` Without Updating `prelistenRow`

`ServerBrowserOverlay::updateList()` at line 67-91 sets `servers = newServers` and calls `listBox.updateContent()`. The `prelistenRow` index is never adjusted. [VERIFIED: ServerBrowserOverlay.cpp lines 67-91]

The plan's `setPrelistenState()` (Task 1.6) looks up the server by hostname in `servers`, but this is only called when a `PrelistenStateEvent` arrives from the backend. There is no re-lookup when the list itself changes.

**Impact:** If the user clicks Refresh while prelistening, or if the server list auto-refreshes:
- `prelistenRow` still points at the old index
- The blue tint, LISTENING badge, and Stop button appear on the wrong row
- Clicking Stop on the wrong row triggers `onStopListenClicked` (correct behavior by accident) but clicking the wrong row's Listen button could cause confusion

### Recommended Mitigation

Store the active prelisten server identity (host + port) as separate fields, not just a row index. In `updateList()`, after replacing `servers`, re-lookup the active prelisten server:

```cpp
void ServerBrowserOverlay::updateList(const std::vector<ServerListEntry>& newServers, const juce::String& error)
{
    servers = newServers;
    // ... existing code ...
    
    // Re-resolve prelistenRow after list change
    if (prelistenRow >= 0 && !prelistenHost.empty())
    {
        prelistenRow = -1;
        for (int i = 0; i < static_cast<int>(servers.size()); ++i)
        {
            if (servers[i].host == prelistenHost && servers[i].port == prelistenPort)
            {
                prelistenRow = i;
                break;
            }
        }
        // If server disappeared from list, prelistenRow stays -1
        // but connection continues (audio still works)
    }
    listBox.updateContent();
    listBox.repaint();
}
```

Add `std::string prelistenHost; int prelistenPort = 0;` to the overlay header. Set them when starting prelisten.

## Gray Area 3: `listBoxItemClicked` Mouse Coordinate System

**Risk Level:** LOW (plan is correct)

### Finding: Coordinates Are Relative to the Row Component

JUCE's `ListBox` uses internal `RowComponent` objects. The `MouseEvent` passed to `listBoxItemClicked` originates from the RowComponent's `mouseDown` handler (via `performSelection`). JUCE's `MouseEvent::getPosition()` returns coordinates relative to the component that received the event -- the RowComponent, not the ListBox. [VERIFIED: juce_ListBox.cpp lines 163-205]

The plan computes button bounds as:
```cpp
int btnX = listBox.getRowPosition(row, true).getWidth() - 70;
int btnY = listBox.getRowPosition(row, true).getHeight() - 22;
```

Since `getRowPosition` returns the row rectangle relative to the ListBox, `.getWidth()` and `.getHeight()` give the row's dimensions (not position), which match the RowComponent's local coordinate space. [VERIFIED: juce_ListBox.h line 460-461]

**Verdict:** The plan's hit-testing logic is correct. The width and height from `getRowPosition` match the row component's local bounds. The `e.getPosition()` coordinates are relative to the row.

**Minor optimization:** Consider using a fixed `kRowHeight = 72` constant instead of the `getRowPosition` call, since the row height is set at line 31 of ServerBrowserOverlay.cpp.

## Gray Area 4: `processBlock` Does Not Pass `justmonitor` to AudioProc

**Risk Level:** CRITICAL

### Finding: Plan Omits `justmonitor=true` for Prelisten Mode

The current `processBlock` at line 486-488 calls:
```cpp
client->AudioProc(inPtrs, numInputChannels, outPtrs, kTotalOutChannels,
                  numSamples, static_cast<int>(storedSampleRate),
                  false, hostPlaying);
```

The 7th argument is `justmonitor`, hardcoded to `false`. [VERIFIED: JamWideJuceProcessor.cpp line 486-488]

Plan 14.1-01 Task 2.2 only adds volume scaling AFTER AudioProc. It does NOT change the AudioProc call to pass `prelisten_mode` as `justmonitor`.

**Impact without fix:** When `prelisten_mode` is true and `m_audio_enable` is 1 (set by SERVER_CONFIG_CHANGE_NOTIFY):
1. `AudioProc` enters the full processing path (line 810-860) because `justmonitor=false` and `m_audio_enable=1` and `m_max_localch > 0` (set by server) [VERIFIED: njclient.cpp lines 782-788]
2. Session position counters increment (line 790-806) -- corrupts session tracking
3. `on_new_interval()` runs full local channel processing including encoder creation
4. `process_samples()` with `justmonitor=0` encodes and queues local audio for transmission (lines 1993-2039)
5. The remote channel mixing at lines 2113-2160 runs (this is what we want)
6. Wave file writing at lines 2165-2173 may trigger

### The `justmonitor` Code Path (What Should Happen)

When `justmonitor=true`, `AudioProc` takes the early-return path at line 786:
```cpp
process_samples(inbuf, innch, outbuf, outnch, len, srate, 0, 1, isPlaying, isSeek, cursessionpos);
return;
```

This calls `process_samples` with `justmonitor=1`, which:
- Still processes local channel monitoring (volume/pan/VU) at lines 1953-2108
- Skips encoding and transmission (guarded by `if (!justmonitor)` at line 1994)
- Skips remote channel mixing (guarded by `if (!justmonitor)` at line 2113) -- **THIS IS A PROBLEM**

Wait -- this means `justmonitor=1` also skips remote channel mixing! Looking at line 2113: `if (!justmonitor)` guards the entire remote user mixing section (lines 2113-2173). So if we pass `justmonitor=true`, we get NO remote audio in the output.

**Deeper analysis:** The `justmonitor` flag is designed for "monitoring local audio without network interaction" -- not for "receive-only mode". Neither `justmonitor=true` nor `justmonitor=false` does exactly what prelisten needs.

What prelisten needs:
- Skip local channel encoding/transmission (justmonitor=1 behavior)
- Include remote channel mixing (justmonitor=0 behavior)
- Skip session position tracking (justmonitor=1 gets this via early return)

### Recommended Mitigation

The cleanest approach: pass `prelisten_mode` as justmonitor to AudioProc, BUT then handle remote mixing separately. Actually, re-reading more carefully:

Looking at `AudioProc` line 782-788:
```cpp
if (!m_audio_enable || justmonitor ||
    (!m_max_localch && remote_user_count == 0))
{
    process_samples(inbuf,innch,outbuf,outnch,len,srate,0,1,isPlaying,isSeek,cursessionpos);
    return;
}
```

When `justmonitor=true`, AudioProc calls `process_samples(..., justmonitor=1)` which monitors local channels but skips BOTH encoding AND remote mixing, then returns immediately. The interval tracking (lines 810-860) and remote mixing are all skipped.

So the actual correct approach is: Do NOT use the `justmonitor` parameter at all. Instead, ensure `processBlock` passes `justmonitor=false` (current behavior), and rely on the fact that with zero local channels (achieved via `DeleteLocalChannel` from Gray Area 1 mitigation), there is nothing to encode. The remaining concern is:

1. **Session position tracking (lines 790-806):** This will increment during prelisten. The fix is either: (a) accept it (Connect resets these), or (b) skip the AudioProc call entirely and let the run thread + NJClient::Run() do the network I/O while processBlock just reads the mixed output. But option (b) requires significant refactoring.

Actually, looking again: the session position is reset by `Connect()` at line 921: `m_session_pos_ms=m_session_pos_samples=0`. So when the user connects normally after prelisten, these counters reset. The loop count (`m_loopcnt`) does NOT reset, but it's a running counter that starts fresh on each connection anyway (incremented in `on_new_interval`).

2. **Local channel encoding (lines 1993-2039):** With stale `m_locchannels`, encoding runs but with `broadcasting=false` (reset by `_reinit()`), so `bcast_active` is false and no blocks are queued. Actually, `_reinit()` doesn't clear `broadcasting` either. Let me check.

After further analysis, the safest approach is:

**In `processBlock`, change the AudioProc call to pass `prelisten_mode` as justmonitor:**
```cpp
bool jm = prelisten_mode.load(std::memory_order_relaxed);
client->AudioProc(inPtrs, numInputChannels, outPtrs, kTotalOutChannels,
                  numSamples, static_cast<int>(storedSampleRate),
                  jm, hostPlaying);
```

But this means remote audio is NOT mixed into the output (because `justmonitor=1` skips remote mixing). The output will be silent even though remote audio is being received and decoded.

**The real solution:** During prelisten, call AudioProc with `justmonitor=false` (to get remote mixing), but ensure that:
- Local channels are deleted (Gray Area 1, Option B) so nothing encodes
- Accept session position increment (reset on real Connect)
- Accept that `on_new_interval()` runs (safe with empty `m_locchannels`)

This means: **the current plan's approach of NOT changing the justmonitor parameter is actually closer to correct**, as long as local channels are cleared. The plan's comment "prelisten connects with justmonitor" is misleading but the code behavior is what matters.

**Final recommendation:**
1. Do NOT change the `justmonitor` parameter in the AudioProc call
2. DO clear local channels before prelisten Connect (Gray Area 1, Option B)
3. The plan's volume scaling in Task 2.2 is sufficient as-is
4. Session position corruption is self-healing (reset on next Connect)

## Gray Area 5: Username from `GetUser()` Before Connect

**Risk Level:** CRITICAL

### Finding: `Disconnect()` Clears `m_user` to Empty String

The plan's PrelistenCommand handler (Task 3.1) does:
```cpp
if (client->GetStatus() >= 0) client->Disconnect();
// ...
std::string username = "[preview]" + std::string(client->GetUser());
client->Connect(addr.c_str(), username.c_str(), "");
```

`Disconnect()` at line 869 sets `m_user.Set("")`. [VERIFIED: njclient.cpp line 869]

Even without the explicit Disconnect, `Connect()` calls `Disconnect()` internally at line 919. [VERIFIED: njclient.cpp line 919]

After either call, `client->GetUser()` returns `""`. The resulting username is just `"[preview]"` with no actual name.

**Impact:** The prelisten user appears on the server as `[preview]` instead of `[preview]musician_name`. Other users cannot identify who is previewing.

### Recommended Mitigation

Use `processor.lastUsername` instead of `client->GetUser()`:

```cpp
std::string uname = processor.lastUsername.toStdString();
if (uname.empty()) uname = "anonymous";
std::string username = "[preview]" + uname;
```

`processor.lastUsername` defaults to `"anonymous"` (line 140 of JamWideJuceProcessor.h) and persists across connections. [VERIFIED: JamWideJuceProcessor.h line 140]

## Gray Area 6: OSC Interaction During Prelisten

**Risk Level:** MEDIUM

### Finding: OSC Server Broadcasts Prelisten Remote Users

The OSC server's 100ms send loop reads `processor.cachedUsers` (via `sendDirtyRemoteUsers`, `sendRemoteVuMeters`, `sendRemoteRoster` in OscServer.cpp). During prelisten, `handleUserInfoChange` in NinjamRunThread updates `cachedUsers` with the prelisten room's users. [VERIFIED: NinjamRunThread.cpp lines 291-324, OscServer.cpp lines 316-318]

**Impact:** If OSC is enabled:
- TouchOSC surfaces will show prelisten room users on remote faders
- OSC roster broadcasts (`/JamWide/remote/{n}/name`) will contain preview usernames
- VU meters will reflect prelisten audio levels

When the user stops prelisten and joins a real session, the roster updates normally. But during prelisten, the OSC surface shows misleading information.

### Additional Side-Effects Confirmed

1. **Channel strip area:** `UserInfoChangedEvent` triggers `refreshChannelStrips()` which populates the mixer with prelisten users. [VERIFIED: JamWideJuceEditor.cpp lines 372-374]

2. **Video companion:** `handleUserInfoChange` calls `videoCompanion->onRosterChanged()` with prelisten users. [VERIFIED: NinjamRunThread.cpp lines 319-323]

3. **Connection bar:** The connection bar shows "Connected" (green dot) and "Connecting..." during prelisten because it reads `cached_status`. [VERIFIED: ConnectionBar.cpp lines 323-328, 467-470]

### Recommended Mitigation

1. **Connection bar:** When `prelisten_mode` is true, the StatusChangedEvent handler should NOT update the connection bar. Or: add a prelisten-specific status display.

2. **Channel strips:** In the `UserInfoChangedEvent` handler, skip `refreshChannelStrips()` when `prelisten_mode` is true. The channel strip area should remain empty during prelisten.

3. **OSC:** In `OscServer::sendDirtyRemoteUsers` and `sendRemoteRoster`, check `processor.prelisten_mode` and skip broadcasting. Or: accept that OSC shows preview data (less confusing than it sounds -- the users are real, just in a different room).

4. **Video companion:** In `handleUserInfoChange`, skip the video companion roster update when `prelisten_mode` is true.

Minimum viable fix: Suppress channel strip refresh and connection bar update during prelisten. OSC and video can be addressed later if problematic.

## Gray Area 7: Session Position State Persistence

**Risk Level:** LOW (self-healing)

### Finding: Session Position Increments During Prelisten But Resets on Connect

When `prelisten_mode` is active and `justmonitor=false` (current plan), `AudioProc` increments `m_session_pos_ms` and `m_session_pos_samples` at lines 790-806. The `m_loopcnt` also increments at each interval boundary via `on_new_interval()`. [VERIFIED: njclient.cpp lines 790-806, 2672]

However, `Connect()` resets `m_session_pos_ms=m_session_pos_samples=0` at line 921. [VERIFIED: njclient.cpp line 921]

The `m_loopcnt` does NOT reset on Connect or Disconnect. But `m_loopcnt` is only used for `GetLoopCount()` which feeds `uiSnapshot.interval_count`. This is a display-only value.

**Impact:** During prelisten, `uiSnapshot.interval_count` and `session_elapsed_ms` will show values from the prelisten session. After transitioning to a real session via Connect, `session_elapsed_ms` resets to 0. `interval_count` continues from the prelisten value (cosmetic issue only).

### Recommended Mitigation

No action required. Session position is self-healing on Connect. The `m_loopcnt` non-reset is a pre-existing behavior unrelated to prelisten. If desired, reset `uiSnapshot.interval_count` to 0 in the ConnectCommand handler.

## Open Questions

1. **Should prelisten users appear in the server's user list?**
   - What we know: Protocol limitation -- server always registers connected clients
   - What's unclear: Whether some NINJAM servers reject zero-channel clients
   - Recommendation: Plan already has fallback (create one silent channel). Test with ninbot.com servers.

2. **Should the metronome play during prelisten?**
   - What we know: The metronome is mixed by NJClient based on `config_metronome` volume. During prelisten, the metronome will play at its configured volume.
   - Recommendation: Consider muting the metronome during prelisten (`config_metronome.store(0.0f)` on prelisten start, restore on stop).
