# Phase 15.1 Plan 06: m_locchan_cs Snapshot Replacement Summary

**Status:** Implementation complete. Awaiting human-verify checkpoint UAT (Task 5).

## One-liner

Replaced `m_locchan_cs.Enter/Leave` at the four audit-cited audio-path sites (`process_samples` 1961/2118, `on_new_interval` 2698/2721) with an SPSC-mediated `LocalChannelMirror[MAX_LOCAL_CHANNELS]` populated by `drainLocalChannelUpdates()` at the top of `AudioProc`. Added Codex HIGH-3 generation-gated deferred-free for canonical `Local_Channel` objects. **Zero `Local_Channel*` / `lc_ptr` / `void*` escape-hatch in the mirror â€” Codex HIGH-2 closed.** Closes AUDIT CR-02.

## What Landed

### Run-thread â†’ audio-thread state propagation

`m_locchan_update_q` (`SpscRing<jamwide::LocalChannelUpdate, 32>`) carries variant payloads from `spsc_payloads.h` (UNTOUCHED â€” Wave-0 finality preserved):

| Mutator | Publishes |
|---|---|
| `SetLocalChannelInfo` (Add path) | `LocalChannelAddedUpdate` (full field set) |
| `SetLocalChannelInfo` (Update path) | `LocalChannelInfoUpdate` |
| `SetLocalChannelMonitoring` | `LocalChannelMonitoringUpdate` (set_* flags identify partial) â€” also `LocalChannelAddedUpdate` if implicitly creating |

| `SetLocalChannelProcessor` (deviation #1) | `LocalChannelProcessorUpdate` on a separate `m_locchan_processor_q` ring |

All publishes happen AFTER the canonical `m_locchan_cs.Enter()/Leave()` block. The canonical mutex is preserved for run/UI/network-thread consistency â€” only the audio-thread side is unlocked. On overflow, `m_locchan_update_overflows` is bumped (M-8 style observable).

### Audio thread (`AudioProc` top of function)

```cpp
drainLocalChannelUpdates();
m_audio_drain_generation.fetch_add(1, std::memory_order_release);
```

Drain applies queued variants to `m_locchan_mirror[u.channel]` via `std::visit`; release-store on the generation counter signals to the run thread that the audio thread has observed everything published before the matching `acquire`-load on the run side.

### `process_samples` and `on_new_interval` rewrite

Both functions now iterate `for (int ch = 0; ch < MAX_LOCAL_CHANNELS; ++ch)` and read `m_locchan_mirror[ch]`. The same `lc->channel_idx >= m_max_localch && !(lc->flags & 2)` gating is preserved (using `ch >= m_max_localch && !(lcm.flags & 2)`).

The legacy `lc->m_bq.AddBlock(...)` calls in both functions are intentionally dormant in 15.1-06 â€” they will be wired in 15.1-07b which lands in the same wave. **During the brief window between 15.1-06 and 15.1-07b, broadcast on local channels is silent.** This is dev-test only and is the documented boundary of the human-verify UAT in Task 5.

### Codex HIGH-3 deferred-free protocol (`DeleteLocalChannel` + `drainLocalChannelDeferredDelete`)

```
Run thread (DeleteLocalChannel):
  1. Under m_locchan_cs: detach Local_Channel from m_locchannels (no delete).
  2. publish_gen_target = m_audio_drain_generation.load(acquire) + 1
  3. m_locchan_update_q.try_push(LocalChannelRemovedUpdate{ch})
  4. yield-spin until m_audio_drain_generation.load(acquire) >= publish_gen_target
     (max 200ms; on timeout â†’ leak, RT-safety > memory hygiene)
  5. m_locchan_deferred_delete_q.try_push(victim)

Audio thread (AudioProc):
  drain â†’ set mirror[ch].active=false â†’ release-store on m_audio_drain_generation

Run thread (NinjamRunThread::run, every 20ms):
  client->drainLocalChannelDeferredDelete()  // pops victim, runs ~Local_Channel()
```

**Synchronicity choice:** synchronous-with-200ms-timeout (UI delete-button â‰¤ 200ms latency is acceptable). Documented decision in Decisions section.

## The Two Deviations from Plan

### Deviation #1: AUDIT H-03 disagreed with reality (Rule 3 â€” auto-fix blocking issue)

**Plan said:** "If `cbf` IS used somewhere, **STOP and signal a checkpoint**."

**Reality:** `juce/NinjamRunThread.cpp:374` registers an Instatalk PTT mute lambda on channel 4 at every connect. The audio thread (`process_samples`) MUST consult `cbf` for PTT muting â€” silently dropping it would break Phase 14.2's voice-chat feature.

**Architectural fix (no checkpoint needed):** The `cbf` is a function pointer + `cbf_inst` is an opaque `void*` owned by `JamWideJuceProcessor` (which outlives `NJClient`). Both are trivially copyable. Adding them to the mirror BY VALUE is **not** a HIGH-2 violation â€” they are NOT a back-pointer into a run-thread-owned `Local_Channel` object; they are a callback context owned by the audio plugin host.

**Implementation:** Defined a new POD `NJClient::LocalChannelProcessorUpdate { int channel; void(*cbf)(...); void* cbf_inst; }` **inline in `njclient.h`** (NOT in `spsc_payloads.h` â€” Wave-0 finality preserved). Added a separate `m_locchan_processor_q` SPSC ring for these updates. `SetLocalChannelProcessor` publishes a `LocalChannelProcessorUpdate`; `drainLocalChannelUpdates` applies it to `mirror[u.channel].cbf / cbf_inst`. `process_samples` invokes `lcm.cbf(src, len, lcm.cbf_inst)` from the mirror.

**Files modified:** `src/core/njclient.h`, `src/core/njclient.cpp`.

### Deviation #2: VU peak migrated to atomic mirror fields (Rule 2 â€” auto-add missing critical functionality)

**Why:** Removing `m_locchan_cs` from `process_samples` (which writes `lc->decode_peak_vol[2]`) AND from `GetLocalChannelPeak` (which reads it) would create a UI/audio-thread race on `lc->decode_peak_vol`. Two lock-free options were considered:
1. Keep peak on the canonical Local_Channel and accept the race (display-only field, but TSan would flag it).
2. Promote peak to `std::atomic<float>` on the mirror.

**Choice:** Option 2 â€” added `peak_vol_l` and `peak_vol_r` as `std::atomic<float>` to `LocalChannelMirror`. `process_samples` writes relaxed; `GetLocalChannelPeak` reads relaxed. The canonical `lc->decode_peak_vol` is no longer the UI source of truth; it remains in the canonical `Local_Channel` only as legacy storage that nothing reads anymore.

**Files modified:** `src/core/njclient.h` (mirror struct), `src/core/njclient.cpp` (`process_samples` peak write, `GetLocalChannelPeak` reads from mirror).
