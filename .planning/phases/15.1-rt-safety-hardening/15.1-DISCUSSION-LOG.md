## Post-Audit Codex Adversarial Review (2026-04-25)

After the auditor completed `15.1-AUDIT.md`, the user invoked `/codex:adversarial-review --base HEAD~2` with the focus *"review the findings and verify real time findings and hardening approach."* Codex returned a `needs-attention` verdict with two HIGH findings against the original CONTEXT.md + AUDIT.md combination:

### Finding C-01: Verification target excluded the JUCE audio callback path

**Codex's claim:** The success criteria mention TSan coverage of the JUCE audio callback path, but the documented test target is `./scripts/build.sh --tests` which sets `JAMWIDE_BUILD_JUCE=OFF`. So `processBlock` and the JUCE-side integration around `AudioProc` are never executed under TSan — sign-off can be green while the actual audio-host boundary remains unverified.

**Resolution:** Confirmed valid. CONTEXT.md updated:
- D-07 split TSan into two scopes: `build-test/` (NJClient core, fast CI) AND a new `build-tsan/` (`JAMWIDE_BUILD_JUCE=ON -fsanitize=thread`) building the standalone for end-to-end audio-callback verification.
- Code Context "Test build target" entry expanded to document both targets and explicitly state that `--tests` alone is insufficient.
- D-08 sign-off bar now reads "TSan-clean (both scopes)."
- Sub-plan 15.1-04 (SPSC infrastructure) gains the responsibility of adding the `--tsan` flag to `scripts/build.sh`.

### Finding C-02: Audit blessed shipping with unresolved CRITICAL audio-thread hazards

**Codex's claim:** The phase's Out-of-scope list says "codec algorithm changes" — but the audit's "deferred follow-up" section lists CR-08 (`start_decode` on audio thread, sessionmode), H-04 (`fread` in `runDecode`), and CR-11 (codec internal allocations) as deferred. Those are documented audio-thread file I/O and heap activity. Phase claiming "callback path is RT-safe" while deferring them is internally inconsistent — the auditor's CRITICAL count cannot reach zero on a run that finds CR-08 still present.

**Resolution:** Confirmed valid; the conflict was between "codec **algorithm/format** changes are out of scope" (correct) and "codec **call-site integration** is out of scope" (incorrect). Distinction added to CONTEXT.md scope:
- *Algorithmic/format edits to Vorbis/FLAC* remain out of scope.
- *Moving codec call-sites off the audio thread* (where `start_decode` is invoked from, where `fread` happens) is brought IN scope.
- AUDIT.md's old `15.1-09 (Deferred follow-ups)` section was renamed and rewritten as `15.1-09 — Codec call-site integration (IN scope, post-Codex realignment)`.
- New `15.1-10` is now the verification sub-plan (was originally `15.1-07`).
- The truly out-of-scope items (H-03 plugin-callback contract, codec algorithmic residual surviving Prealloc, MidiMapper) are listed separately so the deferral is honest and bounded.

### Why both findings were accepted

Both are contract-vs-promise problems, not subjective taste. The success criteria of a phase have to be testable; deferring CRITICAL items the same auditor that runs the verification *will* report is a circular failure mode — the phase couldn't pass its own audit. Adding the JUCE-enabled TSan target is a one-time scripts/build.sh extension, comparatively cheap; adding 15.1-09 to the sub-plan list is in-line with the audit's actual findings. Reverting either would require weakening the success criteria, which is the wrong direction.
