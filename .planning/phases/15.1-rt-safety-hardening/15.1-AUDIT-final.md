### Architecture-mirror audit follow-through

`15.1-MIRROR-AUDIT.md` identified ONE dead-but-live finding at the auth-reply lobby renormalize path
in `src/core/njclient.cpp` (originally cited as line 1405; the actual current location after subsequent
edits is **lines 1648-1653**):

```cpp
if (!m_max_localch)
{
  // went from lobby to room, normalize channel indices (in case the user deleted channels in the lobby)
  for (int x = 0; x < m_locchannels.GetSize(); x ++)
    m_locchannels.Get(x)->channel_idx = x;
}
```

This is the same Bug-A shape that 15.1-07b commit `3799e8a` already fixed in `_reinit`. The audit's
recommendation was either (a) delete it as a closure cleanup, or (b) explicitly flag as documented
technical debt. **Decision for this plan:** option (b) — document as VERIFIED-UNREACHABLE technical
debt. Rationale:

1. **Currently structurally unreachable in modern JamWide.** `_reinit()` resets
   `m_max_localch = MAX_LOCAL_CHANNELS` (njclient.cpp:745) on every connect cycle, so the
   `!m_max_localch` gate at line 1648 is always false on a normal first auth reply. The path
   activates only on a server-side lobby→room handoff that no JamWide deployment triggers, because
   the modern JamWide UI requires the user to specify a room before Connect.
2. **Plan scope discipline.** The 15.1-10 plan's executor prompt explicitly states "This plan does
   NOT modify production code." The architecture-mirror audit recommendation acknowledged option
   (b) as the safer choice with the rationale "currently unreachable in modern JamWide; activates
   only on a server-side lobby→room handoff that no JamWide deployment triggers."
3. **Tracked in deferred-items.md** for future cleanup if ever needed.

**Action:** added to `deferred-items.md` as a tracked technical-debt item with the same
remediation pattern as `3799e8a` (one-line surgical fix when ready).
