# Phase 15.1 Plan 05: DecodeState* Deferred-Delete SPSC

Audio-thread `delete chan->ds` and `delete chan->next_ds[0]` (7 sites identified by AUDIT CR-05/06/07) replaced with `m_deferred_delete_q.try_push` to a per-NJClient SPSC ring; run thread drains the queue at 20ms cadence and runs `~DecodeState()` off-thread. Codex M-8 overflow counter (`m_deferred_delete_overflows`) exposed via `GetDeferredDeleteOverflowCount()` for 15.1-10 phase-close gate.

## What Landed

**Audio thread is now free of DecodeState heap deallocations.** The seven audit-cited delete sites all enqueue onto a 256-slot SPSC ring instead of calling `delete` directly. The run thread (NinjamRunThread::run) drains the ring inside its 20ms locked block, where `~DecodeState()` runs the codec teardown / file close / buffer release off the audio thread. A graceful-shutdown drain after the run-loop exits ensures clean disconnect with zero leaked DecodeStates.

The Codex M-8 overflow counter mechanism is fully wired: when `try_push` would fail (queue full), the audio thread bumps `m_deferred_delete_overflows` and proceeds without deleting (RT-safety dominates memory hygiene at the audio callback boundary). 15.1-10 reads `GetDeferredDeleteOverflowCount()` and fails the phase if it's non-zero post-UAT — making "queue undersized for workload" a phase-failing architectural defect, not a tolerable silent drop.

The "capture-first" pattern (sites 4, 5, 7) preserves the audio-thread's exclusive ownership during the slot shuffle:

```cpp
// site 4 example
DecodeState* old_ds = userchan->ds;
chan = userchan->ds = userchan->next_ds[0];
userchan->next_ds[0] = userchan->next_ds[1];
userchan->next_ds[1] = 0;
deferDecodeStateDelete(m_deferred_delete_q, m_deferred_delete_overflows, old_ds);
```

Per RESEARCH § "Subtle note for the planner": the pointer-shuffle MUST run before the SPSC publish. Reversing the order (push first, then shuffle) would briefly let the run thread see the pointer while the audio thread still owns it — a use-after-free risk window. Capture-first eliminates that window.

## Codex M-8 Overflow Counter — Mechanism and Phase-Close Gate

The plan-checker (Codex) flagged the original "queue full = silently leak" semantics as too lenient. The revised plan keeps the audio-thread fallback (still leak-on-full to preserve RT-safety) but adds an observable counter that makes any overflow during UAT a phase-failing condition.

**Mechanism:**

```cpp
static inline void deferDecodeStateDelete(/*queue*/, /*counter*/, DecodeState*& p)
{
  if (p)
  {
    if (!q.try_push(p))
      overflow_counter.fetch_add(1, std::memory_order_relaxed);
    p = nullptr;
  }
}
```

**Public accessor (consumed by 15.1-10):**

```cpp
uint64_t GetDeferredDeleteOverflowCount() const noexcept {
    return m_deferred_delete_overflows.load(std::memory_order_relaxed);
}
```

Relaxed semantics are sufficient — this is an observability counter, not a synchronization primitive. The only consumer (15.1-10) reads it after the audio thread has been quiesced; no happens-before relationship is needed.

**Test verification of mechanism:** `test_deferred_delete.cpp::test_overflow_counter_increments_correctly` pushes 300 pointers without draining, observes that `~45` are rejected (SpscRing has effective capacity N-1 = 255 for N=256), and asserts the simulated counter equals the rejected count exactly. It then drains and re-pushes the orphans to verify recovery semantics. Final invariant: every constructed FakeDecodeState runs its destructor.

**1. Helper-factored try_push pattern.** The plan's `<action>` block explicitly specifies a static helper (`deferDecodeStateDelete`) to be called from each of the 7 sites — that's what was implemented. The helper-factored approach is what the plan prescribes verbatim; the literal grep would only match if you ignored the action block. No semantic deviation; the acceptance grep was looking for the wrong literal token.
