# Cross-AI Plan Review — Phase 12.1

## Codex Review

### Summary

The plan is directionally correct and maps to the real failure modes in the existing pipeline. The main weakness is that it underspecifies state ownership and lifecycle behavior for auto/manual mode, reconnects, and iframe readiness. Without tightening those contracts, it can appear "fixed" while still failing intermittently.

### Strengths

- Targets the right pipeline stages: plugin broadcast, WebSocket delivery, iframe URL init, and postMessage resend
- Adding diagnostics at each hop is appropriate for a race-condition-heavy path
- Including `&buffer=N` in the initial URL is the correct mitigation for "message sent before VDO.Ninja is ready"
- Re-sending on iframe `load` is well aligned with the likely lifecycle issue
- Manual override is scoped narrowly and supports the phase goal
- Auto/manual split matches the product need

### Concerns

- **HIGH**: Hardcoded 8000ms default is wrong source of truth. Delay should come from current BPM/BPI or last plugin-provided value, not an assumed 8s.
- **HIGH**: Manual mode behavior incomplete — switching back to auto may leave old manual value active until a future BPM/BPI change arrives. Need separate `lastAutoDelayMs` vs `activeDelayMs`.
- **HIGH**: Reconnect/bootstrap ordering undefined. Plugin already sends `config` then cached `bufferDelay`; plan should verify that ordering, not replace with local fallback.
- **MEDIUM**: No acknowledgement/ready signal from VDO.Ninja iframe — `onload` + retries may remain flaky under slow loads.
- **MEDIUM**: `postMessage(..., '*')` security — should use specific iframe origin rather than wildcard.
- **MEDIUM**: Slider at 0ms effectively disables buffering — should label explicitly as "No delay" or set minimum > 0.
- **MEDIUM**: Footer behavior undefined for disconnected/pre-config states.
- **MEDIUM**: Iframe reload triggers beyond initial load not mentioned — settings changes that rebuild iframe must preserve delay.
- **LOW**: Console diagnostics could be noisy — use structured prefix and debug flag.
- **LOW**: No automated tests — race-sensitive feature relying only on manual testing.

### Suggestions

- Replace "8000ms default" with two state values: `lastAutoDelayMs` (from plugin) and `activeDelayMs` (currently applied)
- Define precise mode-switch contracts (auto applies lastAutoDelayMs immediately, manual still updates lastAutoDelayMs silently)
- Add verification matrix for race conditions (bufferDelay before iframe, iframe before bufferDelay, reconnect, iframe reload, manual→auto without new broadcast)
- Use specific `targetOrigin` for postMessage
- Add tests for URL builder, mode transitions, iframe reload
- Gate diagnostics behind debug flag with structured `VideoSync:` prefix

### Risk Assessment

**MEDIUM** — Plan is close but not fully specified where bugs survive: race conditions, reconnects, mode transitions. Tighten state model and the phase is very achievable.

### Key Concerns (Priority Order)

1. **State model needs two separate delay values** — `lastAutoDelayMs` (from plugin, always updated) vs `activeDelayMs` (currently applied to iframe). Without this, auto↔manual transitions break.
