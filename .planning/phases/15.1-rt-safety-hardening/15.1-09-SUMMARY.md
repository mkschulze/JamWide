**Status:** Implementation complete (autonomous; UAT deferred to phase-end 15.1-10 per user direction).

## One-liner

Closes AUDIT CR-08 (start_decode on audio thread — already eliminated by 07a's sessionmode no-op refactor; this plan keeps that closure structural via m_arm_request_q SPSC for any future sessionmode re-enable) and AUDIT H-04 (DecodeState::runDecode `fread` on audio thread IN STEADY STATE — Codex HIGH-1 audit invariant established: every audio-thread-visible DecodeState has decode_fp == nullptr; FILE* lives on the run-thread side in `m_sessionmode_file_readers`; refill loop feeds DecodeMediaBuffer per tick). ## What Landed

### Architecture: file-reader thread / refill queue (Codex HIGH-1 option (b))

The pre-15.1 audio-thread path was:
```
mixInChannel sessionmode rearm (audio thread) → start_decode (open FILE*, alloc, decode loop)
                                              → ds with decode_fp != nullptr
                                              → audio thread runDecode → fread(decode_fp)  [H-04]
```

The 15.1-07a refactor already collapsed the audio-thread sessionmode rearm to an early-return no-op (mixInChannel sessionmode branch defer-deletes any in-flight ds and returns). So the audio thread no longer calls start_decode directly today.

**However**, the run-thread DOWNLOAD_INTERVAL_BEGIN handler (njclient.cpp:1968) still produced audio-thread-visible DecodeStates with `decode_fp != nullptr` (the file-on-disk replay path; `start_decode(guid, flags, 0, NULL)` → opens a FILE* via `fopenUTF8`). Once that DS reached the audio thread mirror, subsequent `chan->runDecode(256)` calls reached the H-04 `fread` path IN STEADY STATE.

**This plan's structural fix:** every run-thread DS construction site that produces an audio-thread-visible DS invokes `inversionAttachSessionmodeReader(ds)` immediately after start_decode returns. That helper:

1. Captures `ds->decode_fp` into a local `fp_for_runthread`.
2. Sets `ds->decode_fp = nullptr` (the **Codex HIGH-1 audit-invariant assignment**).
3. Allocates a fresh `DecodeMediaBuffer`, sets `ds->decode_buf` to it, calls `AddRef` (DS holds 1 ref + SessionmodeFileReader holds another).
4. Primes the buffer with one CHUNK_BYTES read from the file.
5. Pushes a `SessionmodeFileReader{file=fp_for_runthread, buffer=buf, eof=...}` onto `m_sessionmode_file_readers`.

After the helper returns, the audio thread can safely receive `ds` via PeerNextDsUpdate; its `runDecode` will reach `decode_buf->Read` (lock-free SPSC drain from 15.1-07c) — NEVER `fread(decode_fp)`.

On every run-thread tick, `refillSessionmodeBuffers`:
- Iterates `m_sessionmode_file_readers`.
- For each entry whose `buffer->GetRefCount() <= 1` (the audio side has Released its share via the deferred-delete chain), `fclose`s the FILE* and `Release`s the buffer (refcnt → 0 → delete).
- For each active entry, reads up to 4 CHUNK_BYTES (16 KB/file/tick → 800 KB/s/file at 20 ms cadence) and pushes into the corresponding DecodeMediaBuffer.
