# Phase 14.2: Code Review Report

**Reviewed:** 2026-04-16T12:00:00Z
**Depth:** standard
**Files Reviewed:** 13
**Status:** issues_found

## Summary

This review covers the Phase 14.2 instamode video sync feature, which adds:
- Instamode latency measurement state machine in NJClient (C++ audio/run thread)
- PTT (push-to-talk) for Instatalk channel with processor callback
- Measured delay broadcast from NinjamRunThread to VideoCompanion WebSocket server
- Three-tier delay priority (manual > measured > calculated) in companion TypeScript UI
- Syncing overlay and sync mode display in companion footer

The core state machine design is well-structured with clear thread ownership. The SPSC atomic pattern for cross-thread measurement is correct. However, a typo in the enum creates a maintenance hazard, `validatedClients_` is declared but unused (security gap), and there are minor state reset inconsistencies.

## Critical Issues

### CR-01: validatedClients_ declared but never populated -- security bypass

**File:** `juce/video/VideoCompanion.h:174-180` and `juce/video/VideoCompanion.cpp`
**Issue:** The header declares `validatedClients_` with a detailed security comment explaining that "only clients whose Origin and Host headers matched the companion URL are added here on Open. Broadcasts iterate this set instead of wsServer_->getClients() so a rogue webpage cannot receive config/roster even briefly." However, `validatedClients_` is never populated (no insert in the `onClientMessage` callback) and never read (all 6 broadcast sites use `wsServer_->getClients()` directly). This means the documented Origin/Host validation is not implemented -- any local WebSocket client can connect to 127.0.0.1:7170 and receive all config, roster, and beat heartbeat messages.

The attack surface is limited because the server binds to `127.0.0.1` only, but a malicious local webpage (or any process on the same machine) can observe the NINJAM session roster and room configuration. The `wsMutex_` comment also references `validatedClients_` as part of its guarded set, adding to the confusion.

**Fix:** Either implement the Origin validation as documented, or remove the dead declaration and update the security comments to accurately reflect the actual behavior. If implementing:
```cpp
// In startWebSocketServer callback, on Open:
if (msg->type == ix::WebSocketMessageType::Open)
{
    // Validate Origin header matches companion URL origin
    auto headers = msg->openInfo.headers;
    auto origin = headers.find("Origin");
    if (origin != headers.end() && origin->second == "https://jamwide.audio")
    {
        std::lock_guard<std::mutex> lock(wsMutex_);
        validatedClients_.insert(&webSocket);
    }
    sendConfigToClient(webSocket, snap);
}
else if (msg->type == ix::WebSocketMessageType::Close)
{
    std::lock_guard<std::mutex> lock(wsMutex_);
    validatedClients_.erase(&webSocket);
}
```

Then replace all `wsServer_->getClients()` broadcast loops with iteration over `validatedClients_`.

## Warnings

### WR-01: Enum value typo -- kInstaCapured should be kInstaCaptured

**File:** `src/core/njclient.h:264`
**Issue:** The enum value `kInstaCapured` is missing the 't' in "Captured". This typo is consistently used across njclient.h, njclient.cpp (lines 2475, 2758), and the plan document, so it does not cause a build error. However, it is a maintenance hazard: any future developer or code searcher looking for "captured" will not find it, and the inconsistency between "Captured" (intended) and "Capured" (actual) suggests a copy-paste propagation of an original typo.

**Fix:**
```cpp
enum InstaMeasState : int {
    kInstaMeasIdle = 0,
    kInstaCaptured = 1,  // was kInstaCapured
    kInstaMeasured = 2,
    kInstaConsumed = 3
};
```
Update all 4 usage sites (njclient.h:264, njclient.cpp:2479, njclient.cpp:2758, NinjamRunThread.cpp:533 via kInstaMeasured comparison).

### WR-02: cachedSyncMode_ not reset in deactivate() -- stale state on relaunch

**File:** `juce/video/VideoCompanion.cpp:588-620`
**Issue:** The `deactivate()` method resets `cachedDelayMs_` to 0, clears `currentRoom_`, `currentPush_`, `currentCompanionUrl_`, etc., but does not reset `cachedSyncMode_`. If the previous session had a measured delay, `cachedSyncMode_` will still be `"measured"` when the user relaunches via `launchCompanion()`. The `SessionSnapshot` at line 212 captures `cachedSyncMode_` and sends it to the first connecting WS client via `sendConfigToClient()`. This could cause the companion to display "Sync: measured" with a stale delay value of 0ms.

**Fix:**
```cpp
void VideoCompanion::deactivate()
{
    // ... existing resets ...
    cachedDelayMs_ = 0;
    cachedSyncMode_.clear();  // Add this line
    lastBroadcastBeat_ = -1;
    // ...
}
```

### WR-03: Memory ordering inconsistency in insta measurement state machine

**File:** `src/core/njclient.cpp:2474-2479` and `src/core/njclient.cpp:2758-2762`
**Issue:** The insta measurement state machine uses `memory_order_relaxed` for the guard read (`insta_meas_state_.load`) but `memory_order_release` for the subsequent stores. At line 2475, the state check uses relaxed ordering, then at 2477 the user pointer is stored with relaxed, and at 2478-2479 the timestamp and state are stored with release. The concern is that the run thread (consumer in `pollInstamodeDelay`) reads with `memory_order_acquire` (line 532), which correctly pairs with the release stores. However, the guard read at 2475 using relaxed could theoretically allow two audio thread iterations to both see `kInstaMeasIdle` and race to write conflicting values, though in practice the audio thread is single-threaded so this cannot happen.

This is not a bug because the audio thread is the sole writer and is single-threaded. However, using `memory_order_relaxed` for the guard while documenting a state machine with sequential transitions is misleading. A reader might assume multi-writer safety.

**Fix:** Add a comment clarifying single-writer assumption:
```cpp
// Audio thread only (single writer) -- relaxed load is safe.
// The release store below synchronizes with acquire load in pollInstamodeDelay().
if (userchan->ds
    && insta_meas_state_.load(std::memory_order_relaxed) == kInstaMeasIdle)
```

### WR-04: Instatalk channel (ch 4) not deleted during PrelistenCommand cleanup

**File:** `juce/NinjamRunThread.cpp:795`
**Issue:** The PrelistenCommand handler deletes local channels 0-3 in reverse order (`for (int ch = 3; ch >= 0; --ch) client->DeleteLocalChannel(ch)`), but in a normal session, channel 4 (Instatalk) is also created (line 359). If the user connects to a normal session (creating channels 0-4), then initiates a prelisten, channel 4 will persist. When the prelisten completes and a new normal connection is made, `SetLocalChannelInfo(4, ...)` at line 359 will still work (it creates or updates), but the stale channel 4 from the prior session might cause the server to receive an Instatalk channel announcement during the prelisten connection.

**Fix:**
```cpp
// Step 2: Clear all local channels including Instatalk (ch 4)
for (int ch = 4; ch >= 0; --ch)
    client->DeleteLocalChannel(ch);
```

### IN-03: Duplicate broadcast pattern across VideoCompanion methods

**File:** `juce/video/VideoCompanion.cpp:450-456, 486-492, 512-518, 536-541, 555-561, 597-604`
**Issue:** Six separate methods all follow the same pattern: lock `wsMutex_`, check `wsServer_` null, get clients, iterate, check `ReadyState::Open`, send JSON string. This repeated pattern could be factored into a `broadcastJson(const juce::String& json)` helper to reduce duplication and ensure consistent behavior (e.g., if skip-dead-connection logic changes, it only needs to change in one place).

**Fix:** Extract a private helper:
```cpp
void VideoCompanion::broadcastJson(const juce::String& json)
{
    // Caller must NOT hold wsMutex_
    std::lock_guard<std::mutex> lock(wsMutex_);
    if (!wsServer_) return;
    auto clients = wsServer_->getClients();
    for (auto& client : clients)
    {
        if (client->getReadyState() != ix::ReadyState::Open) continue;
        client->send(json.toStdString());
    }
}
```
