<objective>
Move codec call-sites that perform allocation/blocking I/O off the audio thread. **Call-site integration only — no codec algorithm or format edits.** Closes AUDIT CR-08 (`start_decode` on audio thread, sessionmode), H-04 (`fread` in `DecodeState::runDecode`), and CR-11 partial (decoder internal allocations are mitigated by 15.1-08's Prealloc; their first-frame residual is structurally moved off the audio thread because the decoder constructor itself now runs on the run thread).

Promoted IN-scope after the post-AUDIT Codex adversarial review per CONTEXT D-05 / DISCUSSION-LOG § "Finding C-02". Without this sub-plan the success criterion "auditor reports zero CRITICAL on audio path" cannot be met — CR-08 is a CRITICAL finding that 15.1-02..08 do not touch.

**Architecture per RESEARCH § "Codec Internal Allocation Residual" + AUDIT § "15.1-09":**

1. The audio thread no longer constructs decoders. The sessionmode-rearm path at `src/core/njclient.cpp:2419` (mixInChannel) currently does:
   ```cpp
   start_decode(guid, ...);   // CRITICAL: heap alloc + fopenUTF8 + new VorbisDecoder/FlacDecoder + while-loop fread
   ```
   After this plan: audio thread emits a `DecodeArmRequest` onto an SPSC queue and continues without constructing anything.

2. The run thread (existing 20ms-cadence loop in `juce/NinjamRunThread.cpp::run()`) drains the arm-request queue *inside* the locked block, calls `start_decode` itself (it's not RT — `fopenUTF8` and `new VorbisDecoder/FlacDecoder` are fine here), and publishes the resulting `DecodeState*` to the audio thread by reusing the **existing** 15.1-07 `PeerNextDsUpdate` mechanism: the run thread sets `next_ds[slot_idx]` of the matching peer-channel via the same SPSC update protocol that 15.1-07 uses for live-interval prearm. The audio thread's existing advance pattern (`chan->ds = chan->next_ds[0]`) does the swap on the next interval boundary or seek-exhaustion check.

3. `DecodeState::runDecode`'s `fread` path (H-04) is unreachable from the audio thread by construction once start_decode lives on the run thread — sessionmode files are opened on the run thread, the initial fill loop (`while (Available() <= 0) runDecode()` at `njclient.cpp:1916`) runs on the run thread, and the audio thread only ever interacts with already-buffered DecodeStates that were produced off-thread.

**Sequencing:** This plan is in Wave 4 alongside 15.1-08 (which depends on this plan to land first per RESEARCH § "Sequencing of 15.1-08 vs 15.1-09" — the decoder constructor must run off the audio thread before its Prealloc calls are guaranteed safe to make audio-thread-side concurrent). 15.1-08's `depends_on` already lists this plan.

**Out of scope (NOT in this plan):** Algorithmic changes to libvorbis/libflac decoder internals; the residual heap activity inside libogg `_ogg_realloc` and libFLAC stream-decoder during the first 3-5 frames per stream. Those are codec-internal and CONTEXT.md says codec algorithm/format changes are out of scope. They are mitigated by the call-site move (the first-frame work happens on the run thread now) plus 15.1-08's WDL-wrapper Prealloc (which covers the steady-state path). Any residual algorithmic bound surfaces in 15.1-10 verification.

Output: CR-08 cleared; H-04 cleared; mixInChannel allocation-free and lock-free at steady state per RESEARCH § "mixInChannel Allocation-Free Invariant".

    static void test_concurrent_arm_handoff() {
        TEST("Concurrent producer/consumer: 1000 arm requests, all delivered exactly once");
        jamwide::SpscRing<jamwide::DecodeArmRequest, 256> arm_ring;
        jamwide::SpscRing<jamwide::PeerNextDsUpdate, 256> publish_ring;
        std::atomic<int> started_decoder_count{0};
        constexpr int kTotal = 1000;

        // "Run-thread" worker: drain arms, "construct" stub DecodeState (just an int*),
        // publish PeerNextDsUpdate. Mirrors NJClient::drainArmRequests shape.
        std::atomic<bool> stop{false};
        std::thread runner([&]{
            int delivered = 0;
            while (!stop.load() || delivered < kTotal) {
                arm_ring.drain([&](const jamwide::DecodeArmRequest& req) {
                    ++started_decoder_count;
                    jamwide::PeerNextDsUpdate upd;
                    upd.slot = req.slot;
                    upd.channel = req.channel;
                    upd.slot_idx = req.slot_idx;
                    upd.ds = reinterpret_cast<DecodeState*>(uintptr_t(0xDEADBEEF));  // stub
                    publish_ring.try_push(upd);
                    ++delivered;
                });
                if (delivered >= kTotal) break;
                std::this_thread::yield();
            }
        });

        for (int i = 0; i < kTotal; ++i) {
            jamwide::DecodeArmRequest req{};
            req.slot = i % 64;
            req.channel = i % 8;
            req.slot_idx = i % 2;
            while (!arm_ring.try_push(req)) {
                std::this_thread::yield();  // ring full — yield to runner
            }
        }
        stop.store(true);
        runner.join();

        if (started_decoder_count.load() != kTotal) {
            FAIL("started_decoder_count mismatch (lost arm requests)"); return;
        }

        // Drain publish_ring; verify N entries received.
        int published = 0;
        publish_ring.drain([&](const jamwide::PeerNextDsUpdate&){ ++published; });
        if (published != kTotal) { FAIL("publish_ring count mismatch"); return; }

        PASS();
    }
