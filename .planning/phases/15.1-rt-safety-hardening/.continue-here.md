After landing 07b (commits `dbdaf98`, `edb2769`, `b43aeb6`), three regressions surfaced through real-DAW UAT. Each got a surgical fix:

| Fix | Commit | What it fixed |
|---|---|---|
| `e631aef` | `fix(15.1-07b): restore on_new_interval bcast_active state machine` | The 07b executor port lost the legacy state-transition logic — was emitting boundary AND stop markers unconditionally every interval, never set `lcm.bcast_active=true` for normal channels, so per-block sample push at process_samples:2241 never fired |
| `1633bfd` | `fix(15.1-07b): guard NJClient::Run encoder loop against null m_netcon` | `Disconnect()` clears `m_netcon` then tears down per-channel state. | `3799e8a` | `fix(15.1-07b): drop _reinit channel_idx renormalize that broke ch0 VU` | The CURRENT bug. `_reinit` was renormalizing channel_idx on every Disconnect (→ Connect's first step). ## Bug A root cause (so a fresh context can verify the fix)

Sequence that produced "local Ch1-4 VU dead" in build 257-260:

1. Plugin loads. `m_locchannels` empty.
2. **First tick of `NinjamRunThread::run()`** → `syncInstatalkBroadcast()` checks `want=true (probe-on default) != lastInstatalkPtt_=false (member default)` → calls `SetLocalChannelInfo(4, …)` BEFORE the user has clicked Connect. List becomes `[{channel_idx=4}]`. AddedUpdate{ch=4} published, `mirror[4].active=true`.
3. User clicks Connect → `client->Connect()` → calls `Disconnect()` → calls `_reinit()` which had:
   ```cpp
   for (x = 0; x < m_locchannels.GetSize(); x ++) {
     Local_Channel *c=m_locchannels.Get(x);
     c->channel_idx = x;  // legacy "lobby renormalize"
     ...
   }
   ```
   This silently overwrites `channel_idx=4 → channel_idx=0` on the canonical Local_Channel. The mirror is NOT told. `mirror[4].active=true` lingers, and `mirror[0].active=false` stays.
4. Auth completes → handleStatusChange "Normal connection" branch → `SetLocalChannelInfo(0, "Ch1", …)`:
   - Search finds the renamed entry with `channel_idx=0`
   - `was_add=false` → publishes **InfoUpdate** (does NOT write `m.active=true`)
5. `mirror[0].active` stays false, the audio thread skips ch0 in `process_samples`, VU stays at 0.

**Diagnostic that proved this** (preserved at `.planning/debug/build253-vu-and-disconnect.md`):
- Visitor counters per channel (`add_visits`, `remove_visits`, `info_visits`, `monitor_visits`) — log showed `ch0{add=0, inf=1}` ← the smoking gun
- stderr-printf at every `SetLocalChannelInfo` call — log showed the literal sequence, including the `was_add=0 list_size_before=1` for ch=0 even though only ch=4 had been added prior

**Fix (`3799e8a`):** delete the `c->channel_idx = x;` line in `_reinit`. The legacy renormalize is dead code for JamWide because the UI doesn't expose Local_Channel deletion (channels are fixed at 0-4). Long comment in the source explains the architectural reason.

The duplicate renormalize at line 1384 (auth-reply handler, gated on `!m_max_localch`) is dead in modern JamWide (lobby mode unused) and was left untouched — flagging here in case a future audit wants to clean it up.

## Critical Anti-Patterns

| Severity | Pattern | Why it matters |
|----------|---------|----------------|
| blocking | DO NOT introduce locks, heap allocations, or I/O on the audio path | The whole phase exists to remove these. Use `realtime-audio-reviewer` agent to verify any audio-path code change. |
| blocking | DO NOT call `delete` on `DecodeState*` from any audio-thread-reachable function | Use `deferDecodeStateDelete` (15.1-05 helper). Pattern: capture old pointer first, advance slot, then defer-delete. |
| blocking | DO NOT introduce `lc_ptr` / `user_ptr` raw back-pointers in payload definitions | Codex HIGH-2. The 15.1-04 grep gate fails the build on regression. `spsc_payloads.h` is FINAL. |
| blocking | DO NOT block the audio thread waiting for the run-thread to drain a queue | Use `try_push` (returns false on full); increment overflow counter; never spin or sleep on the audio thread. |
| blocking | DO NOT mutate `Local_Channel.channel_idx` outside `SetLocalChannelInfo`/`SetLocalChannelMonitoring`'s explicit add path | This was the Bug A root cause. The audio-thread mirror is indexed BY channel_idx — any silent renormalize strands the mirror. If you ever need to renormalize, publish `RemovedUpdate{old_idx} + AddedUpdate{new_idx}` with the full field set. |
| advisory | DO NOT trust an executor's "verify only X paths" UAT scope when X excludes a user-visible happy path | The 15.1-06 UAT specifically excluded broadcast verification ("non-broadcast paths only"). I approved it. The user lost broadcast for two days. See `feedback_uat_scope_redflags.md` in user memory. |
