# Phase 14.1: Audio Prelisten - Context

**Gathered:** 2026-04-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Add audio prelisten to the server browser — users can hear what's happening in a NINJAM room before joining. Listen button per populated server row, one-at-a-time via main NJClient in receive-only mode, dedicated volume control.

</domain>

<decisions>
## Implementation Decisions

### UX Trigger
- Dedicated Listen button per server row (not click-to-listen, not auto-on-hover)
- Single-click still fills address bar, double-click still connects normally

### Concurrency
- One prelisten at a time — clicking Listen on a new room stops the previous one

### Visibility
- Invisible preferred — connect with zero local channels, `[preview]` username prefix
- Protocol limitation: server still sees user with 0 channels, may show in user list

### Audio Output
- Mix into same plugin output with dedicated prelisten volume control (0.0-1.0)
- Works in both VST3/plugin and standalone modes

### Architecture
- Approach B: Reuse main NJClient with mode switch (not second instance)
- Trade-off accepted: cannot prelisten while already in a session

### Empty Rooms
- No Listen button for rooms with 0 users

</decisions>

<specifics>
## Specific Ideas

- `NJClient::process_samples()` has `justmonitor` flag that skips encoding/transmission
- `config_autosubscribe = 1` already set in constructor — prelisten leverages this
- `license_callback` can return 1 immediately to auto-accept
- Zero local channels means `SetLocalChannelInfo()` is never called
- `PrelistenStateEvent` provides connection feedback to UI
- `[preview]` prefix signals intent to other users in the room

</specifics>
