# Phase 15.1 Plan 07a: m_users_cs RemoteUserMirror Migration Summary

**Status:** Implementation complete. Awaiting human-verify checkpoint UAT (Task 5).

## One-liner

Replaced `m_users_cs.Enter/Leave` at the audio-path call sites (AudioProc, process_samples remote-user mix loop, on_new_interval remote-user advance, mixInChannel) with an SPSC-mediated `RemoteUserMirror[MAX_PEERS=64]` populated by `drainRemoteUserUpdates()` at the top of `AudioProc`. `mixInChannel` signature changed from `(RemoteUser*, int chanidx, …)` to `(int slot, int chanidx, …)` — audio thread no longer dereferences ANY run-thread-owned `RemoteUser` / `RemoteUser_Channel` object (Codex HIGH-2 closed). Added Codex HIGH-3 generation-gated deferred-free for canonical `RemoteUser` objects via `m_remoteuser_deferred_delete_q`. Mirror keyed by **STABLE SLOT** (NOT m_remoteusers list index) — mitigates the Bug-A-shape risk identified in `15.1-MIRROR-AUDIT.md`. **Closes AUDIT CR-01.**

## What Landed

### Mirror schema (`src/core/njclient.h`)

    int  user_index = 0;             // server-assigned, stable per-session
    int  submask = 0;

    int  solomask = 0;

};

RemoteUserMirror m_remoteuser_mirror[MAX_PEERS];
jamwide::SpscRing<jamwide::RemoteUserUpdate, 64> m_remoteuser_update_q;
jamwide::SpscRing<RemoteUser*, jamwide::REMOTE_USER_DEFERRED_DELETE_CAPACITY> m_remoteuser_deferred_delete_q;

**Codex HIGH-2 verification:** zero matches for `user_ptr|RemoteUser\s*\*\s*user|RemoteUser_Channel\s*\*` inside `RemoteUserMirror`/`RemoteUserChannelMirror` definitions in `src/core/njclient.h`. The mirror is populated by VALUE through the variant-mutation stream; nothing in the audio path dereferences run-thread-owned objects.

### Run-thread → audio-thread state propagation

`m_remoteuser_update_q` (`SpscRing<jamwide::RemoteUserUpdate, 64>`) carries variant payloads from the FINAL `spsc_payloads.h` (UNTOUCHED — Wave-0 finality preserved per Codex M-9):

| Mutator                                                    | Publishes |
|------------------------------------------------------------|-----------|
| `MESSAGE_USER_CHANNELNOTIFY` (auth handler, peer add/state) | `PeerAddedUpdate` (allocSlot success), `PeerChannelMaskUpdate` (mask change), `PeerRemovedUpdate` (chanpresentmask=0, with HIGH-3 publish-wait-defer dance) |
| `SetUserState` (UI vol/pan/mute on a peer)                 | `PeerVolPanUpdate` |
| `SetUserChannelState` (UI subscribe/mute/solo on a channel) | `PeerChannelMaskUpdate` (also nulls canonical ds/next_ds for unsubscribe; mirror's apply visitor defer-deletes the audio-thread-owned copies) |
| `MESSAGE_SERVER_DOWNLOAD_INTERVAL_BEGIN` (download arrives) | `PeerNextDsUpdate` (zero-guid silence variant or freshly-decoded ds) |
| `RemoteDownload::startPlaying` (download complete)         | `PeerNextDsUpdate` + `PeerCodecSwapUpdate` |
| `Disconnect()` (per-peer)                                  | `PeerRemovedUpdate` × N + single 200ms generation-gate wait + bulk `m_remoteuser_deferred_delete_q.try_push` |
| `~NJClient` (destructor)                                   | direct delete (audio thread already stopped per JUCE shutdown ordering); drains pending deferred-deletes first |

All publishes happen AFTER the canonical `m_users_cs.Enter()/Leave()` block. The canonical mutex is preserved for run/UI/network-thread consistency — only the audio-thread side is unlocked. On overflow, `m_remoteuser_update_overflows` is bumped (M-8 style observable).

### Audio thread (`AudioProc` top of function)

```cpp
drainLocalChannelUpdates();   // 15.1-06
drainRemoteUserUpdates();     // 15.1-07a NEW
m_audio_drain_generation.fetch_add(1, std::memory_order_release);  // single counter covers BOTH
```

Drain applies queued variants to `m_remoteuser_mirror[u.slot]` via `std::visit`; release-store on the generation counter signals to the run thread that the audio thread has observed everything published before the matching `acquire`-load on the run side.

### `process_samples` rewrite

```cpp
for (int s = 0; s < MAX_PEERS; ++s) {
  auto& um = m_remoteuser_mirror[s];
  if (!um.active) continue;
  int a = um.chanpresentmask;
  for (int ch = 0; ch < MAX_USER_CHANNELS && a; ++ch) {
    if (a & 1) {
      // … pan/mute/solo computation reads um.* and um.chans[ch].*
      mixInChannel(s, ch, muteflag, …);
    }
    a >>= 1;
  }
}
```

No `m_users_cs.Enter` anywhere. Identity is the slot index `s`.

### `mixInChannel(int slot, int chanidx, …)` REWRITE

- Signature changed: `RemoteUser*` → `int slot`.
- All field reads through `auto& chan_mirror = m_remoteuser_mirror[slot].chans[chanidx]`.
- DecodeState* pointer-shuffle (chan.ds = chan.next_ds[0]) operates ENTIRELY on the mirror; old pointers cross to the run thread via `deferDecodeStateDelete` (15.1-05 helper).
- VU peak read/write via `chan_mirror.peak_vol_l/r` (atomic relaxed).
- Sessionmode (flags & 4) is a no-op (defer-delete in-flight ds, return). Documented above.
- Insta-measurement identity is `static_cast<uintptr_t>(slot)` (was `reinterpret_cast<uintptr_t>(user)`).
- Tail recursion: `mixInChannel(slot, chanidx, …)` (was `mixInChannel(user, chanidx, …)`).

### `on_new_interval` remote-user advance REWRITE

- Iterates `for (int s = 0; s < MAX_PEERS; ++s)` over the mirror.
- next_ds advance pointer-shuffle on `chan_mirror.next_ds[0/1]` with capture-first-then-defer-delete ordering.
- Insta-measurement INSTA_CAPTURED → MEASURED uses slot identity (matches the IDLE → CAPTURED store in mixInChannel).
- No `m_users_cs.Enter`.

### Codex HIGH-3 deferred-free protocol

```
Run thread (per-peer remove):
  1. Under m_users_cs: detach RemoteUser from m_remoteusers (no delete).
  2. publish_gen_target = m_audio_drain_generation.load(acquire) + 1
  3. m_remoteuser_update_q.try_push(PeerRemovedUpdate{slot})
  4. yield-spin until m_audio_drain_generation.load(acquire) >= publish_gen_target
     (max 200ms; on timeout → leak, RT-safety > memory hygiene)
  5. m_remoteuser_deferred_delete_q.try_push(victim)
  6. releaseRemoteUserSlot(slot)

Audio thread (AudioProc):
  drain → set mirror[slot].active=false + defer-delete in-flight ds → release-store on m_audio_drain_generation

Run thread (NinjamRunThread::run, every 20ms):
  client->drainRemoteUserDeferredDelete()  // pops victim, runs ~RemoteUser()
```

For `Disconnect()` (bulk peer removal), the same protocol with a single 200ms wait covers all peers.

**1. [Rule 3 — Blocking] DecodeState type bridge (jamwide vs global namespace)**
- **Found during:** Task 1 build verification (first compile attempt failed)
- **Issue:** The FINAL `spsc_payloads.h` forward-declares `jamwide::DecodeState` in `namespace jamwide` (the variant payload type carried by `PeerNextDsUpdate.ds`), but the production class definition lives in the global namespace (`::DecodeState` at `njclient.cpp:260`). The mirror holds `::DecodeState*` because the audio thread invokes member functions on the canonical type. Direct assignment `chan.next_ds[u.slot_idx] = u.ds` failed with "incompatible pointer types". The same mismatch applied to `deferDecodeStateDelete` calls.
- **Fix:** Bridge via `reinterpret_cast<::DecodeState*>(u.ds)` in the apply visitor (PeerNextDsUpdate branch) AND in the run-thread publishers when constructing PeerNextDsUpdate. Production has no `jamwide::DecodeState` definition (it's an opaque ownership-transfer tag in `spsc_payloads.h` per the header comment), so the cast is a documented no-op type-bridge. `spsc_payloads.h` is UNTOUCHED (Codex M-9 preserved).
- **Files modified:** `src/core/njclient.cpp` (5 sites: drainRemoteUserUpdates' PeerNextDsUpdate branch + DOWNLOAD_INTERVAL_BEGIN publish + RemoteDownload::startPlaying publish).
- **Verification:** Builds green, all tests pass, TSan reports zero races.
- **Committed in:** `582b906`.

**2. [Rule 3 — Blocking] RemoteUser_Channel.run_thread_next_ds_idx shadow field**
- **Found during:** Task 4 (run-thread publish wiring for DOWNLOAD_INTERVAL_BEGIN)
- **Issue:** The pre-07a code chose the next_ds slot via `useidx = !!theuser->channels[chidx].next_ds[0]` — a direct read of the canonical `next_ds[0]` pointer. Post-07a, the audio-thread mirror owns those pointers and shuffles them independently in mixInChannel/on_new_interval. The canonical `next_ds[0]` is no longer the source of truth; reading it from the run thread would race with the mirror's shuffle.
- **Fix:** Added a single int field `run_thread_next_ds_idx` to `RemoteUser_Channel` (private class definition local to `njclient.cpp`). The run thread alternates 0/1 per publish, eliminating the cross-thread read. The field is initialized to 0 and toggled with `^= 1` after each publish.
- **Files modified:** `src/core/njclient.cpp` (RemoteUser_Channel struct + 2 use sites in DOWNLOAD_INTERVAL_BEGIN handler + 1 use site in RemoteDownload::startPlaying).
- **Verification:** Builds green; alternation pattern is deterministic.
- **Committed in:** `582b906`.

**3. [Rule 2 — Missing critical] Disconnect path generation-gate dance NOT in plan**
- **Found during:** Task 4 review of m_remoteusers mutation sites
- **Issue:** The plan's enumeration listed the auth handler peer-remove path but not the two `Disconnect()` paths (`NJClient::Disconnect` at line ~1050 and `~NJClient` at line ~860). The 15.1-MIRROR-AUDIT.md explicitly listed lines 865 and 1054 as `m_remoteusers.Empty()` mutation sites that "wipe all" peers. These are run-thread paths that would race with the audio thread if the canonical `delete RemoteUser*` ran while the audio thread still held mirror state. Without coverage, peer disconnect would have a UAF window.
- **Fix:** Added the publish-wait-defer dance to `Disconnect()` (bulk: per-peer PeerRemovedUpdate, single 200ms generation-gate wait, bulk enqueue onto deferred-delete queue). For `~NJClient`, audio thread is already stopped (per JUCE shutdown ordering: AudioProcessor::releaseResources runs before destruction), so direct-delete is safe — but we still drain pending deferred-deletes first to mop up any prior-Disconnect leftovers.
- **Files modified:** `src/core/njclient.cpp` (Disconnect, ~NJClient).
- **Verification:** Builds green; the deferred-delete drain in NinjamRunThread::run handles the Disconnect-side enqueues on the next 20ms tick.
- **Committed in:** `582b906`.

**4. [Plan-faithful expansion] Defer-delete on PeerChannelMaskUpdate present→not-present transition**
- **Found during:** Task 4 review of run-thread publishing for SetUserChannelState (subscription toggle)
- **Issue:** When SetUserChannelState toggles `sub=false`, the canonical code used to `delete tmp; delete tmp2; delete tmp3` (the channel's ds + next_ds[2]). Pre-07a these were on the run thread and synchronous-safe. Post-07a, those pointers are audio-thread-owned via past PeerNextDsUpdate handovers — direct delete would UAF. The mirror's apply visitor needed to handle the transition and defer-delete the in-flight ds when a present bit drops to 0.
- **Fix:** Updated `PeerChannelMaskUpdate` apply visitor to compare `prev_present` vs `m.chanpresentmask` per channel; for any channel transitioning present→not-present, defer-delete `m.chans[ch].ds`, `m.chans[ch].next_ds[0]`, `m.chans[ch].next_ds[1]` via `deferDecodeStateDelete`. The run-thread side just nulls the canonical pointers (no delete).
- **Files modified:** `src/core/njclient.cpp` (drainRemoteUserUpdates' PeerChannelMaskUpdate branch + SetUserChannelState run-thread side).
- **Verification:** Builds green, TSan tests pass.
- **Committed in:** `582b906`.
