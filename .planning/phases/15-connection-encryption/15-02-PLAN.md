REDESIGNED NEGOTIATION FLOW (addresses review HIGH concern #1):
```
Step 1: Server -> Client: AUTH_CHALLENGE
  server_caps bit 1 SET = "I support encryption"
  challenge[8] = random salt

Step 2: Client processes AUTH_CHALLENGE:
  IF server_caps bit 1 is SET AND password is non-empty:
    derive key = SHA-256(password + challenge)
    set m_netcon->SetEncryptionKey(key)
    build AUTH_USER message normally
    encryption hook in Run() will encrypt the payload automatically
    set client_caps bit 2 = "I encrypted this AUTH_USER"
  ELSE (legacy server OR no password):
    build AUTH_USER normally, no encryption
    do NOT set client_caps bit 2

Step 3: Client -> Server: AUTH_USER (encrypted if capable)

Step 4: Server -> Client: AUTH_REPLY
  IF client sent encrypted AUTH_USER (client_caps bit 2):
    Server decrypted it (server derived same key from same password+challenge)
    flag bit 1 SET = "encryption confirmed for session"
    AUTH_REPLY itself is encrypted (server already has key active)
  ELSE:
    flag bit 0 only = success, no encryption
    Session proceeds unencrypted

Step 5: All subsequent messages encrypted (both directions)
```

WHY this works for backward compatibility:
- Legacy server: does NOT set server_caps bit 1 -> client sends unencrypted AUTH_USER -> session unencrypted (SEC-03)
- Legacy client: ignores server_caps bit 1 -> sends unencrypted AUTH_USER without client_caps bit 2 -> server sees no encryption bit -> session unencrypted
- JamWide client + JamWide server: both negotiate -> encrypted from AUTH_USER onward (SEC-01)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> server (TCP) | Encrypted NINJAM messages in transit; attacker on network sees encrypted bytes after negotiation |
| server AUTH_CHALLENGE -> client | server_caps bit signals encryption support; attacker cannot forge (TCP stream integrity) |
| client AUTH_USER -> server | First encrypted message: contains passhash credential (SEC-01) |
| server AUTH_REPLY -> client | Confirms encryption active; encrypted itself since server has key |
| encrypted payload -> Net_Connection::Run() | Received encrypted bytes must be valid [IV][ciphertext] or connection drops |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-15-06 | Spoofing | mpb.h capability bits | mitigate | Three-phase negotiation: server advertises in AUTH_CHALLENGE (server_caps bit 1), client signals in AUTH_USER (client_caps bit 2), server confirms in AUTH_REPLY (flag bit 1). All three must agree for encryption. |
| T-15-07 | Tampering | netmsg.cpp decrypt path | mitigate | On decryption failure, set m_error=-5 (generic) and disconnect. Do NOT reveal whether failure was padding error, block alignment error, or other (prevents padding oracle). result.data always empty on failure. |
| T-15-08 | Information Disclosure | netmsg.h headers | accept | Per D-07: message type (1 byte) and size (4 bytes) remain cleartext. Observer sees message types and sizes but not contents. This is NOT "end-to-end" confidentiality for metadata — only payload confidentiality. Accepted trade-off for framing compatibility. |
| T-15-09 | Denial of Service | NET_MESSAGE_MAX_SIZE | mitigate | Accept messages up to NET_MESSAGE_MAX_SIZE_ENCRYPTED (16416) to prevent rejection of legitimate encrypted messages. NJ_CRYPTO_MAX_PLAINTEXT (16384) in crypto module prevents oversized encryption attempts. |
| T-15-10 | Information Disclosure | njclient.cpp key material | mitigate | enc_key zeroed on stack after SetEncryptionKey(). ClearEncryption() scrubs m_encryption_key with memset(0). m_auth_challenge zeroed on disconnect. m_pass cleared by Disconnect(). |
| T-15-11 | Elevation of Privilege | njclient.cpp auth timing | mitigate | Encryption key derived BEFORE sending AUTH_USER (when server advertises support). AUTH_USER is the FIRST encrypted message. This satisfies SEC-01. |
| T-15-12 | Tampering | njclient.cpp downgrade | mitigate | If client encrypted AUTH_USER but server reply lacks encryption confirmation, client explicitly clears encryption (ClearEncryption). No silent mismatch state. Active MITM cannot force encryption-capable peers to skip encryption because: (a) server_caps bit removal would require modifying the TCP stream (TCP provides ordering/integrity), (b) removing client_caps bit would cause server to not decrypt AUTH_USER, resulting in auth failure, not silent downgrade. Note: a MITM who can modify TCP bytes can perform more powerful attacks; this is the accepted limitation of application-layer encryption without TLS. |
| T-15-13 | Information Disclosure | keepalive passthrough | accept | Zero-length keepalive payloads are NOT encrypted. Keepalive type (0xFD) and size (0) are already in cleartext headers per D-07. Encrypting empty payload adds 32 bytes overhead every 3 seconds with zero confidentiality benefit. |
