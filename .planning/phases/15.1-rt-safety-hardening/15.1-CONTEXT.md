# Phase 15.1: RT-Safety Hardening - Context

**Gathered:** 2026-04-25
**Status:** Ready for planning

<domain>
## Phase Boundary

Eliminate real-time-safety violations on the JUCE audio callback path of the NINJAM client. Specifically: remove mutex acquisitions, heap deallocations, and file I/O that currently run inside `NJClient::AudioProc` and its callees (`on_new_interval`, `process_samples`, `mixInChannel`).

**In scope:**
- Audit the entire audio-thread reachable code: `processBlock` → `AudioProc` → `on_new_interval` / `process_samples` / `mixInChannel` and downstream.
- Replace `m_users_cs` / `m_locchan_cs` mutex acquisitions with SPSC-queue-mediated state updates.
- Move `delete chan->ds` and similar deallocations off the audio thread.
- Eliminate `writeLog` / `writeUserChanLog` / the `JAMWIDE_DEV_BUILD` `fopen` block from the audio path.
- Verify the `mixInChannel` per-peer loop is allocation-free and lock-free.
- Add ThreadSanitizer runs to CI for the audio path.

**Out of scope:**
- Network protocol changes (no on-the-wire format edits).
- Codec algorithm changes (no Vorbis/FLAC encoding path edits).
- UI layout / mixer behavior changes.
- The Phase 14.2 Instatalk continuous-encoding fix (already shipped in commits `9cd23c0` + `9fa0d32`).
- Runtime performance tuning beyond removing RT-safety violations.

**Driving evidence:** The bug investigation in the discussion immediately preceding this phase identified concrete file:line violations in `src/core/njclient.cpp:2690` (`on_new_interval`) and `src/core/njclient.cpp:1955` (`process_samples`). User reported that the visible CPU spikes from those violations are no longer symptomatic in build 227 (which includes Instatalk fixes), but the underlying RT-safety violations remain latent and could surface again under different load patterns. This phase is proactive hardening, not reactive bugfix.

</domain>

<decisions>
## Implementation Decisions

### Architecture

- **D-01:** Replace `m_users_cs` and `m_locchan_cs` audio-thread reads with **SPSC queue-mediated state updates**. The run thread pushes state changes (peer added/removed, channel mutated, codec swapped, `next_ds` chain advanced) onto a per-target SPSC queue; the audio thread drains the queue and applies updates locally. Reuses the existing `src/threading/spsc_ring.h` primitive — no new lock-free infrastructure to design.

- **D-02:** Deferred-free strategy follows the same SPSC pattern: when the audio thread would have called `delete chan->ds`, it instead pushes the pointer onto a "deferred-delete" SPSC queue drained by the run thread. The run thread does the actual `delete` off the audio thread. This pattern applies to all heap deallocations identified by the auditor.

- **D-03:** Audio-thread logging is **eliminated** in production paths and **routed through SPSC queues** in development paths. Specifically: `writeLog` / `writeUserChanLog` calls inside `on_new_interval` and `process_samples` are removed entirely (they were "interval N happened" diagnostic noise — not actionable). The `JAMWIDE_DEV_BUILD` `fopen("/tmp/jamwide.log")` block at `njclient.cpp:2133` is deleted unconditionally — it's a one-shot debug log that's been there since dev bringup and has zero ongoing value.

- **D-04:** `mixInChannel` is kept structurally as the per-peer per-block mixing routine (it has to do that work — that's its job). The auditor agent will verify that `mixInChannel` and its callees are themselves allocation-free and lock-free; any violations found become additional sub-plans.

### Migration

- **D-05:** Sequence the work as **incremental sub-plans**, each landing as its own atomic commit. Tentative breakdown (auditor findings may shift this):
  - `15.1-01` — Auditor pass: run the `realtime-audio-reviewer` agent over the audio path, file findings as `15.1-AUDIT.md`.
  - `15.1-02` — SPSC queue infrastructure for cross-thread state updates (extend or wrap `spsc_ring.h` as needed).
  - `15.1-03` — Replace `m_users_cs` audio-thread reads with snapshot consumption.
  - `15.1-04` — Replace `m_locchan_cs` audio-thread reads with snapshot consumption.
  - `15.1-05` — Deferred-delete queue for `chan->ds` and any other heap frees the auditor flags.
  - `15.1-06` — Eliminate / route logging out of audio path.
  - `15.1-07` — Verification: TSan run, auditor re-run, UAT.

- **D-06:** Each sub-plan ships individually and is merged before the next starts. No "stack of WIP commits" — one violation, one PR-equivalent commit, one verification. This matches the project's existing GSD discipline and lets us bisect if a regression shows up.

### Verification

- **D-07:** Three-signal verification per sub-plan:
  1. **ThreadSanitizer** in CI on the test build (`./scripts/build.sh --tests`). Catches data races between audio thread and run thread.
  2. **`realtime-audio-reviewer` agent** re-run after each fix sub-plan. Flags any regression or newly-introduced violations.
  3. **Manual UAT** on a populated NINJAM server (3+ peers) — same protocol the user just performed: connect, listen for audible glitches, watch Activity Monitor for spike pattern.

- **D-08:** TSan-passes-and-listening-test-passes is the bar for closing each sub-plan. Profiling (Instruments) is *available* for diagnosis but not required for sign-off.

### Process

- **D-09:** **Auditor-first.** The `realtime-audio-reviewer` agent runs as `15.1-01` *before* any architecture sub-plans are written. The auditor's findings (`15.1-AUDIT.md`) are an input to `gsd-phase-researcher`, which produces `15.1-RESEARCH.md`. The planner then writes sub-plans 15.1-02 through 15.1-07 against that empirical map of violations — not against my (Claude's) bug-investigation hypothesis from the preceding discussion.

- **D-10:** The audit deliverable is structured per the agent's output spec (severity-ranked, file:line, suggested fix). It supersedes the informal findings in the preceding discussion — those are the trigger, not the plan-of-record.

### Claude's Discretion

The following gray areas were not explicitly chosen by the user (auto-discussed per `--all`); the planner has flexibility within the locked architecture:

- **Deferred-delete granularity** — whether one shared deferred-free queue or per-data-type queues. Planner decides based on auditor's enumeration.
- **Audio-path logging — keep any?** — D-03 says eliminate diagnostic logs entirely. If the auditor surfaces a *useful* logging need (e.g., "first decoder error per peer"), planner may add a single SPSC-queued log path; otherwise no logging from audio thread at all.
- **`mixInChannel` optimization** — beyond RT-safety, no perf tuning is in scope. If the auditor flags allocations or locks inside the loop, fix them; otherwise leave the loop alone.
- **NJClient internal API changes** — NJClient was vendored from WDL but the project owns the fork. Internal struct layout changes (e.g., adding queue members to `RemoteUser`) are fine. The C-style callback API surface (`AudioProc`, `Run`, `Set/GetUser*`) should be kept stable so the JUCE wrapper and tests don't need to be rewritten.
- **Test coverage detail** — D-07 mandates TSan + auditor + UAT. Planner decides whether to add unit tests for the new SPSC primitives (likely yes, small surface area) and whether to write an integration test that simulates interval boundaries + peer churn (likely yes for `15.1-07`).

## Existing Code Insights

### Reusable Assets

- **`src/threading/spsc_ring.h`** — single-producer-single-consumer ring buffer, already in the codebase. Direct primitive for D-01, D-02, D-03. The planner should read this header before designing the new state-sync queues — its semantics dictate what SPSC patterns are easy and what's awkward.
- **`realtime-audio-reviewer` subagent** at `.claude/agents/realtime-audio-reviewer.md` — runs as the 15.1-01 audit. Built earlier this session specifically for this kind of work.
- **JUCE primitives available**: `juce::AbstractFifo` (alternative SPSC), `juce::SpinLock` (rejected per D-01 but available), `juce::Atomic<T>` (for primitive-typed snapshot fields).

### Established Patterns

- **NJClient is vendored from WDL but the project owns the fork.** Internal struct edits are fine; C-style public API should remain stable (`AudioProc`, `Run`, `SetLogFile`, etc.) so the JUCE wrapper compiles unchanged.
- **`m_users_cs` / `m_locchan_cs` are `WDL_Mutex`** — wrapper around an OS mutex. Currently taken on both the run thread and the audio thread. The replacement removes audio-thread acquisition; the run thread retains a normal mutex for its own consistency (run thread is not RT).
- **The run thread already polls at 20ms cadence** (per NinjamRunThread comments) — it can drain SPSC queues from the audio thread on the same tick without adding a new thread.

### Integration Points

- **Audio thread entry**: `juce/JamWideJuceProcessor.cpp:494` (`client->AudioProc(...)`) — unchanged.
- **Run thread driver**: `juce/NinjamRunThread.cpp:run()` (line 626) — gets new responsibilities (SPSC queue drain, deferred-delete drain).
- **Test build target**: `./scripts/build.sh --tests` configures `JAMWIDE_BUILD_TESTS=ON -DJAMWIDE_BUILD_JUCE=OFF`. TSan can be enabled here without rebuilding the JUCE plugin.

|---|---|---|
| `2698`, `2723` | `m_locchan_cs.Enter()` / `m_users_cs.Enter()` — WDL_Mutex on audio thread | CRITICAL |
| `2693` | `writeLog(...)` — calls `vfprintf` to FILE* with internal mutex | CRITICAL |
| `2743` | `delete chan->ds` — heap deallocation per peer × per channel | CRITICAL |
| `2746` | `delete chan->next_ds[0]` — second conditional heap dealloc | CRITICAL |
| `2753` | `writeUserChanLog(...)` — formatted string + log mutex | HIGH |

### Reproducer / non-reproducer status

- The visible *audible* CPU spikes the user originally reported are **not currently reproducible** in build 227 — the recent Instatalk fixes (`9cd23c0` / `9fa0d32`) addressed the symptomatic case.
- The latent RT-safety violations above are still present in the code regardless. This phase fixes the *category* of issue, not the specific past symptom.
- Hypothesis-testing path for any future regression: connect to a populated server (3+ peers) and watch Activity Monitor's CPU history graph; spike-period = `(BPI / BPM) * 60` seconds confirms an `on_new_interval` regression.
