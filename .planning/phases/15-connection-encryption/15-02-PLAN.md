<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> server (TCP) | Encrypted NINJAM messages in transit; attacker on network can observe encrypted bytes |
| server auth reply -> client | Server's encryption-active bit triggers key derivation and encryption activation |
| encrypted payload -> Net_Connection::Run() | Received encrypted bytes must be valid [IV][ciphertext] or connection drops |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-15-06 | Spoofing | mpb.h capability bits | mitigate | Only activate encryption when BOTH client advertises support AND server confirms; legacy clients/servers silently skip |
| T-15-07 | Tampering | netmsg.cpp decrypt path | mitigate | On decryption failure (EVP_DecryptFinal_ex error), set m_error=-5 and disconnect cleanly; do NOT reveal padding error details (prevents padding oracle) |
| T-15-08 | Information Disclosure | netmsg.h headers | accept | Per D-07: message type (1 byte) and size (4 bytes) remain cleartext; observer can see message types and sizes but not contents. Accepted design trade-off for framing compatibility. |
| T-15-09 | Denial of Service | NET_MESSAGE_MAX_SIZE | mitigate | Increase max size to NET_MESSAGE_MAX_SIZE_ENCRYPTED (16416) to prevent rejection of legitimate encrypted messages that exceed 16384 due to IV+padding overhead |
| T-15-10 | Information Disclosure | njclient.cpp key material | mitigate | Zero-fill key on stack after SetEncryptionKey(); ClearEncryption() scrubs m_encryption_key with memset(0); m_auth_challenge zeroed on disconnect |
| T-15-11 | Elevation of Privilege | njclient.cpp auth timing | mitigate | Encryption activated ONLY after MESSAGE_SERVER_AUTH_REPLY with success+encrypt bits; first encrypted message is MESSAGE_CLIENT_SET_CHANNEL_INFO (post-auth); auth messages themselves are never encrypted |
</threat_model>
