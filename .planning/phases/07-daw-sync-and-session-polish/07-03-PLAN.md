**B. Create `.planning/references/MCP-ASSESSMENT.md`:**

Per D-19: Half-page, quick feasibility verdict.
Use consistent template per Codex suggestion.
Separate transport sync vs session control vs workflow tooling per Codex LOW concern.

**Document structure:**
1. **Goal** -- Assess MCP (Model Context Protocol) for JamWide DAW bridge and session control
2. **Constraints** -- Must provide real-time or near-real-time communication; must work with existing NINJAM protocol; Anthropic's MCP is the reference implementation
3. **Findings**
   - **What MCP Is** -- Anthropic's protocol for giving AI models access to tools and data sources. JSON-RPC based, designed for request/response patterns with language models.
   - **Use Case Separation** (addresses Codex LOW review concern -- clear separation):

     | Use Case | MCP Fit | Rationale |
     |----------|---------|-----------|
     | Transport sync (play/stop/tempo) | Poor | Request/response model adds 50-500ms latency per round-trip. DAW transport requires sub-millisecond timing. No streaming support for continuous position updates. |
     | Session control (connect, vote, settings) | Fair | Non-real-time operations fit request/response model. But no DAW implements MCP natively -- would need a custom bridge server. |
     | Workflow tooling (AI-assisted server selection, chat moderation, preset management) | Good | Natural fit for AI tool integration. Non-latency-sensitive. Could enable "AI DJ" or "AI session manager" features. |

   - **Why It Doesn't Fit DAW Sync** -- MCP is request/response, not streaming. DAW transport sync requires sub-millisecond timing. MCP adds AI inference latency. No DAW implements MCP natively. The "bridge" concept (MCP server that controls a DAW) is theoretically possible but adds complexity without benefit over OSC/MIDI.
   - **Where MCP Could Help JamWide** -- Session management (AI-assisted server selection), chat (AI-powered translation, moderation), preset management. These are non-real-time tasks where MCP's strengths apply.
4. **Recommendation** -- Not feasible for DAW sync (XSYNC-03). Potentially valuable for non-real-time AI-assisted features in v3+. If pursued, implement as a separate MCP server process that exposes JamWide session state as MCP resources and accepts session control commands as MCP tools.
5. **Open Questions** -- Whether MCP's tool-calling model could wrap OSC commands for AI-mediated DAW control; whether session state exposure via MCP resources has privacy implications
6. **Requirements Addressed** -- XSYNC-03 (from REQUIREMENTS.md v2)

Add date/version context to both documents: "Analysis as of April 2026".
