# Cross-AI Plan Review — Phase 14.2

## Codex Review

### PLAN 14.2-01 Review

#### Summary
This plan is close to the phase goal on the plugin side, especially the visible `Instatalk` channel, PTT routing, measured-delay broadcast, and fallback path. The main weakness is the measurement design itself: the proposed hooks and state ownership are still ambiguous enough that the implementation could produce the wrong delay or a stale one while still appearing to "work." The highest-risk area is proving that `t_insta` and `t_interval` refer to the same remote user and the actual playback moment, not just interval bookkeeping.

#### Strengths
- Uses the existing `flags=0x02` instamode path, which matches the locked decisions and avoids protocol changes.
- Reuses the existing `bufferDelay` message instead of introducing a second sync protocol.
- Preserves backward fallback to BPM/BPI calculation, which is necessary for mixed JamWide/non-JamWide sessions.
- Recognizes real pitfalls already present in this codebase: channel index collision, callback lifetime, disconnect cleanup, and audio-thread allocation.
- Keeps the UI-facing sync decision simple by pushing a resolved measured delay to `VideoCompanion`.

#### Concerns
- **HIGH**: The plan says to capture `t_interval` in `on_new_interval`, but the research section already says `on_new_interval` is too early and the correct point is the `ds` assignment/playback transition. That is a direct contradiction and can bias the measured delay.
- **HIGH**: Measurement state is split across both `JamWideJuceProcessor` atomics and new `NJClient` atomics. The plan does not define a single owner or exact synchronization flow, so reset/broadcast races are likely.
- **HIGH**: "Same remote user" is required, but the proposed state uses `measured_user` without defining a stable identity. User indices can shift as roster state changes; that can pair `t_insta` from one user with `t_interval` from another.
- **MEDIUM**: The one-shot "write only if current value is 0" pattern is fragile. A partial write, stale reconnect state, or missed reset can permanently suppress future measurement.
- **MEDIUM**: Channel 4 setup is reasonable, but the plan does not say how that extra local channel interacts with existing local-channel assumptions, UI persistence, or any code that currently assumes only channels `0-3` are user-facing.
- **MEDIUM**: `SetLocalChannelProcessor` is a C callback API, not a capturing lambda API. The plan mentions a "PTT lambda" but does not define the static callback + stable context object needed to make that safe.
- **MEDIUM**: "Sends silence when PTT inactive" does not necessarily deliver the claimed bandwidth savings. Continuous instamode transmission may still consume meaningful bandwidth depending on codec behavior.
- **MEDIUM**: "Broadcast once per session" may be too strict. If video is activated after connect, or if the first measurement is invalid, the session may never recover to measured mode.

#### Suggestions
- Use one measurement owner only, preferably inside `NJClient`, and let the run thread consume a single coherent snapshot from it.
- Replace user-index matching with a stable key for the measurement pair, such as remote username plus channel index, or another identifier guaranteed stable across roster updates.
- Move `t_interval` capture to the actual decode/playback promotion point for the regular interval channel, not `on_new_interval`.
- Define an explicit measurement state machine: `empty -> t_insta captured -> matched interval captured -> consumed -> reset`.
- Specify the PTT callback implementation as a static function with a processor-owned context object whose lifetime spans connect/disconnect.
- Add an explicit rule for when re-measurement is allowed: on reconnect, on video activation, and when the first measurement is invalid or negative.
- Call out whether channel 4 is intentionally hidden from local-channel UI/state persistence, or if it is a managed system channel.

#### Risk Assessment
**HIGH**. The plan is directionally correct, but the timing hook and identity-matching details are still weak enough to miss the core success criterion while looking superficially complete.

### PLAN 14.2-02 Review

#### Summary
This plan is solid for the companion's mode display and delay-priority logic, and TDD is the right approach here. The main gap is that the overlay design does not match the stated requirement: the roadmap calls for per-tile black/fade-in behavior, while the plan implements a single global overlay in `#main-area`. The rest of the risk is around preserving the distinction between manual override and the last auto mode.

#### Strengths
- Correctly keeps manual override on the companion side, which matches the existing architecture.
- Extends the existing `bufferDelay` message instead of inventing a second sync event type.
- Updates the display model from `auto/manual` to `measured/calculated/manual`, which is necessary to meet the success criteria.
- Includes tests first, which is useful because the current companion delay model is already stateful and easy to regress.
- Adds cached sync mode for new WebSocket clients, which aligns with how delay caching already works.

#### Concerns
- **HIGH**: The overlay implementation does not satisfy D-09/D-10 as written. A single `#sync-overlay` in `#main-area` is not a per-user/per-tile black fade-in overlay.
- **MEDIUM**: A single `currentSyncMode` is not enough if manual mode must override while still preserving whether the last auto source was `measured` or `calculated`. Without a separate "last auto mode," switching back to auto can display the wrong mode.
- **MEDIUM**: Runtime validation is not updated enough. `BufferDelayMessage` currently only validates `delayMs`; if `syncMode` is added, invalid strings should be rejected or normalized.
- **MEDIUM**: The tests listed are missing some important transitions: measured -> manual, calculated -> manual, manual -> auto restore, reconnect/new client replay, and fallback when `syncMode` is omitted.
- **MEDIUM**: `showSyncingOverlay(delayMs)` appears timeout-based, but the plan does not define what happens if a second measured message arrives before the first fade completes.
- **LOW**: The human verification checklist is too narrow. It should explicitly include the "no Instatalk peer available" fallback path and a reconnect/new-tab replay check.

#### Suggestions
- Rework the overlay task so it matches the actual requirement: per-tile state, not one global overlay.
- Track `lastAutoDelayMs` and `lastAutoSyncMode` separately from `manualMode`, then derive the active display from those plus manual override.
- Extend runtime validation so `syncMode` is constrained to known values and older messages still default cleanly to `calculated`.
- Add tests for transition semantics, not just static rendering.
- Define overlay timer behavior for repeated measured updates: replace, restart, or ignore.
- Expand the human verification step to include:
  - a mixed session with no JamWide peer
  - reconnect/new companion tab
  - manual override followed by return to auto
  - measured mode arriving after initially calculated mode

#### Risk Assessment
**MEDIUM**. The state-model direction is good, but the overlay requirement is currently under-specified and partially mismatched to the roadmap.

## Consensus Summary

*Single reviewer (Codex) — consensus analysis based on cross-plan patterns.*

### Agreed Strengths
- Reuses existing infrastructure (instamode protocol, bufferDelay messages, existing delay state model) without protocol changes
- Correct fallback architecture: measured > calculated, with manual always winning
- TDD approach for companion side is appropriate for the stateful delay model
- Threat models are reasonable and cover the actual attack surface

### Key Concerns (by severity)

**HIGH:**
1. **Measurement timing ambiguity** — Plan 01 contradicts its own research on where to capture `t_interval` (on_new_interval vs ds assignment). This directly affects measurement accuracy.
2. **Split measurement ownership** — State across NJClient and Processor atomics without clear single owner creates reset/race risk.
3. **User identity stability** — User indices can shift during roster changes, potentially pairing timestamps from different users.
4. **Overlay/D-09 mismatch** — Plan 02 implements a single global overlay, but D-09 specifies per-tile black + fade-in behavior.

**MEDIUM:**
5. One-shot broadcast pattern may be too strict (video activated after connect, invalid first measurement)
6. Channel 4 interaction with existing local-channel UI/persistence assumptions
7. Lambda vs C callback API mismatch for SetLocalChannelProcessor
8. Missing transition tests in companion test suite

### Divergent Views
N/A — single reviewer.

### Recommended Actions Before Execution
1. **Resolve t_interval hook location** — Commit to ds assignment moment (not on_new_interval) and update Plan 01
2. **Consolidate measurement ownership** — All measurement atomics in NJClient only, processor just has broadcast guard + pttActive
3. **Define measurement state machine** — Explicit states prevent one-shot fragility
4. **Clarify overlay scope** — Either simplify D-09 to global overlay (and update CONTEXT.md) or implement per-tile overlays
5. **Add re-measurement rules** — Allow re-measurement on reconnect and video activation
