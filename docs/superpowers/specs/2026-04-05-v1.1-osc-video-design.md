# v1.1 Design: OSC Remote Control + VDO.Ninja Video

**Date:** 2026-04-05
**Milestone:** v1.1
**Status:** Draft — open research questions flagged

## Summary

JamWide v1.1 adds two independent features:

1. **OSC Server** — full bidirectional OSC control of all plugin parameters, enabling remote control via TouchOSC (iPad/phone) or any OSC client
2. **VDO.Ninja Video** — browser-based video companion synced to the NINJAM session's interval timing, with grid and per-user popout display modes

Both features share the JUCE plugin core but are architecturally independent — either can ship without the other.

## Feature 1: OSC Server

### Architecture

- Built on JUCE's `juce_osc` module (no external dependencies)
- Follows the IEM Plugin Suite pattern: `OSCParameterInterface` with bidirectional UDP
- Receiver: realtime callback thread for incoming messages (low latency)
- Sender: timer-based (configurable interval, default 100ms) for outgoing state changes — only sends changed values

### OSC Namespace

Hierarchical addressing with index-based remote user slots.

```
# Connection & Session
/JamWide/connect              0 | 1        (trigger connect/disconnect)
/JamWide/status               int          (NJC status code, read-only)
/JamWide/session/bpm          float        (read-only)
/JamWide/session/bpi          int          (read-only)
/JamWide/session/beat         int          (current beat, read-only)
/JamWide/session/users        int          (user count, read-only)

# Local Channels
/JamWide/local/volume         0.0–1.0
/JamWide/local/pan           -1.0–1.0
/JamWide/local/mute           0 | 1
/JamWide/local/codec          0 | 1        (0=FLAC, 1=Vorbis)
/JamWide/local/tx             0 | 1        (transmit on/off)

# Remote Users (by stable index, 1-based)
/JamWide/remote/count         int          (total remote users, read-only)
/JamWide/remote/{idx}/name    string       (read-only — broadcast on roster change)
/JamWide/remote/{idx}/volume  0.0–1.0
/JamWide/remote/{idx}/pan    -1.0–1.0
/JamWide/remote/{idx}/mute    0 | 1
/JamWide/remote/{idx}/solo    0 | 1
/JamWide/remote/{idx}/codec   string       (read-only — "FLAC" or "Vorbis")

# Metronome
/JamWide/metro/volume         0.0–1.0
/JamWide/metro/pan           -1.0–1.0
/JamWide/metro/mute           0 | 1

# Video Control (via OSC)
/JamWide/video/open           trigger      (open companion page in browser)
/JamWide/video/close          trigger      (close video)
/JamWide/video/room           string       (get/set VDO.Ninja room ID)
/JamWide/video/mode           string       (grid | popout)
/JamWide/video/popout/{idx}   trigger      (pop out user {idx} to own window)
```

### User Index Stability

- Remote users are assigned stable indices (1-based) in join order
- When a user disconnects, their index is freed and reused by the next joiner
- On any roster change, the plugin broadcasts `/JamWide/remote/count` and `/JamWide/remote/{idx}/name` for all active indices
- TouchOSC layouts bind faders to fixed indices (1-8) and update labels from `/name` messages
- Username-based addressing (`/remote/Dave/volume`) is deferred to a future version

### Configuration

- OSC receiver port: configurable in plugin UI (default: 9000, range: 1000-15000)
- OSC sender: configurable target IP + port (for TouchOSC feedback)
- Send interval: configurable (1-1000ms, default: 100ms)
- Config persisted in plugin state (saved/restored with DAW session)
- UI: footer status indicator (IEM-style) with config dialog on click

### Reference Implementation

IEM Plugin Suite (`/Users/cell/dev/IEMPluginSuite`):
- `resources/OSC/OSCParameterInterface.h/cpp` — parameter mapping
- `resources/OSC/OSCUtilities.h` — OSCReceiverPlus/OSCSenderPlus wrappers
- `resources/OSC/OSCStatus.h/cpp` — UI status indicator + config dialog

## Feature 2: VDO.Ninja Video

### Architecture

- Plugin hosts a lightweight local HTTP + WebSocket server
- "Video" button in plugin UI opens the companion page in the user's default browser
- Companion page loads VDO.Ninja as an iframe with sync-optimized parameters
- Plugin pushes sync data (buffer delay, roster) to companion page via WebSocket
- Companion page relays timing to VDO.Ninja iframe via `postMessage` API

### VDO.Ninja Integration

URL parameters for music sync (from the music-sync-buffer-demo):
```
&chunked              — custom chunked transfer protocol (not standard WebRTC)
&noaudio              — audio handled by NINJAM, not VDO.Ninja
&chunkbufferadaptive=0 — fixed buffer (no drift)
&chunkbufferceil=180000 — allow up to 3 minutes of buffering
&buffer2=0            — RTT compensated
&lowlatency           — disable echo cancellation/AGC for music
```

### Sync Mechanism

1. Plugin tracks NINJAM interval boundaries (BPM, BPI, interval count)
2. On each interval boundary, calculates target buffer delay:
   ```
   delay_ms = (60000 / BPM) * BPI * bars_to_delay + extra_offset_ms
   ```
3. Pushes `{ setBufferDelay: delay_ms }` to companion page via WebSocket
4. Companion page forwards to VDO.Ninja iframe via `postMessage`
5. VDO.Ninja adjusts its chunked buffer to match

### Room Management

- **Auto-generated room ID**: Derived from NINJAM server address (e.g., `jamwide-ninbot-com-2049`)
- **User override**: Settings field to specify a custom room name
- Everyone on the same NINJAM server joins the same VDO.Ninja room by default

### Display Modes

**Grid Mode (default):**
- Single companion browser window
- All remote users displayed in a responsive grid
- VDO.Ninja `&view` parameter shows all room participants

**Popout Mode:**
- Each remote user gets their own browser window
- Windows can be dragged to separate monitors
- Triggered per-user via UI button or OSC (`/JamWide/video/popout/{idx}`)
- Each popout window hosts one VDO.Ninja iframe (`&view=streamID`)
- Individual `setBufferDelay` per stream

### User ↔ Stream Mapping

- Plugin maintains a mapping of NINJAM user index → VDO.Ninja stream ID
- Mapping broadcast to companion page via WebSocket on roster changes
- Companion page uses stream IDs to control per-user iframes

## Open Research Questions

These grey areas need investigation during the planning phase before implementation.

### Security

- [ ] **Local server exposure**: The HTTP + WebSocket server opens a port on localhost. Can other applications or browser tabs connect? Do we need auth tokens (e.g., random token in URL query param)?
- [ ] **Firewall prompts**: Will Windows Defender / macOS firewall prompt the user when the plugin opens a listening socket? Can we bind to `127.0.0.1` only to avoid this?
- [ ] **VDO.Ninja privacy**: STUN/TURN servers can expose user IP addresses. Should we document this? Offer TURN-only mode?
- [ ] **OSC security**: UDP OSC has no authentication. Is this acceptable for a local-network-only use case? Should we offer IP filtering?

### Performance

- [ ] **Audio thread impact**: Does running a WebSocket + OSC server inside an audio plugin affect processBlock latency? Should servers run on dedicated threads?
- [ ] **Memory overhead**: What's the footprint of juce_osc + a lightweight HTTP/WS server?
- [ ] **Scalability**: Behavior with 10+ remote users (10 OSC parameter sets, 10 video streams)?
- [ ] **Browser resource usage**: Multiple popout windows each running a VDO.Ninja iframe — what's the CPU/memory impact?

### Latency & Sync

- [ ] **Sync chain jitter**: NINJAM interval → WebSocket → postMessage → setBufferDelay — what's the real-world jitter across this chain? Is it tight enough for musical sync?
- [ ] **Buffer delay accuracy**: Does VDO.Ninja's chunked mode maintain the exact requested delay, or does it drift? How often does `setBufferDelay` need to be refreshed?
- [ ] **OSC round-trip**: What's the typical UDP OSC round-trip time on WiFi (TouchOSC on iPad)? Is 100ms send interval sufficient for smooth fader feedback?

### VDO.Ninja Specifics

- [ ] **Stream ID discovery**: How does the companion page discover which VDO.Ninja stream belongs to which NINJAM user? Is there a room roster API, or do users need to share stream IDs manually?
- [ ] **Reconnection handling**: What happens when a user's video stream drops and reconnects? Does the stream ID change? How does the companion page recover?
- [ ] **NAT traversal**: Does `&chunked` mode work reliably behind corporate/university NATs? What percentage of users will need a TURN relay?
- [ ] **Codec settings**: What video codec does chunked mode use? Is H.264 baseline the default? Can we lower resolution/framerate for bandwidth-constrained sessions?

### Implementation

- [ ] **HTTP/WebSocket library choice**: Use JUCE's built-in networking, or embed a lightweight library (e.g., cpp-httplib, uWebSockets)? What's the smallest reliable option?
- [ ] **Companion page hosting**: Should the HTML/JS be embedded as binary resources in the plugin, or served from files alongside the plugin binary?
- [ ] **Multi-instance**: What happens if the user loads two JamWide instances? Port conflicts on HTTP/OSC?
- [ ] **Standalone vs plugin**: Does the local server work the same in standalone mode as in a DAW plugin context?

## Decisions Made

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Milestone scope | Both OSC + video in v1.1 | Independent features, medium-sized, no conflicts |
| Video display | Hybrid: plugin button → browser | Keeps plugin lightweight, browser handles rendering |
| Video modes | Grid (default) + per-user popout | Multi-monitor support for musicians |
| Room management | Auto-generate from NINJAM server + user override | Zero-config default, flexible for edge cases |
| OSC library | JUCE juce_osc | No external deps, proven in IEM suite |
| OSC namespace | Hierarchical with index-based remote users | Maps naturally to TouchOSC pages, stable indices |
| User identification | Index is stable address, name broadcast separately | TouchOSC layout stays consistent on roster changes |
| Video control via OSC | Yes | Single TouchOSC layout controls audio + video |
| OSC reference impl | IEM Plugin Suite pattern | Battle-tested across 20+ plugins, ~1000 LOC |
| Video sync reference | VDO.Ninja music-sync-buffer-demo | Purpose-built for NINJAM-style interval sync |
