<objective>
Pre-allocate WDL queue buffers so the audio thread never reallocates them. Closes AUDIT M-01 (`tmpblock.Resize`), reclassifies M-02 to LOW, adds debug-build assertion for M-03, and mitigates CR-11 (decoder internal allocations) via Prealloc on libvorbis/libflac WDL wrapper queues.

**Codex per-plan delta for 15.1-08 (incorporated):** the original plan included a test that inspected decoder-internal queue sizes (`dec.m_buf.GetSize() == 0`). If those fields are private (or are renamed by future libvorbis/libflac wrapper updates), the test breaks for reasons unrelated to the Prealloc functionality. **This revision uses a LIGHTER sanity test**: construct the decoder cleanly, feed it a small valid frame, verify no error; the strict allocation-count check moves to 15.1-10 Instruments UAT (where it is a runtime measurement, not a compile-time field-access dependency).

**Codex M-7 review response (incorporated):** SetMaxAudioBlockSize is the prepareToPlay-time enforcement of the MAX_BLOCK_SAMPLES bound documented in 15.1-04. If a host claims `samplesPerBlock > MAX_BLOCK_SAMPLES`, this function throws `std::runtime_error` IMMEDIATELY — preventing the situation where a too-large host block could trigger truncation in 15.1-07b's BlockRecord push.

Per RESEARCH § "Pre-Allocation Strategy" (lines 217-313):
- Add `NJClient::SetMaxAudioBlockSize(int)` public API; it asserts vs MAX_BLOCK_SAMPLES, then calls `tmpblock.Prealloc(maxSamplesPerBlock * sizeof(float))`.
- Wire from `JamWideJuceProcessor::prepareToPlay`.
- VorbisDecoder constructor calls `m_buf.Prealloc(8192 * 2)` (~64KB).
- FlacDecoder::initDecoder calls `m_inbuf.Prealloc(16384)` and `m_outbuf.Prealloc(8192 * 2)`.
- Debug-build assertion `jassert(numSamples <= prevPreparedSize)` at top of processBlock.

**Sequencing note (per RESEARCH § "Open Questions for Planner" #1)**: Decoder Prealloc calls execute in the decoder constructor. Per CR-08, decoder construction MUST run on the run thread (15.1-09 lands that). 15.1-08 depends on 15.1-09 — the decoder-prealloc safety only holds when construction is on the run thread.

Output: All audio-thread heap-grow paths identified by the audit are eliminated at steady state.
