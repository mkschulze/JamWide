## Local_Channel.channel_idx — 3 write sites

### Site 1: `src/core/njclient.cpp:1405` (auth-reply lobby renormalize) — **DEAD-BUT-LIVE FINDING**

```cpp
if (!m_max_localch)
{
  // went from lobby to room, normalize channel indices
  for (int x = 0; x < m_locchannels.GetSize(); x ++)
    m_locchannels.Get(x)->channel_idx = x;
}
```

**Status:** Same shape as the bug Bug A fixed in `_reinit` (commit `3799e8a`). The renormalize overwrites `channel_idx` on canonical `Local_Channel` objects without publishing `RemovedUpdate{old} + AddedUpdate{new}`. If it ever fires, audio-thread mirror entries are stranded the same way Bug A stranded `mirror[0]`.

**Reachability:** Currently dead in modern JamWide. The path requires a server to send TWO auth replies in one connect cycle — first with `maxchan=0` (lobby), then with `maxchan>0` (room). Modern JamWide's UI requires the user to specify a room before Connect, so the server should never report `maxchan=0`. `_reinit()` resets `m_max_localch = MAX_LOCAL_CHANNELS` (line 745) on every connect cycle, so the `!m_max_localch` gate is always false on a normal first auth reply.

**Risk:** Live latent. If a future server protocol or feature ever introduces lobby → room transitions, this would silently corrupt the mirror exactly like Bug A. Same shape, same blast radius (dead VU on affected channels).

**Recommendation:** Delete the renormalize entirely (matches the `_reinit` fix). If lobby-to-room handoff is ever needed, replace with `RemovedUpdate{old_idx} + AddedUpdate{new_idx}` per renumbered channel — the same correct pattern that `SetLocalChannelInfo`'s explicit-add path uses.

**Decision:** Defer to 15.1-10 audit per `.continue-here.md` (line was already flagged as "Auth-reply lobby renormalize at njclient.cpp:1384 — dead in modern JamWide; flag for 15.1-10 audit"). Adding a 1-line fix to the 07a dispatch increases blast radius for no reachability gain. Confirmed dead in modern JamWide; 15.1-10 verification will close it under the same architectural mirror invariant as the rest of the phase.

### Site 2: `src/core/njclient.cpp:3738` (`SetLocalChannelInfo`) — **SAFE**

```cpp
for (x = 0; x < m_locchannels.GetSize() && m_locchannels.Get(x)->channel_idx!=ch; x ++);
const bool was_add = (x == m_locchannels.GetSize());
if (was_add) m_locchannels.Add(new Local_Channel);
Local_Channel *c=m_locchannels.Get(x);
c->channel_idx=ch;
```

**Status:** SAFE. Search loop locates the entry by `channel_idx`. If found (`was_add=false`), the assignment `c->channel_idx=ch` is a no-op (same value). If not found (`was_add=true`), it's a fresh entry getting its first channel_idx, and `LocalChannelAddedUpdate{ch, ...}` is published right after on line 3768. Mirror invariant preserved by construction.

### Site 3: `src/core/njclient.cpp:3822` (`SetLocalChannelMonitoring`) — **SAFE**

```cpp
for (x = 0; x < m_locchannels.GetSize() && m_locchannels.Get(x)->channel_idx!=ch; x ++);
const bool was_add = (x == m_locchannels.GetSize());
if (was_add) m_locchannels.Add(new Local_Channel);
Local_Channel *c=m_locchannels.Get(x);
c->channel_idx=ch;
```

**Status:** SAFE. Same shape as Site 2 — search by channel_idx, assign-on-add, publish `LocalChannelAddedUpdate` (was_add=true) or `LocalChannelMonitoringUpdate` (was_add=false). Mirror invariant preserved.

## RemoteUser_Channel.channel_index — STRUCTURAL FINDING

`RemoteUser_Channel` (`src/core/njclient.cpp:392-432`) does NOT have a `channel_index` field. The class definition has `volume, pan, out_chan_index, flags, name, dump_samples, ds, next_ds[2], decode_peak_vol[2], codec_fourcc, curds_lenleft, sessioninfo, sessionlist_mutex` — channel position is implicit in the array index `ch` of `RemoteUser::channels[ch]`.

The grep hit for `channel_index` in `njclient.h:305` is a different struct (`NJClient::RemoteChannelInfo`, a UI-snapshot type used by the snapshot publisher at line 3355). `RemoteChannelInfo.channel_index` is initialized exactly once per snapshot via `ch_info.channel_index = ch;` in the run-thread snapshot loop. It's not an audio-thread-mirror key — it's a write-once field in a snapshot consumed by the UI thread. **Not a Bug A risk.**

The hits in `juce/ui/ChannelStripArea.cpp`, `juce/midi/MidiMapper.cpp`, `juce/osc/OscServer.cpp` write to a `cmd.channel_index` field on OSC/MIDI command structs, also unrelated.

### So what IS the 07a mirror invariant?

Since `RemoteUser_Channel` has no settable index, 15.1-07a's mirror cannot have a Bug-A-shape direct analog. But it has a DIFFERENT shape of the same class of bug, surfacing through `m_remoteusers` list mutations:

**`m_remoteusers` mutations:**

| Site | Operation | Shifts indices? |
|------|-----------|----------------|
| `njclient.cpp:865` | `m_remoteusers.Empty()` (Disconnect path) | Wipes all |
| `njclient.cpp:1044` | `m_remoteusers.Empty()` (second Disconnect path) | Wipes all |
| `njclient.cpp:1474` | `m_remoteusers.Add(theuser)` (auth path, new peer) | Append (no shift) |
| `njclient.cpp:1555` | `m_remoteusers.Delete(x)` (peer removal, when `chanpresentmask==0`) | **YES — shifts subsequent indices down by 1** |

**The Bug A shape on the RemoteUser side:**

If `RemoteUserMirror[]` is keyed by `m_remoteusers` list index, then `m_remoteusers.Delete(x)` is a Bug A clone — every user at index > x has their stored "user index" silently change without a mirror update. A peer iteration in the audio thread that uses the mirror's [old-shifted] index will reach into stale state.

**Mitigations (per the 15.1-07a plan's must_haves):**

1. The plan's `RemoteUserUpdate` SPSC carries explicit add/remove records — peer removal SHOULD publish `RemoteUserRemovedUpdate{user_id}` and the audio thread should locate the mirror slot by user id, not by list index.
2. The mirror should be keyed by something stable (user GUID or a monotonic peer-id), not by transient list position.
3. The deferred-free for `RemoteUser` (Task 4 of the plan, parallels 15.1-06 Task 4 for `Local_Channel`) provides the lifetime contract; the mirror invariant is what protects against silent shifts.

**Critical guidance for 15.1-07a executor:**

Whatever stable identity 15.1-07a's mirror chooses (GUID / peer-id / fixed slot), there must be NO code path that updates `m_remoteusers` order or indices without publishing the corresponding mirror update. The 4 mutation sites above are the audit's canonical list — any new code that calls `m_remoteusers.Delete/Add/Insert/Empty` MUST also publish a `RemoteUserUpdate` variant.

## Sister-bug risks for 15.1-07a (forward-looking)

The same pattern (legacy invariant violated by code outside the migrated functions) could emerge in 15.1-07a. The defenses:

1. **Mirror by GUID or fixed slot, not list position.** Rejects the Bug A shape entirely.
2. **Make every mutation site publish.** All 4 `m_remoteusers` write sites in the audit must publish a corresponding `RemoteUserUpdate` variant. The grep `grep -n 'm_remoteusers\.' src/core/njclient.cpp | grep -E 'Add|Delete|Empty|Insert'` is the canonical list — track them in 07a's SUMMARY.
3. **Audit-mirror in 15.1-10:** the same kind of grep — every write to canonical state that the mirror is keyed by must have a corresponding publish.
